## 🟢 Junior (базовое понимание)

### 1. Что такое Prometheus и для чего он используется?
**Кратко**: Open-source система мониторинга и алертинга с pull-моделью, многомерной моделью данных и мощным языком запросов PromQL.

**Подробно**: Prometheus собирает метрики по расписанию (pull), хранит их в специализированной TSDB, позволяет запрашивать данные через PromQL и генерировать алерты. Интегрируется с Grafana для визуализации и Alertmanager для уведомлений.

**DevOps-контекст**: Де-факто стандарт мониторинга в Kubernetes-экосистеме. Критичен для реализации SRE-практик: SLO/SLI, error budget, алертинг на симптомы.

**Команды**: `prometheus --config.file=prometheus.yml`, `curl http://localhost:9090/api/v1/targets`.

### 2. Какие 4 типа метрик существуют в Prometheus?
**Кратко**: Counter (растёт), Gauge (колеблется), Histogram (распределение по бакетам), Summary (предвычисленные квантили).

**Подробно**: 
- `Counter`: монотонно растёт, используется для подсчёта событий (`http_requests_total`)
- `Gauge`: может расти/падать, для текущего состояния (`node_memory_Active_bytes`)
- `Histogram`: распределяет наблюдения по бакетам, позволяет считать квантили (`http_request_duration_seconds_bucket`)
- `Summary`: предвычисляет квантили на стороне приложения, но нельзя агрегировать между инстансами

**DevOps-контекст**: Неправильный выбор типа метрики ломает аналитику: нельзя посчитать p95 из Counter, нельзя агрегировать квантили Summary.

**Команды**: `promtool check metrics < endpoint`, `curl http://app:8080/metrics | grep "^#"`.

### 3. В чём разница между pull и push моделью сбора метрик?
**Кратко**: Pull — Prometheus сам запрашивает метрики; Push — приложение отправляет метрики в Prometheus (или Pushgateway).

**Подробно**: 
- `Pull`: Prometheus делает HTTP GET к `/metrics` по расписанию. Преимущества: контроль частоты, автоматическое обнаружение "мёртвых" целей, безопасность (приложение не знает адрес Prometheus).
- `Push`: используется для short-lived jobs (batch задачи), которые не могут ждать скрейпа. Требует Pushgateway как промежуточный буфер.

**DevOps-контекст**: В Kubernetes pull-модель идеально сочетается с service discovery: Prometheus автоматически находит новые поды. Pushgateway — анти-паттерн для долгоживущих сервисов.

**Команды**: `curl http://target:8080/metrics` (ручная проверка pull), `pushgateway` для push-сценариев.

### 4. Что такое экспортеры и зачем они нужны?
**Кратко**: Экспортеры — прокси-сервисы, преобразующие метрики из сторонних систем в формат Prometheus.

**Подробно**: Экспортеры подключаются к системам, которые не экспортируют метрики нативно (ОС, БД, сетевое оборудование), и отдают их на `/metrics` в Prometheus text format. Примеры: `node_exporter` (Linux), `mysqld_exporter`, `blackbox_exporter` (HTTP/TCP проверки).

**DevOps-контекст**: Позволяют унифицировать мониторинг разнородной инфраструктуры без изменения исходных систем. Запускать с минимальными привилегиями из-за доступа к чувствительным данным.

**Команды**: `node_exporter --web.listen-address=:9100`, `curl http://localhost:9100/metrics`.

### 5. Как работает service discovery в Prometheus?
**Кратко**: Автоматическое обнаружение целей для скрейпинга без ручного обновления конфига.

**Подробно**: Prometheus поддерживает SD для Kubernetes, Consul, DNS, EC2, Azure и др. При использовании `kubernetes_sd_configs` Prometheus автоматически находит поды/сервисы по лейблам и аннотациям. `relabel_configs` позволяют фильтровать и трансформировать найденные цели.

**DevOps-контекст**: Критично для динамических сред: новые поды автоматически добавляются в мониторинг, удалённые — удаляются. Без SD пришлось бы перезагружать Prometheus при каждом деплое.

**Команды**: `kubectl get pods -l prometheus.io/scrape=true`, проверка в UI: `Status → Targets`.

### 6. Что такое лейблы и как они влияют на кардинальность?
**Кратко**: Лейблы — ключ-значение пары, добавляемые к метрике; каждая уникальная комбинация = отдельный временной ряд.

**Подробно**: 
```
Метрика: http_requests_total
Лейблы: {method="POST", handler="/api", status="200", instance="api-1:8080"}
Уникальный ряд = метрика + отсортированные лейблы
```
Кардинальность = количество уникальных комбинаций лейблов. Высокая кардинальность (например, `user_id` как лейбл) ведёт к росту памяти и деградации производительности.

**DevOps-контекст**: Понимание кардинальности критично для проектирования метрик. Никогда не используйте высококардинальные данные (UUID, email) как лейблы.

**Команды**: `count by (__name__) ({__name__=~".+"})` — топ метрик по кардинальности.

### 7. Как написать базовый запрос PromQL для подсчёта RPS?
**Кратко**: `sum(rate(http_requests_total[5m]))`

**Подробно**: 
- `rate(http_requests_total[5m])` — скорость изменения counter'а за 5 минут, экстраполированная на секунду
- `sum()` — агрегирует по всем инстансам, методам, статусам
- Окно `[5m]` должно быть ≥4× `scrape_interval` для стабильности

**DevOps-контекст**: Базовая метрика для дашборда "здоровья" сервиса. В алертах: резкое падение RPS может означать простой.

**Команды**: В Grafana: добавить панель → Query: `sum(rate(http_requests_total[5m]))` → Legend: `RPS`.

[[#🗂️ Оглавление для навигации|↑ К оглавлению]]

### 8. Что делает функция rate() и зачем ей временное окно?
**Кратко**: Вычисляет скорость изменения counter'а в секунду, используя экстраполяцию для компенсации пропущенных скрейпов.

**Подробно**: 
- Формула: `(последнее - первое) / (время_между_ними) × экстраполяция`
- Возвращает float, даже если counter целочисленный
- Автоматически корректирует сброс counter'а при рестарте процесса
- Окно должно быть ≥4× `scrape_interval` для стабильности

**DevOps-контекст**: Понимание экстраполяции помогает интерпретировать "дробные" значения от `increase()` и избегать ложных алертов.

**Команды**: `rate(http_requests_total[2m])` при `scrape_interval: 30s`.

### 9. Как настроить простой алерт на высокую загрузку CPU?
**Кратко**: Использовать `node_cpu_seconds_total` с `rate()` и пороговым условием.

**Подробно**:
```yaml
- alert: HighCPUUsage
  expr: avg(rate(node_cpu_seconds_total{mode!="idle"}[10m])) by (instance) > 0.8
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "High CPU on {{ $labels.instance }}"
    description: "CPU usage is {{ $value | humanizePercentage }} for 10m"
```

**DevOps-контекст**: `for: 5m` предотвращает ложные срабатывания на кратковременные пики. Использовать `avg by (instance)` для усреднения по ядрам.

**Команды**: `promtool check rules rules.yml` — валидация перед деплоем.

[[#🗂️ Оглавление для навигации|↑ К оглавлению]]

### 10. Что такое up метрика и как её использовать для проверки здоровья?
**Кратко**: Специальная метрика: 1 = скрейпинг успешен, 0 = ошибка подключения или парсинга.

**Подробно**: 
- Добавляется автоматически ко всем target'ам
- `up{job="my-service"} == 0` означает, что Prometheus не может получить метрики
- Не гарантирует, что приложение работает — только что `/metrics` доступен

**DevOps-контекст**: Базовая проверка после деплоя конфига или нового сервиса. Комбинировать с бизнес-метриками для полного мониторинга здоровья.

**Команды**: `up{job="prometheus"}`, `absent(up{job="critical"})` для алерта на пропавшие метрики.

### 11. ⭐ В чём разница между rate() и irate()? Когда что использовать?
**Кратко**: `rate()` усредняет по всему окну, `irate()` использует последние 2 точки; `rate()` стабильнее для алертов, `irate()` реактивнее для дашбордов.

**Подробно**: 
- `rate(counter[5m])`: берёт все точки за 5 минут, вычисляет линейный тренд → стабильно, но с задержкой
- `irate(counter[5m])`: берёт только последние 2 точки → мгновенная реакция, но шумно при нерегулярных скрейпах
- Обе функции корректируют сброс counter'а при рестарте

**DevOps-контекст**: Для алертов используйте `rate()` с окном ≥2м — меньше ложных срабатываний. Для дашбордов можно `irate()` для реактивности, но с осторожностью.

**Команды**: Сравнить визуально в Grafana: два графика с `rate()` и `irate()` на одном counter'е.

### 12. Как посчитать 95-й перцентиль из гистограммы?
**Кратко**: `histogram_quantile(0.95, sum(rate(metric_bucket[5m])) by (le))`

**Подробно**: 
- `metric_bucket{le="..."}` — бакеты гистограммы
- `rate(...[5m])` — скорость попадания в каждый бакет
- `sum by (le)` — агрегируем бакеты, сохраняя лейбл `le` (less or equal)
- `histogram_quantile(0.95, ...)` — интерполирует 95-й перцентиль

**DevOps-контекст**: p95 лучше среднего для оценки пользовательского опыта: показывает, что видят 95% пользователей, игнорируя выбросы. Критично для SLO по latency.

**Команды**: Если метрика — summary, использовать: `metric{quantile="0.95"}` (но нельзя агрегировать между инстансами).

### 13. Что такое cardinality explosion и как его избежать?
**Кратко**: Взрыв кардинальности — когда количество уникальных временных рядов растёт экспоненциально, приводя к росту памяти и деградации производительности.

**Подробно**: 
- Кардинальность = ∏ (уникальные значения каждого лейбла)
- Пример проблемы: `http_requests_total{user_id}` с 1млн пользователей = 1млн+ серий
- Последствия: рост памяти (~2-3 байта/серия), медленные запросы, риск OOM

**Решения**:
- Не использовать высококардинальные данные как лейблы (user_id, request_id, pod_name без агрегации)
- Агрегировать на уровне приложения или recording rules
- Использовать `metric_relabel_configs` для дропа лишних лейблов

**DevOps-контекст**: Кардинальность — главный лимитирующий фактор масштабируемости Prometheus. Мониторить обязательно: `count by (__name__) ({__name__=~".+"})`.

### 14. ⭐ Как работает absent() и когда он возвращает 1?
**Кратко**: `absent()` возвращает вектор с одним элементом = 1, если входной вектор пуст (нет серий), иначе возвращает пустой вектор.

**Подробно**: 
- Используется для алертов на "пропавшие" метрики
- Работает только с instant vector (не с range selector)
- Учитывает staleness: если серия не обновлялась >5м, считается отсутствующей

**Пример**:
```promql
# Алерт, если метрика пропала на 2 минуты
absent(up{job="critical-service"})
```

**DevOps-контекст**: Критичный алерт: "мониторинг не видит сервис" → проверить деплой, сеть, exporter. Комбинировать с `for: 2m` для избежания ложных срабатываний.

**Команды**: `absent(http_requests_total{service="api", env="prod"})` для проверки наличия метрик.

### 15. В чём разница между recording rules и alerting rules?
**Кратко**: Recording rules предвычисляют метрики для оптимизации запросов; alerting rules генерируют алерты при выполнении условий.

**Подробно**: 
- `Recording rule`:
  ```yaml
  - record: job:http_requests:rate5m
    expr: sum(rate(http_requests_total[5m])) by (job)
  ```
  → создаёт новую метрику, которая записывается в TSDB
  
- `Alerting rule`:
  ```yaml
  - alert: HighErrorRate
    expr: ... > 0.05
    for: 2m
    labels: {...}
    annotations: {...}
  ```
  → при выполнении условия создаёт алерт для Alertmanager

**DevOps-контекст**: Recording rules снижают нагрузку на Prometheus при 100+ одновременных запросах из Grafana. Alerting rules — основа проактивного мониторинга.

**Команды**: `promtool check rules rules.yml` — валидация обоих типов правил.

### 16. Как использовать relabel_configs для фильтрации метрик?
**Кратко**: `relabel_configs` применяются к target'ам ДО скрейпа, `metric_relabel_configs` — к метрикам ПОСЛЕ скрейпа.

**Подробно**: 
- `relabel_configs` (до скрейпа):
  ```yaml
  - source_labels: [__meta_kubernetes_pod_annotation_team]
    target_label: team
    action: replace
  ```
- `metric_relabel_configs` (после скрейпа):
  ```yaml
  # Дропнуть метрики с user_id
  - source_labels: [__name__]
    regex: .*_with_user_id
    action: drop
  ```

**Доступные действия**: `replace`, `keep`, `drop`, `hashmod`, `labelmap`, `labeldrop`, `labelkeep`.

**DevOps-контекст**: Позволяет стандартизировать лейблы и снижать кардинальность без изменения кода приложений. Порядок правил важен — выполняются последовательно.

**Команды**: `promtool check config prometheus.yml` — валидация конфигов.

### 17. Что такое staleness и как Prometheus определяет "мёртвые" серии?
**Кратко**: Если серия не получала новые семплы в течение 5 минут (по умолчанию), она считается "устаревшей" и исключается из запросов.

**Подробно**: 
- Настраивается через `--query.lookback-delta` (default: 5m)
- При запросах: устаревшие серии не возвращаются
- В алертах: `absent()` учитывает staleness
- Приложение может явно послать "stale marker" для завершения серии

**DevOps-контекст**: Понимание staleness помогает настраивать чувствительность алертов на "пропавшие" сервисы и интерпретировать пустые результаты запросов.

**Команды**: `absent(up{job="critical"})` для алерта на пропавшие метрики.

### 18. Как настроить Alertmanager для группировки и маршрутизации алертов?
**Кратко**: Использовать `group_by`, `routes` и `receivers` в `alertmanager.yml`.

**Подробно**:
```yaml
route:
  group_by: ['alertname', 'service']
  group_wait: 30s      # ждать перед первой отправкой группы

  - name: pagerduty-critical
    pagerduty_configs: [...]
  - name: slack-frontend
    slack_configs: [...]
```

**DevOps-контекст**: Правильная группировка снижает alert fatigue: вместо 50 уведомлений о сбое кластера — одно сгруппированное. Маршрутизация обеспечивает эскалацию критичных инцидентов.

### 19. В чём разница между histogram и summary? Когда что выбрать?
**Кратко**: Histogram хранит распределение по бакетам и позволяет агрегировать; Summary предвычисляет квантили на стороне приложения, но нельзя агрегировать между инстансами.

**Подробно**:
| Критерий | Histogram | Summary |
|----------|-----------|---------|
| **Агрегация** | ✅ Можно суммировать бакеты между инстансами | ❌ Квантили нельзя агрегировать |
| **Квантили** | Аппроксимация постфактум | Точные, но только для одного инстанса |
| **Бакеты** | Фиксированы при компиляции | Не применимо |
| **Использование** | Latency для дашбордов и алертов | Точный p99 на одном поде (редко) |

**Выбор**: В 95% случаев используйте Histogram — он гибче и позволяет агрегацию.

**DevOps-контекст**: Неправильный выбор ломает аналитику: нельзя посчитать глобальный p95 из Summary. Планируйте метрики на этапе дизайна сервиса.

### 21. ⭐ Как устроена внутренняя архитектура TSDB Prometheus?
**Кратко**: WAL для durability → Head block в памяти → immutable blocks на диске с индексами и сжатием.

**Подробно**:
```
1. WAL (.wal/):
   • Последовательная запись всех семплов
   • Гарантирует сохранность при краше
   • Удаляется после создания блока

2. Head block (в памяти):
   • Последние 2-3 часа данных
   • Оптимизирован для быстрой записи и recent queries

3. Immutable blocks (на диске, 2 часа каждый):
   • Структура: meta.json, index, chunks/, postings/
   • Формат chunks: XOR-delta encoding + zlib compression
   • Индекс: inverted index для быстрого поиска по метрикам/лейблам

4. Compaction:
   • Блоки 2h → 8h → 24h → ... (иерархическая агрегация)
   • Уменьшает количество файлов, ускоряет long-range queries
```

**DevOps-контекст**: Понимание TSDB помогает тюнить `--storage.tsdb.retention.*`, диагностировать медленные запросы и планировать миграцию на Thanos для long-term storage.

**Команды**: `curl http://prometheus:9090/api/v1/status/tsdb`, `ls -lh /prometheus/data/blocks/`.

### 22. Почему WAL (Write-Ahead Log) критичен для надёжности Prometheus?
**Кратко**: WAL гарантирует, что данные не потеряются при краше Prometheus до их компакции в блоки.

**Подробно**: 
- Запись: каждый семпл → WAL → Head block (in-memory)
- При рестарте: чтение WAL с последнего чекапоинта → восстановление Head
- Формат записи: [4B длина][тип: series/sample][данные][CRC32]
- Оптимизации: batched writes, async fsync, checkpointing

**DevOps-контекст**: Проблемы с диском (медленный I/O, переполнение) → потеря метрик или падение Prometheus. Мониторить: `prometheus_tsdb_wal_corruptions_total`.

**Алерт на проблемы с WAL**:
```yaml
- alert: PrometheusWALCorruption
  expr: increase(prometheus_tsdb_wal_corruptions_total[5m]) > 0
  severity: critical
```

### 23. Как работает compaction в TSDB и когда запускать его вручную?
**Кратко**: Преобразование in-memory Head block в оптимизированные on-disk блоки с индексами и сжатием.

**Подробно**:
```
Этапы:
1. Head → Block: при достижении 2ч или ~512MB
   • Сортировка серий, XOR-encoding, построение индексов
   • Результат: immutable block в data/blocks/<ulid>/

2. Vertical compaction: блоки 2h → 8h → 24h → ...
   • Уменьшает количество файлов для long-range queries

3. Очистка: блоки старше retention удаляются
```

**Ручной запуск**: Обычно не нужен — compaction работает в фоне. Исключения:
- После импорта больших объёмов данных
- При отладке проблем с блоками
- Команда: `curl -XPOST http://prometheus:9090/api/v1/admin/tsdb/snapshot`

**DevOps-контекст**: Compaction влияет на query latency и disk usage. Мониторить: `prometheus_tsdb_compactions_total`, `prometheus_tsdb_blocks_loaded`.

### 24. Что происходит "под капотом" при выполнении PromQL-запроса?
**Кратко**: Парсинг → планирование → загрузка данных → выполнение операций → форматирование ответа.

**Подробно**:
```
1. Парсинг → AST: лексический анализ, построение дерева операций
2. Планирование:
   • Определение необходимых данных
   • Оптимизация: pushdown фильтрации, выбор индексов
3. Загрузка данных:
   • Поиск серий в индексе по метрике и лейблам
   • Загрузка и декодирование чанков (XOR + zlib)
4. Выполнение:
   • Векторные операции поэлементно
   • Агрегации: группировка по лейблам
5. Форматирование: преобразование в JSON для API
```

**DevOps-контекст**: Понимание query engine помогает писать эффективные запросы: фильтровать по лейблам ДО агрегации для использования индексов.

**Профилирование**: Добавить `?stats=all` к API-запросу для детальной статистики выполнения.

### 25. ⭐ Как безопасно масштабировать Prometheus: federation vs remote_write vs Thanos?
**Кратко**: Federation — для <10 инстансов, remote_write — для централизации, Thanos/Cortex/Mimir — для горизонтального масштабирования.

**Подробно**:
| Решение | Лучше всего для | Ограничения |
|---------|----------------|-------------|
| **Federation** | <10 Prometheus, агрегация конкретных метрик | Не масштабируется линейно, ручное распределение |
| **remote_write** | Централизация в Thanos Receive / Cortex / Mimir | Требует внешнего хранилища, задержка синхронизации |
| **Thanos** | Long-term storage + глобальный view к существующим Prometheus | Средняя операционная сложность |
| **Cortex/Mimir** | Построение масштабируемой multi-tenant платформы с нуля | Высокая сложность, требует экспертизы |

**Рекомендация**: Начните с remote_write в Thanos Receive — это даёт long-term storage без полной миграции архитектуры.

**DevOps-контекст**: Выбор стратегии масштабирования — стратегическое решение. Ошибка стоит дорого: миграция между решениями сложна.

### 26. Как удалить чувствительные данные из метрик без перезаписи истории?
**Кратко**: Использовать `metric_relabel_configs` для дропа/маскирования лейблов на этапе скрейпа; для уже записанных данных — только замена через remote_write с фильтрацией.

**Подробно**: 
- **Профилактика** (новые данные):
  ```yaml
  metric_relabel_configs:
    # Дропнуть лейбл user_id
    - source_labels: [user_id]
      action: labeldrop
    # Заменить значение на хеш
    - source_labels: [email]
      target_label: email_hash
      regex: (.+)
      replacement: sha256($1)
      action: replace
  ```
- **Исторические данные**: 
  • Нельзя изменить уже записанные блоки
  • Решение: настроить remote_write с фильтрацией → новые данные пойдут "чистыми", старые со временем удалятся по retention

**DevOps-контекст**: Соответствие требованиям безопасности (GDPR, SOC2): не хранить PII в метриках. Автоматизировать проверку через пре-коммит хуки и CI (`gitleaks` для конфигов).

**Команды**: `promtool check config prometheus.yml` — валидация relabel rules.

### 27. В чём преимущества и риски использования remote_write в production?
**Кратко**: Преимущества: долгосрочное хранение, централизация, гибкость запросов. Риски: задержка синхронизации, сложность отладки, зависимость от сети.

**Подробно**:
```
Преимущества:
✅ Долгосрочное хранение (месяцы/годы) в object storage
✅ Централизация метрик из множества Prometheus
✅ Интеграция с Thanos, Cortex, Mimir для масштабирования
✅ Гибкость: разные retention policies для разных данных

Риски:
❌ Задержка синхронизации: данные появляются с лагом
❌ Сложность отладки: проблемы в цепочке Prometheus → remote_write → storage
❌ Зависимость от сети: разрыв → потеря данных в буфере
❌ Стоимость: object storage + дополнительные компоненты

Митигация рисков:
• Настроить queue_config с retry и backoff
• Мониторить лаг: `prometheus_remote_storage_highest_timestamp_in_seconds - time()`
• Использовать circuit breaker для graceful degradation
• Тестировать failover сценарии
```

**DevOps-контекст**: Remote_write — must-have для production, но требует мониторинга самой инфраструктуры отправки метрик.

### 28. Как оптимизировать Prometheus для работы с >1M временных рядов?
**Кратко**: Снижение кардинальности, тюнинг ресурсов, использование recording rules, sharding через Thanos/Cortex.

**Подробно**:
```
1. Снижение кардинальности:
   • Удалить высококардинальные лейблы через metric_relabel_configs
   • Агрегировать на уровне приложения или recording rules
   • Использовать `__name__` и несколько ключевых лейблов

2. Тюнинг ресурсов:
   • Память: ~2-3 байта/серия для Head + индекс → 1M series ≈ 3GB RAM минимум
   • Диск: ~1-2 байта/серия/час после сжатия → 1M × 1.5B × 24h × 15d ≈ 540GB
   • CPU: ~1 ядро на 50-100k семплов/сек для ingestion

3. Оптимизация запросов:
   • Recording rules для предвычисления тяжёлых запросов
   • Фильтрация по лейблам ДО агрегации для использования индексов
   • Уменьшение диапазона запросов в дашбордах

4. Горизонтальное масштабирование:
   • Thanos Sidecar для выгрузки блоков в object storage
   • Cortex/Mimir для native sharding и multi-tenancy
```

**DevOps-контекст**: 1M+ серий — граница, после которой вертикальное масштабирование становится неэффективным. Планируйте миграцию на распределённую архитектуру заранее.

**Мониторинг**:
```promql
prometheus_tsdb_head_series  # серии в памяти
prometheus_tsdb_storage_blocks_bytes  # использование диска
rate(prometheus_tsdb_head_samples_appended_total[5m])  # ingestion rate
```

### 29. Как устроена кластеризация Alertmanager и что такое gossip-протокол?
**Кратко**: Gossip-протокол обеспечивает eventual consistency между репликами Alertmanager без центрального координатора.

**Подробно**:
```
Как работает gossip (Hashicorp memberlist):
1. Обнаружение узлов: при старте подключается к известным пирам
2. Распространение состояния:
   • Изменение алерта → рассылка 3 случайным пирам (fanout=3)
   • Получатели мерджат изменения, разрешая конфликты по времени
   • Eventual consistency: все узлы сходятся за ~секунды
3. Дедупликация: один алерт от нескольких Prometheus не дублируется
4. Отправка уведомлений: только один узел кластера отправляет уведомление (хеш от fingerprint)

Конфигурация:
```yaml
cluster:
  listen-address: 0.0.0.0:9094
  peers:
    - alertmanager-0:9094
    - alertmanager-1:9094
    - alertmanager-2:9094  # нечётное число для кворума
```

**DevOps-контекст**: Кластеризация критична для отказоустойчивости алертинга. Требует стабильной сети (TCP/UDP 9094) и мониторинга самого кластера.

**Мониторинг кластера**:
```promql
alertmanager_cluster_members  # сколько узлов в кластере
alertmanager_cluster_health_score  # 0-1, чем ближе к 1 тем лучше
```

### 30. ⭐ Как спроектировать отказоустойчивую observability-платформу для 100+ сервисов?
**Кратко**: Нет SPOF, graceful degradation, изоляция отказов, мониторинг самой платформы.

**Подробно**:
```
Принципы:
1. Нет единой точки отказа:
   • Prometheus: 2+ реплики за load balancer
   • Alertmanager: кластер из 3+ узлов (кворум для gossip)
   • Storage: object storage с репликацией (S3/GCS)

2. Graceful degradation:
   • При потере long-term storage: Prometheus продолжает работать локально
   • При потере одного Prometheus: другие покрывают нагрузку
   • Circuit breakers на remote_write: не блокировать локальную запись

3. Изоляция отказов:
   • Разные команды → разные scrape jobs → сбой одного не влияет на других
   • Resource limits в Kubernetes: чтобы "шумный сосед" не уронил Prometheus

Референсная архитектура:
```
┌─────────────────┐
│  2× Prometheus  │  ← реплики за LB
│  (scrape apps)  │
└────────┬────────┘
         │ remote_write
         ▼
┌─────────────────┐
│  Thanos Receive │  → Object Storage (S3)
│  + Query + Compactor │
└────────┬────────┘
         │ query/alerting
         ▼
┌─────────────────┐
│ Alertmanager    │  ← кластер 3 узла
│ (Slack/PagerDuty)│
└─────────────────┘
```

**DevOps-контекст**: Платформа мониторинга должна быть надёжнее, чем системы, которые она мониторит. Регулярные chaos-тесты и задокументированные runbooks обязательны.

**Чек-лист перед продакшеном**:
- [ ] 2+ реплики Prometheus за LB
- [ ] Alertmanager кластер из 3 узлов
- [ ] Object storage с versioning и cross-region replication
- [ ] Resource limits для всех компонентов
- [ ] Алерты на здоровье самой observability-платформы
- [ ] Задокументированные runbooks и регулярные учения
```
