### 1. Как устроена модель данных Python: объекты, ссылки, мутабельность?
**Суть**: В Python "всё есть объект", переменные — это ссылки на объекты в памяти, а не сами значения.

**Подробно**:
```
Объект = тип + значение + метаданные (refcount, hash, etc.)
• int, str, tuple — immutable (неизменяемые)
• list, dict, set — mutable (изменяемые)

Ссылки и присваивание:
```
```python
a = [1, 2, 3]    # создаётся объект list в памяти, a ссылается на него
b = a            # b ссылается на тот же объект (не копия!)
b.append(4)      # изменяется общий объект
print(a)         # [1, 2, 3, 4] — a тоже изменился!

# Проверка идентичности:
a is b           # True — одна и та же ячейка памяти
a == b           # True — равные значения
id(a) == id(b)   # True — одинаковый адрес в памяти
```

Копирование:
• `b = a[:]` или `b = a.copy()` — shallow copy (копия контейнера, но не вложенных объектов)
• `import copy; b = copy.deepcopy(a)` — deep copy (рекурсивное копирование всего)

**DevOps-контекст**: Понимание мутабельности предотвращает баги в конфигурациях: изменение "копии" конфига может неожиданно поменять оригинал. Всегда копируй мутабельные дефолты в функциях.

**Связанные инструменты**: `sys.getrefcount()`, `id()`, `copy` модуль.

### 2. В чём разница между list, tuple, dict, set и когда что использовать?
**Суть**: Разные структуры данных оптимизированы под разные операции доступа и изменения.

**Подробно**:
| Структура | Изменяемость | Порядок | Уникальность | Сложность доступа | Когда использовать |
|-----------|-------------|---------|-------------|------------------|-------------------|
| **list** | ✅ | ✅ | ❌ | O(1) по индексу | Упорядоченная коллекция, итерация по индексу |
| **tuple** | ❌ | ✅ | ❌ | O(1) по индексу | Неизменяемые группы, ключи словаря, возврат нескольких значений |
| **dict** | ✅ | ✅ (3.7+) | Ключи уникальны | O(1) в среднем | Маппинг ключ→значение, конфигурации, кэши |
| **set** | ✅ | ❌ | ✅ | O(1) в среднем | Проверка принадлежности, уникализация, операции множеств |

**Примеры использования в DevOps**:
```python
# list: очередь задач, порядок деплоя
deploy_order = ["db-migrate", "cache-warmup", "app-deploy"]

# tuple: неизменяемая конфигурация, ключ кэша
cache_key = ("user-stats", "2024-05", "RU")

# dict: конфигурация приложения, метрики
config = {"db_host": "postgres", "replicas": 3, "timeout": 30}

# set: уникальные теги, проверка членства
allowed_envs = {"dev", "staging", "prod"}
if env in allowed_envs: ...
```

**DevOps-контекст**: Выбор правильной структуры влияет на производительность: `x in list` — O(n), `x in set` — O(1). Для больших коллекций это критично.

### 3. Как работает управление памятью: reference counting и garbage collector?
**Суть**: Python использует подсчёт ссылок для немедленного освобождения памяти + циклический GC для сбора циклических ссылок.

**Подробно**:
```
Reference counting:
• Каждый объект хранит счётчик ссылок на него
• При присваивании: refcount++
• При удалении ссылки: refcount--
• При refcount==0: объект немедленно удаляется

Проблема: циклические ссылки
```
```python
class Node:
    def __init__(self):
        self.ref = None

a = Node()
b = Node()
a.ref = b  # a → b
b.ref = a  # b → a (цикл!)
# Даже если удалить a и b, refcount не станет 0 из-за цикла
```

Garbage collector (модуль `gc`):
• Периодически ищет недостижимые циклы
• Работает по поколениям (generational GC): молодые объекты проверяются чаще
• Можно настроить пороги: `gc.set_threshold(700, 10, 10)`

**DevOps-контекст**: В долгоживущих процессах (демоны, операторы) циклические ссылки могут привести к утечкам памяти. Используй `weakref` для обратных ссылок и профилируй память в production.

**Связанные инструменты**: `gc` модуль, `tracemalloc`, `objgraph`, `pympler`.

### 4. Что такое итерируемые объекты, итераторы и генераторы?
**Суть**: Протокол итерации позволяет лениво обрабатывать последовательности без загрузки всего в память.

**Подробно**:
```
Итерируемый объект (Iterable):
• Реализует __iter__() → возвращает итератор
• Примеры: list, dict, str, range, генераторы

Итератор (Iterator):
• Реализует __iter__() (возвращает self) и __next__()
• __next__() возвращает следующий элемент или бросает StopIteration
• Состояние итерации хранится внутри итератора

Генератор (Generator):
• Функция с yield или генераторное выражение
• Автоматически реализует протокол итератора
• Сохраняет состояние между вызовами (локальные переменные)

Пример:
```
```python
# Генератор — ленивая обработка
def read_large_file(path):
    with open(path) as f:
        for line in f:  # читает по одной строке
            yield line.strip()

# Использование: потребляет O(1) памяти независимо от размера файла
for line in read_large_file("10gb.log"):
    if "ERROR" in line:
        process(line)
```

**DevOps-контекст**: Генераторы критичны для обработки больших логов, потоковой передачи данных, работы с API pagination. Никогда не загружай гигабайтные файлы целиком через `read()`.

### 5. Как работает область видимости: LEGB rule и замыкания?
**Суть**: Разрешение имён переменных происходит по правилу LEGB: Local → Enclosing → Global → Built-in.

**Подробно**:
```
LEGB порядок поиска:
1. Local — внутри текущей функции
2. Enclosing — в охватывающих функциях (для вложенных)
3. Global — на уровне модуля
4. Built-in — встроенные имена (len, print, etc.)

Пример:
```
```python
x = "global"

def outer():
    x = "enclosing"
    def inner():
        x = "local"
        print(x)  # "local"
    inner()

# Ключевое слово nonlocal для изменения переменной из enclosing scope:
def counter():
    count = 0
    def increment():
        nonlocal count  # разрешаем менять переменную из внешней функции
        count += 1
        return count
    return increment
```

Замыкание (Closure):
• Функция, "запоминающая" переменные из охватывающей области
• Используется в декораторах, фабриках функций, коллбэках

**DevOps-контекст**: Понимание областей видимости предотвращает баги с глобальными переменными в скриптах автоматизации. Используй замыкания для создания конфигурируемых обработчиков.

### 6. В чём разница между __str__ и __repr__?
**Суть**: `__str__` — человекочитаемое представление для пользователя; `__repr__` — однозначное представление для разработчика/отладки.

**Подробно**:
```
__str__:
• Вызывается через str(obj) или print(obj)
• Цель: удобочитаемый вывод для конечного пользователя
• Может быть неформальным, сокращённым

__repr__:
• Вызывается через repr(obj) или в интерактивной консоли
• Цель: однозначное представление, желательно валидный Python-код
• Должен позволять воссоздать объект: eval(repr(obj)) == obj (в идеале)

Пример:
```
```python
from datetime import datetime

now = datetime(2024, 5, 7, 14, 30)
print(str(now))   # "2024-05-07 14:30:00" — удобно читать
print(repr(now))  # "datetime.datetime(2024, 5, 7, 14, 30)" — можно скопировать в код

# Реализация в своём классе:
class DeployConfig:
    def __init__(self, service, env):
        self.service = service
        self.env = env
    
    def __repr__(self):
        return f"DeployConfig({self.service!r}, {self.env!r})"
    
    def __str__(self):
        return f"{self.service} → {self.env}"
```

**DevOps-контекст**: Хороший `__repr__` критичен для логирования и отладки автоматизаций. В логах должно быть видно, с какими именно объектами работала система.

### 7. Что такое декораторы и как они работают "под капотом"?
**Суть**: Декоратор — функция, принимающая другую функцию и возвращающая обёртку с дополнительной логикой.

**Подробно**:
```
Синтаксический сахар:
```
```python
@decorator
def func(): ...

# Эквивалентно:
def func(): ...
func = decorator(func)
```

Механизм работы:
```
```python
def retry(max_attempts=3):
    def decorator(func):
        def wrapper(*args, **kwargs):
            for attempt in range(max_attempts):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if attempt == max_attempts - 1:
                        raise
                    time.sleep(2 ** attempt)  # exponential backoff
        return wrapper
    return decorator

# Использование:
@retry(max_attempts=5)
def call_api(url):
    return requests.get(url)
```

Ключевые моменты:
• Декоратор с аргументами возвращает декоратор (два уровня вложенности)
• `*args, **kwargs` в wrapper передаёт аргументы оригинальной функции
• `functools.wraps(func)` сохраняет метаданные оригинальной функции (__name__, __doc__)

**DevOps-контекст**: Декораторы — основа для cross-cutting concerns в автоматизации: retry, logging, metrics, timeout, circuit breaker. Позволяют добавлять поведение без изменения бизнес-логики.

### 8. Как работает контекстный менеджер и протокол with?
**Суть**: Контекстный менеджер гарантирует выполнение кода "на выходе" из блока, даже при исключениях.

**Подробно**:
```
Протокол:
• Объект должен реализовать __enter__() и __exit__()
• __enter__() выполняется при входе в with, возвращает объект для as-переменной
• __exit__(exc_type, exc_val, exc_tb) выполняется при выходе, обрабатывает исключения

Пример реализации:
```
```python
class DatabaseConnection:
    def __init__(self, dsn):
        self.dsn = dsn
        self.conn = None
    
    def __enter__(self):
        self.conn = connect(self.dsn)
        return self.conn
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        if self.conn:
            if exc_type is None:
                self.conn.commit()  # только если не было исключения
            self.conn.close()
        return False  # не подавлять исключения

# Использование:
with DatabaseConnection("postgres://...") as conn:
    conn.execute("INSERT INTO logs VALUES (...)")
# conn.close() вызван автоматически, даже если execute упал
```

Контекстные менеджеры из стандартной библиотеки:
• `open()` — файлы
• `threading.Lock()` — синхронизация
• `contextlib.closing()` — для объектов с close()
• `@contextmanager` декоратор — создание через генератор

**DevOps-контекст**: Контекстные менеджеры критичны для управления ресурсами: соединения с БД, файлы, сокеты, блокировки. Предотвращают утечки ресурсов при ошибках.

### 9. Что такое *args и **kwargs и как работает распаковка?
**Суть**: Механизм передачи переменного числа позиционных и именованных аргументов.

**Подробно**:
```
В определении функции:
• *args — собирает лишние позиционные аргументы в tuple
• **kwargs — собирает лишние именованные аргументы в dict

В вызове функции:
• *iterable — распаковывает последовательность в позиционные аргументы
• **dict — распаковывает словарь в именованные аргументы

Примеры:
```
```python
# Передача переменного числа аргументов
def deploy(*services, env="dev", **options):
    for svc in services:
        print(f"Deploying {svc} to {env} with {options}")

deploy("api", "web", env="prod", replicas=3)
# services = ("api", "web"), env="prod", options={"replicas": 3}

# Распаковка при вызове
config = {"timeout": 30, "retries": 3}
call_api("https://...", **config)  # эквивалентно timeout=30, retries=3

# Распаковка вложенных структур
nested = {"db": {"host": "localhost", "port": 5432}}
connect(**nested["db"])  # connect(host="localhost", port=5432)
```

**DevOps-контекст**: Распаковка критична для гибких конфигураций: передача конфигов как dict, динамическое формирование аргументов к CLI-утилитам, обёртки над внешними API.

### 10. Как устроена система модулей и пакетов: import, __init__, sys.path?
**Суть**: Механизм загрузки и кэширования модулей, управление путями поиска, инициализация пакетов.

**Подробно**:
```
Процесс импорта:
1. Проверка sys.modules (кэш уже загруженных модулей)
2. Поиск файла по sys.path (список директорий)
3. Загрузка и выполнение кода модуля
4. Создание объекта модуля в sys.modules
5. Возврат ссылки на модуль

sys.path:
• Список строк-путей, где Python ищет модули
• Инициализируется из: скрипт-директория, PYTHONPATH, стандартная библиотека, site-packages
• Можно модифицировать: sys.path.insert(0, "/custom/path")

__init__.py:
• Делает директорию пакетом (в Python <3.3 обязательно)
• Выполняется при первом импорте пакета
• Может экспортировать символы через __all__

Кэширование:
• Модуль выполняется только один раз за интерпретатор
• Повторные import возвращают тот же объект из sys.modules
• Для перезагрузки: importlib.reload(module)

Пример структуры:
```
```
my_package/
├── __init__.py      # from .core import deploy; __all__ = ["deploy"]
├── core.py          # def deploy(...): ...
├── utils/
│   ├── __init__.py
│   └── helpers.py
```

**DevOps-контекст**: Понимание импорта критично для упаковки инструментов, избегания циклических зависимостей, динамической загрузки плагинов. Используй относительные импорты внутри пакетов.

### 11. ⭐ Как работает GIL и почему он влияет на многопоточность?
**Суть**: Global Interpreter Lock — мьютекс, защищающий доступ к объектам Python, позволяющий только одному потоку выполнять байт-код одновременно.

**Подробно**:
```
Зачем нужен GIL:
• Упрощает управление памятью (reference counting не требует атомарных операций)
• Делает C-расширения проще (не нужно беспокоиться о race conditions)
• Историческое наследие CPython

Как работает:
• Поток захватывает GIL перед выполнением байт-кода
• Освобождает при:
  - Системных вызовах (I/O)
  - Каждые `sys.getswitchinterval()` секунд (по умолчанию 5ms)
  - При явном yield в генераторах/корутинах

Последствия:
```
```python
# CPU-bound задача: потоки не дают ускорения из-за GIL
def cpu_heavy():
    return sum(i*i for i in range(10_000_000))

threads = [Thread(target=cpu_heavy) for _ in range(4)]
# Запустится последовательно, не параллельно!

# I/O-bound задача: потоки эффективны, т.к. GIL освобождается при ожидании
def io_heavy():
    return requests.get("https://api.example.com/data")

threads = [Thread(target=io_heavy) for _ in range(10)]
# Параллельные запросы, т.к. GIL освобождается во время network wait
```

Обход GIL:
• `multiprocessing` — отдельные процессы с собственной памятью и GIL
• C-расширения, освобождающие GIL (numpy, PIL)
• Альтернативные интерпретаторы (PyPy STM — экспериментальный, Jython, IronPython)

**DevOps-контекст**: Для CPU-bound автоматизаций (парсинг, вычисления) используй `multiprocessing` или `concurrent.futures.ProcessPoolExecutor`. Для I/O-bound (API, БД, файлы) — `threading` или `asyncio`.

**Связанные инструменты**: `multiprocessing`, `concurrent.futures`, `threading`, `asyncio`.

### 12. Разница между threading, multiprocessing и asyncio: когда что выбирать?
**Суть**: Три модели конкурентности с разными компромиссами по производительности, сложности и использованию ресурсов.

**Подробно**:
| Модель | Параллелизм | Память | Переключение | Лучше всего для |
|--------|------------|--------|-------------|-----------------|
| **threading** | Псевдопараллелизм (GIL) | Общая | OS-level, с GIL | I/O-bound задачи, простые сценарии |
| **multiprocessing** | Настоящий параллелизм | Раздельная (IPC) | OS-level | CPU-bound задачи, изоляция сбоев |
| **asyncio** | Кооперативная многозадачность | Общая | В event loop, явный yield | Высококонкурентные I/O задачи (1000+ соединений) |

**Примеры выбора**:
```python
# Threading: 10 параллельных запросов к API
from threading import Thread
def fetch(url): ...
threads = [Thread(target=fetch, args=(url,)) for url in urls]

# Multiprocessing: параллельная обработка данных
from multiprocessing import Pool
def process(data_chunk): ...
with Pool(4) as pool:
    results = pool.map(process, data_chunks)

# Asyncio: 10000 одновременных health checks
import asyncio, httpx
async def check(url): async with httpx.AsyncClient() as c: ...
asyncio.run(asyncio.gather(*(check(u) for u in urls)))
```

**DevOps-контекст**: 
• Threading — просто, но ограничено GIL; подходит для скриптов с умеренной конкурентностью
• Multiprocessing — мощно, но накладные расходы на IPC; для тяжёлых вычислений
• Asyncio — максимально эффективно для I/O, но требует async/await во всём стеке

### 13. Как работает async/await: event loop, корутины, задачи?
**Суть**: Асинхронная модель на основе кооперативной многозадачности: корутины явно отдают управление при ожидании.

**Подробно**:
```
Ключевые компоненты:
1. Event loop — центральный цикл, управляющий выполнением корутин
2. Coroutine — функция, определённая через async def, возвращает coroutine object
3. Await — точка приостановки: корутина ждёт результат и отдаёт управление loop'у
4. Task — обёртка над корутиной для планирования выполнения в loop'е

Жизненный цикл:
```
```
1. async def fetch(): ...  # определение корутины
2. coro = fetch()          # создание объекта корутины (не выполняется!)
3. task = loop.create_task(coro)  # планирование выполнения
4. loop.run_until_complete(task)  # запуск loop'а до завершения задачи

Пример с несколькими await:
```
```python
async def deploy_service(service):
    config = await fetch_config(service)      # ждём конфиг, отдаём управление
    await validate(config)                     # ждём валидацию
    result = await execute_deploy(config)      # ждём деплой
    await notify(result)                       # ждём уведомление
    return result

# Параллельное выполнение:
services = ["api", "web", "worker"]
results = await asyncio.gather(*(deploy_service(s) for s in services))
# Все деплои запускаются "параллельно", await переключается между ними при I/O
```

Преимущества:
• Эффективное использование одного потока для тысяч соединений
• Предсказуемое переключение (нет race conditions на уровне Python-кода)
• Явные точки ожидания (await) упрощают отладку

Ограничения:
• Блокирующий код (time.sleep, sync I/O) блокирует весь loop
• Все зависимости в цепочке вызовов должны быть async-friendly

**DevOps-контекст**: Asyncio идеален для health checks, сбора метрик, параллельных деплоев, обработки вебхуков. Но требует дисциплины: нельзя использовать синхронные библиотеки внутри async-кода без executors.

### 15. Как работает система исключений: иерархия, traceback, контекст?
**Суть**: Механизм обработки ошибок через иерархию исключений, сохранение контекста и стек вызовов.

**Подробно**:
```
Иерархия исключений:
```
```
BaseException
├── SystemExit, KeyboardInterrupt, GeneratorExit  # не ловить общим except
└── Exception  # базовый класс для "обычных" исключений
    ├── ValueError, TypeError, KeyError, ...  # встроенные
    └── CustomError  # пользовательские
```

Traceback (стек вызовов):
• Сохраняется при выбросе исключения
• Доступен через `sys.exc_info()` или `__traceback__` атрибут
• Форматируется через `traceback.format_exc()`

Контекст исключений:
• `raise ... from ...` — явная причинно-следственная связь
• `__context__` — неявное предыдущее исключение
• `__cause__` — явная причина (через `from`)

Пример:
```
```python
try:
    config = load_config()
except FileNotFoundError as e:
    raise ConfigError("Config not found") from e  # e будет __cause__

# При обработке:
try:
    ...
except ConfigError as e:
    if e.__cause__:
        log.error(f"Root cause: {e.__cause__}")
```

Best practices:
• Лови конкретные исключения, а не `except:`
• Используй `raise ... from ...` для сохранения контекста
• Не глотай исключения молча: логируй или перебрасывай

**DevOps-контекст**: Правильная обработка исключений критична для надёжности автоматизаций: предотвращает молчаливые сбои, сохраняет контекст для отладки, позволяет реализовать retry/circuit breaker.

### 16. Что такое дескрипторы и как они используются в фреймворках?
**Суть**: Дескриптор — объект, реализующий протокол __get__/__set__/__delete__, контролирующий доступ к атрибутам класса.

**Подробно**:
```
Протокол дескриптора:
• __get__(self, obj, objtype=None) — при чтении атрибута
• __set__(self, obj, value) — при записи (делает дескриптор "data descriptor")
• __delete__(self, obj) — при удалении

Пример: валидация значения
```
```python
class PositiveInt:
    def __set_name__(self, owner, name):
        self.name = name  # имя атрибута в классе
    
    def __get__(self, obj, objtype=None):
        return obj.__dict__.get(self.name)
    
    def __set__(self, obj, value):
        if not isinstance(value, int) or value <= 0:
            raise ValueError(f"{self.name} must be positive int")
        obj.__dict__[self.name] = value

class DeployConfig:
    replicas = PositiveInt()  # дескриптор применяется ко всем экземплярам
    
    def __init__(self, replicas):
        self.replicas = replicas  # вызовет __set__

cfg = DeployConfig(3)  # OK
cfg.replicas = -1      # ValueError
```

Использование во фреймворках:
• SQLAlchemy: Column — дескриптор для маппинга на БД
• Django: Field — валидация и сериализация полей моделей
• Pydantic: Field — валидация через type hints + дескрипторы

**DevOps-контекст**: Дескрипторы позволяют создавать декларативные конфиги с валидацией "из коробки". Полезны для построения DSL для автоматизации.

### 17. Как работает сериализация: pickle, json, msgpack trade-offs?
**Суть**: Преобразование объектов Python в байтовый поток для хранения или передачи, с разными компромиссами.

**Подробно**:
| Формат | Типы данных | Безопасность | Скорость | Размер | Когда использовать |
|--------|------------|-------------|----------|--------|-------------------|
| **pickle** | Любые Python-объекты | ❌ (code execution) | Быстро | Средний | Кэширование внутри доверенной системы |
| **json** | Базовые типы (dict, list, str, num, bool, null) | ✅ | Медленно | Большой | API, конфиги, межъязыковой обмен |
| **msgpack** | Как JSON + binary, немного расширенные типы | ✅ | Очень быстро | Маленький | Высокопроизводительный бинарный обмен |
| **protobuf** | Строго типизированная схема | ✅ | Очень быстро | Очень маленький | gRPC, высоконагруженные сервисы |

**Примеры**:
```python
# Pickle — только для доверенных данных!
import pickle
cache = {"state": complex_state_object}
with open("cache.pkl", "wb") as f:
    pickle.dump(cache, f)  # ⚠️ не загружай pickle из ненадёжных источников!

# JSON — безопасный, но только базовые типы
import json
config = {"env": "prod", "replicas": 3}

    json.dump(config, f)  # Человекочитаемый, межъязыковой

# Msgpack — бинарный, эффективный
import msgpack
metrics = {"cpu": 0.75, "mem": 2048}
packed = msgpack.packb(metrics)  # компактный байтовый поток
```

**DevOps-контекст**: 
• Никогда не используй `pickle` для данных извне (риск RCE)
• JSON — стандарт для конфигов и API
• Msgpack/protobuf — для внутренних высокопроизводительных коммуникаций (очереди, кэш)

### 19. Как устроено тестирование: unit vs integration, mocking, fixtures?
**Суть**: Уровни тестирования с разными целями, изоляцией и скоростью выполнения.

**Подробно**:
```
Уровни тестирования:
1. Unit tests:
   • Тестируют одну функцию/класс в изоляции
   • Мокают все внешние зависимости (БД, API, файлы)
   • Быстрые, детерминированные, много тестов

2. Integration tests:
   • Тестируют взаимодействие нескольких компонентов
   • Используют реальные или тестовые БД, API, файловую систему
   • Медленнее, но ловят проблемы интеграции

3. End-to-end (E2E) tests:
   • Тестируют полный сценарий от входа до выхода
   • Запускают реальное приложение в тестовом окружении
   • Самые медленные, но ближе всего к production

Mocking:
• Подмена зависимостей для изоляции тестируемого кода
• `unittest.mock.Mock` / `patch` — стандартная библиотека
• Проверяют, что функция вызывалась с ожидаемыми аргументами

Fixtures (pytest):
• Переиспользуемая настройка/очистка для тестов
• Поддерживают области видимости: function, class, module, session
• Автоматическая инъекция через аргументы тест-функций

Пример:
```
```python
# conftest.py

### 20. Что такое dependency injection и зачем он нужен в автоматизации?
**Суть**: Паттерн, при котором зависимости передаются извне, а не создаются внутри, для улучшения тестируемости и гибкости.

**Подробно**:
```
Без DI (жесткая связность):
```
```python
class Deployer:
    def deploy(self, service):
        db = PostgreSQL()  # жёстко зашита зависимость
        api = SlackAPI()   # сложно заменить для тестов
        ...
```

С ручной инъекцией:
```
```python
class Deployer:
    def __init__(self, db, notifier):
        self.db = db
        self.notifier = notifier
    
    def deploy(self, service):
        self.db.save(...)      # теперь можно подменить в тестах
        self.notifier.send(...)

# Использование:
deployer = Deployer(PostgreSQL(), SlackAPI())
# В тестах:
deployer = Deployer(MockDB(), MockNotifier())
```

Контейнеры DI (для сложных проектов):
• `injector`, `dependency-injector` — автоматическое разрешение зависимостей
• Определяешь граф зависимостей, фреймворк собирает объекты
• Полезно при сотнях сервисов и конфигураций

Преимущества:
• Тестируемость: легко мока зависимости
• Гибкость: можно подменить реализацию без изменения кода
• Читаемость: явные зависимости вместо скрытых импортов

**DevOps-контекст**: В автоматизациях с множеством интеграций (БД, API, очереди, хранилища) DI предотвращает "спагетти-код" и упрощает тестирование. Начинай с ручной инъекции, переходи к контейнерам при росте сложности.

### 21. ⭐ Детально: как работает CPython: bytecode, VM, оптимизации?
**Суть**: CPython компилирует исходный код в байт-код, который исполняется виртуальной машиной на основе стека.

**Подробно**:
```
Компиляция:
1. Исходный код (.py) → парсинг → AST (Abstract Syntax Tree)
2. AST → оптимизации (constant folding, dead code elimination)
3. AST → байт-код (.pyc в __pycache__)
4. Байт-код — платформенно-независимая последовательность инструкций

Виртуальная машина:
• Стек-машина: операнды берутся из стека, результат кладётся в стек
• Пример байт-кода для `x = 1 + 2`:
  ```
  LOAD_CONST 1      # положить 1 в стек
  LOAD_CONST 2      # положить 2 в стек
  BINARY_ADD        # сложить два верхних, результат в стек
  STORE_NAME 'x'    # взять из стека, сохранить в x
  ```
• Интерпретатор: цикл fetch-decode-execute для каждой инструкции

Оптимизации CPython:
• Peephole optimizer: упрощает байт-код на этапе компиляции
  ```python
  # Исходный код
  if True: ...
  # Байт-код: условие убирается, остаётся только тело
  
  # Исходный код
  x = 2 + 3 * 4
  # Байт-код: x = 14 (constant folding)
  ```
• Method caching: кэширование поиска методов в словаре класса
• Adaptive interpreter (3.11+): специализация байт-кода под типы аргументов

Профилирование байт-кода:
```
```python
import dis
def example():
    return [x*2 for x in range(10)]

dis.dis(example)  # вывод байт-кода функции
```

### 22. Как работает garbage collector: generations, cycles, tuning?
**Суть**: Поколенческий сборщик мусора для обработки циклических ссылок, которые не ловит reference counting.

**Подробно**:
```
Поколения (generational GC):
• Generation 0: новые объекты, проверяются чаще всего
• Generation 1: пережившие несколько сборок в gen 0
• Generation 2: долгоживущие объекты, проверяются реже

Пороги срабатывания:
```
```python
import gc
gc.get_threshold()  # (700, 10, 10) по умолчанию
# Сборка gen 0: когда #allocations - #deallocations > 700
# Сборка gen 1: после 10 сборок gen 0
# Сборка gen 2: после 10 сборок gen 1
```

Обнаружение циклов:
• Алгоритм на основе подсчёта ссылок между объектами
• Объекты, недостижимые из корней (globals, stack), помечаются как мусор
• Для объектов с __del__ — особая обработка (могут быть "неубиваемыми")

Тюнинг для production:
```
```python
# Увеличить пороги, если много короткоживущих объектов (снижает overhead GC)
gc.set_threshold(1000, 20, 20)

# Отключить автоматическую сборку для критичных по latency участков
gc.disable()
# ... критичный код ...
gc.collect()  # явная сборка в безопасный момент
gc.enable()

# Отладка: какие объекты не собираются?
gc.set_debug(gc.DEBUG_SAVEALL)
gc.collect()
unreachable = gc.garbage  # список несобранных объектов
```

**DevOps-контекст**: В долгоживущих процессах (демоны, операторы) неправильный тюнинг GC может вызывать паузы или утечки памяти. Профилируй память в production, настраивай пороги под паттерны использования.

**Суть**: Систематический подход к поиску и устранению производительностных проблем: измерение → анализ → оптимизация → валидация.

**Подробно**:
```
Инструменты профилирования:
1. CPU profiling:
   • cProfile — детерминированный профилировщик, показывает время по функциям
   • py-spy — sampling profiler, работает без изменения кода, production-safe
   • line_profiler — построчное профилирование

2. Memory profiling:
   • tracemalloc — отслеживание выделений памяти
   • memory_profiler — построчное потребление памяти

```
```

2. Найди горячие точки (cProfile/py-spy)

• Блокирующий код в async → выноси в run_in_executor

```

### 25. Архитектурные паттерны для надёжной автоматизации: retry, circuit breaker, idempotency
**Суть**: Паттерны повышения отказоустойчивости распределённых систем, адаптированные для скриптов автоматизации.

**Подробно**:
```
Retry с экспоненциальной задержкой:
• Повторяет операцию при временных сбоях (таймауты, 5xx, network errors)
• Exponential backoff: задержка растёт как 2^attempt (предотвращает лавину)
• Jitter: случайная добавка к задержке (предотвращает синхронизацию повторных попыток)

Circuit breaker:
• Состояния: Closed (норма) → Open (сбой, быстрые отказы) → Half-Open (проверка восстановления)
• Открывается после N последовательных сбоев
• Позволяет системе "остыть" и предотвращает каскадные сбои

Idempotency:
• Операция даёт одинаковый результат при многократном выполнении с теми же входными данными
• Реализация: идемпотентность ключ (request ID), проверка состояния перед выполнением
• Критично для операций с побочными эффектами (деплой, оплата, создание ресурсов)

Пример реализации:
```
```python
from tenacity import retry, stop_after_attempt, wait_exponential
import hashlib

class IdempotentDeploy:
    def __init__(self, state_store):
        self.state = state_store  # Redis/DB для хранения статуса
    
    def _idempotency_key(self, service, version):
        return hashlib.sha256(f"{service}:{version}".encode()).hexdigest()
    
    @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1))
    def execute(self, service, version):
        key = self._idempotency_key(service, version)
        
        # Проверка идемпотентности
        if self.state.get(key) == "done":
            return {"status": "already-done"}
        
        # Выполнение деплоя
        result = self._do_deploy(service, version)
        
        # Фиксация результата
        self.state.set(key, "done", ttl=3600)
        return result
```

**DevOps-контекст**: Эти паттерны отличают production-ready автоматизации от скриптов "на коленке". Интегрируй retry/circuit breaker во все внешние вызовы, обеспечивай идемпотентность для операций с побочными эффектами.

### 26. Как работает multiprocessing: fork vs spawn, shared memory, queues?
**Суть**: Механизм создания параллельных процессов с изолированной памятью и средствами межпроцессного взаимодействия.

**Подробно**:
```
Методы запуска процессов (start_method):
• fork (Linux default): копирует память родителя через copy-on-write
  + Быстро, разделяет память до первой записи
  - Может копировать нежелательные ресурсы (открытые файлы, соединения)
  - Не работает на Windows

• spawn (Windows default, кроссплатформенный): запускает свежий интерпретатор
  + Чистое окружение, предсказуемое поведение
  - Медленнее (запуск нового процесса + импорт модулей)

• forkserver (Python 3.7+): гибридный подход
  + Быстрее spawn, безопаснее fork
  - Требует предварительной настройки сервера

Межпроцессное взаимодействие (IPC):
• Queue — потокобезопасная очередь для обмена данными
• Pipe — двусторонний канал для связи двух процессов
• Manager — высокоуровневый API для разделяемых структур (dict, list)
• SharedMemory (3.8+) — разделяемая память для больших массивов

Пример:
```
```python
from multiprocessing import Process, Queue, set_start_method

def worker(input_q, output_q):
    while True:
        task = input_q.get()
        if task is None:  # сигнал завершения
            break
        result = process(task)
        output_q.put(result)

if __name__ == "__main__":
    set_start_method("spawn")  # явно указать метод для кроссплатформенности
    
    in_q, out_q = Queue(), Queue()
    processes = [Process(target=worker, args=(in_q, out_q)) for _ in range(4)]
    
    for p in processes: p.start()
    for task in tasks: in_q.put(task)
    for _ in processes: in_q.put(None)  # сигналы завершения
    
    results = [out_q.get() for _ in tasks]
    for p in processes: p.join()
```

**DevOps-контекст**: Multiprocessing обходит GIL для CPU-bound задач, но добавляет overhead на сериализацию и IPC. Используй для параллельной обработки данных, вычислений, тестов. Для I/O-bound задач предпочитай asyncio.

### 27. Безопасность Python-кода: sandboxing, code injection, supply chain risks
**Суть**: Угрозы безопасности в экосистеме Python и методы их смягчения.

**Подробно**:
```
Код-инъекции:
• eval/exec/compile с пользовательским вводом → выполнение произвольного кода
• pickle.loads() с ненадёжными данными → code execution через __reduce__
• subprocess с shell=True и непроверенными аргументами → command injection

Меры защиты:
```
```python
# ❌ Опасно:
eval(user_input)
pickle.loads(untrusted_data)

# ✅ Безопасно:

ast.literal_eval(user_input)  # только литералы (dict, list, str, num)

import json

subprocess.run(["ls", user_path])  # список аргументов вместо строки
```

Supply chain атаки:
• Зависимости от ненадёжных пакетов в PyPI
• Typosquatting: пакеты с похожими именами (requets вместо requests)
• Компрометация мейнтейнеров, внедрение малвари в обновления

Защита:
• Фиксируй точные версии зависимостей (poetry.lock, requirements.txt)
• Сканируй на уязвимости: `pip-audit`, `safety`, `osv-scanner`
• Проверяй хеши пакетов: `pip install --require-hashes`
• Используй приватный PyPI-прокси с вайтлистом (Artifactory, Nexus)

Sandboxing (ограниченное выполнение):
• `restrictedpython` — безопасный подмножество Python
• Контейнеризация: запуск недоверенного кода в Docker с ограничениями
• Seccomp, namespaces, cgroups — изоляция на уровне ОС

**DevOps-контекст**: Безопасность автоматизаций критична: скрипты часто работают с высокими привилегиями. Валидируй весь внешний ввод, сканируй зависимости, изолируй выполнение недоверенного кода.

### 29. Альтернативные интерпретаторы: PyPy, Jython, IronPython — когда рассматривать?

### 30. ⭐ Принципы проектирования масштабируемых платформ автоматизации на Python
**Суть**: Архитектурные принципы для создания надёжных, поддерживаемых и расширяемых систем автоматизации.

```

Принцип 2: Конфигурация как код
• Валидация конфигов на старте (Pydantic, jsonschema)
• Иерархия источников: defaults → file → env → CLI args
• Секреты — отдельно, через Vault/Secrets Manager

Принцип 3: Надёжность по умолчанию
• Retry с экспоненциальной задержкой для всех внешних вызовов
• Circuit breaker для предотвращения каскадных сбоев
• Идемпотентность для операций с побочными эффектами
• Таймауты на всех уровнях (network, DB, external API)

Принцип 4: Наблюдаемость (Observability)
• Структурированное логирование (JSON) с корреляционными ID
• Метрики (Prometheus) для ключевых операций: длительность, ошибки, очередь

• Высокое покрытие unit-тестами для бизнес-логики
• Integration-тесты с testcontainers для внешних зависимостей
• Контрактные тесты для интеграций между сервисами
• Версионирование API и конфигов, миграции данных

Принцип 6: Безопасность и комплаенс
• Валидация всего внешнего ввода
• Сканирование зависимостей на уязвимости в CI
• Минимальные привилегии для runtime-окружения
• Аудит критичных операций

**DevOps-контекст**: Эти принципы отличают enterprise-платформы от скриптов. Инвестиции в архитектуру окупаются при масштабировании команды и функциональности. Начинай с малого, но закладывай возможность роста.
