### 1. В чём разница между SQL и NoSQL? Приведите примеры использования

**Ответ**:
```
SQL (реляционные БД):
• Данные в таблицах со строгой схемой
• Язык запросов: стандартизированный SQL
• Гарантии ACID
• Вертикальное масштабирование (чаще)
• Примеры: PostgreSQL, MySQL, Oracle

NoSQL (нереляционные БД):
• Гибкая схема, разные модели данных
• Документные (MongoDB), key-value (Redis), column-family (Cassandra), graph (Neo4j)
• Гарантии BASE / eventual consistency
• Горизонтальное масштабирование
• Примеры: MongoDB, Redis, DynamoDB, Cassandra
```

**Когда что использовать**:
```
→ SQL: финансовые транзакции, заказы, пользователи (где важна консистентность)
→ NoSQL: кэш (Redis), логи/аналитика (MongoDB), сессии (Redis), IoT-данные (TimescaleDB)
```

**DevOps-контекст**: Выбор БД влияет на архитектуру бэкапов, репликации, мониторинга и масштабирования.

### 2. Что такое первичный ключ и зачем он нужен?

**Ответ**:
```
Первичный ключ (PRIMARY KEY):
• Уникально идентифицирует каждую строку в таблице
• Не может содержать NULL
• В таблице может быть только один PK
• Обычно создаёт кластерный индекс (данные физически сортируются по ключу)

Зачем нужен:
• Гарантия уникальности записей
• Быстрый поиск по ключу (индекс)
• Основа для внешних ключей (связи между таблицами)
• Используется в репликации для идентификации строк
```

**Пример**:
```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,  -- автоинкрементный PK
    email VARCHAR(100) UNIQUE NOT NULL,  -- тоже уникален, но не PK
    name VARCHAR(50)
);
```

**DevOps-контекст**: При репликации и шардировании первичный ключ часто используется как distribution key — неправильный выбор ведёт к перекосу данных.

### 3. Как работает INDEX и почему он ускоряет запросы?

**Ответ**:
```
Индекс — структура данных (обычно B-tree), которая хранит отсортированные значения колонки + ссылки на строки.

Без индекса:
• Запрос WHERE email = 'a@b.com' → полный проход по таблице (Seq Scan) → O(n)

С индексом:
• Поиск по B-tree → O(log n) → сразу находим нужную строку

Типы индексов:
• B-tree: для =, <, >, BETWEEN, LIKE 'prefix%' (дефолт)
• Hash: только для точного равенства
• Composite: по нескольким колонкам (важен порядок!)
• Partial: индекс только для подмножества строк (WHERE status='active')
```

**Важно**:
```
✅ Индексы ускоряют SELECT
❌ Но замедляют INSERT/UPDATE/DELETE (нужно обновлять индекс)
❌ Занимают место на диске и в памяти
```

**DevOps-контекст**: Мониторьте использование индексов через `pg_stat_user_indexes`. Неиспользуемые индексы — лишний overhead.

### 4. Что такое транзакция? Объясните на примере

**Ответ**:
```
Транзакция — последовательность операций, выполняемых как единое целое: либо все, либо ничего.

Пример: перевод денег
```
```sql
BEGIN;  -- начало транзакции

UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- списали
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- зачислили

-- Если всё ОК:
COMMIT;  -- фиксируем изменения

-- Если ошибка:
ROLLBACK;  -- отменяем ВСЕ изменения в транзакции
```

**Свойства транзакции (ACID)**:
- Атомарность: всё или ничего
- Консистентность: БД остаётся валидной
- Изоляция: параллельные транзакции не мешают друг другу
- Долговечность: после COMMIT данные не потеряются

**DevOps-контекст**: Долгие транзакции блокируют ресурсы и мешают VACUUM. Минимизируйте их длительность в автоматизациях.

### 5. В чём разница между INNER JOIN, LEFT JOIN, RIGHT JOIN?

**Ответ**:
```
Две таблицы: users (id, name) и orders (id, user_id, amount)

INNER JOIN:
• Только строки, где есть совпадение в обеих таблицах
```
```sql
SELECT u.name, o.amount 
FROM users u 
INNER JOIN orders o ON u.id = o.user_id;
-- Результат: только пользователи, у которых есть заказы
```

```
LEFT JOIN (LEFT OUTER JOIN):
• Все строки из левой таблицы + совпадения из правой
• Если нет совпадения → NULL в полях правой таблицы
```
```sql
SELECT u.name, o.amount 
FROM users u 
LEFT JOIN orders o ON u.id = o.user_id;
-- Результат: все пользователи, даже без заказов (amount = NULL)
```

```
RIGHT JOIN:
• Все строки из правой таблицы + совпадения из левой
• Используется редко (можно заменить на LEFT JOIN с перестановкой таблиц)
```

**DevOps-контекст**: При репликации и шардировании JOIN между таблицами на разных узлах может быть очень медленным — проектируйте схему с учётом этого.

### 6. Что такое нормализация и денормализация?

**Ответ**:
```
Нормализация — процесс организации данных для минимизации дублирования:
• 1НФ: атомарные значения, нет повторяющихся групп
• 2НФ: все неключевые поля зависят от всего первичного ключа
• 3НФ: нет транзитивных зависимостей (неключевые поля не зависят друг от друга)

Преимущества:
✅ Меньше дублирования → меньше места, проще обновлять
✅ Целостность данных (изменил в одном месте — везде обновилось)

Денормализация — намеренное дублирование данных для ускорения чтения:
• Добавляем вычисляемые поля (order_total вместо SUM каждый раз)
• Дублируем данные из связанных таблиц (user_name прямо в orders)

Преимущества:
✅ Быстрее чтение (меньше JOIN)
✅ Проще запросы

Недостатки:
❌ Риск несогласованности (обновил в одном месте, забыл в другом)
❌ Больше места
```

**DevOps-контекст**: В аналитических системах (OLAP) часто используют денормализацию для скорости. В транзакционных (OLTP) — нормализацию для целостности.

### 8. Что такое ACID? Расшифруйте каждую букву

**Ответ**:
```
A — Atomicity (Атомарность):
• Транзакция выполняется целиком или не выполняется вообще
• Пример: если при переводе денег зачисление упало — списание тоже отменится

C — Consistency (Консистентность):
• Транзакция переводит БД из одного валидного состояния в другое
• Все ограничения (PK, FK, UNIQUE, CHECK) соблюдаются

I — Isolation (Изоляция):
• Параллельные транзакции не влияют друг на друга
• Уровни: Read Committed (дефолт в PG) → Repeatable Read → Serializable

D — Durability (Долговечность):
• После COMMIT изменения не теряются даже при сбое питания
• Реализация: WAL + fsync на диск
```

**Пример нарушения**:
```
Без атомарности: списали деньги, но не зачислили → потеря средств
Без долговечности: после перезагрузки пропали последние заказы
```

**DevOps-контекст**: Для финансовых операций выбирайте БД с полной поддержкой ACID. Настройка `synchronous_commit=on` в PostgreSQL гарантирует durability.

### 9. Как сделать резервную копию БД и восстановить её?

**Ответ**:
```
PostgreSQL — logical backup:
```
```bash
# Создать бэкап
pg_dump -U postgres myapp > backup.sql

# Сжатие
pg_dump -U postgres myapp | gzip > backup.sql.gz

# Восстановление
psql -U postgres -d myapp < backup.sql
# или для gz:
gunzip backup.sql.gz
psql -U postgres -d myapp < backup.sql
```

```
PostgreSQL — physical backup:
```
```bash
# Создать
pg_basebackup -U postgres -D /backup/myapp -Fp -Xs -P

# Восстановление: скопировать файлы в data directory и запустить БД
```

```
MySQL:
```
```bash
# Бэкап
mysqldump -u root -p myapp > backup.sql

# Восстановление
mysql -u root -p myapp < backup.sql
```

```
Redis:
```
```bash
# RDB snapshot (автоматически создаётся)
# Копируем файл:
cp /var/lib/redis/dump.rdb /backup/

# Восстановление: положить dump.rdb в data directory и перезапустить Redis
```

**DevOps-контекст**: 
- Регулярно тестируйте восстановление — бэкап без проверки не считается
- Храните бэкапы отдельно от сервера БД (иначе при падении диска потеряете всё)
- Автоматизируйте и добавьте алерты при неудаче

### 10. Что такое connection string и из чего он состоит?

**Ответ**:
```
Connection string — строка подключения, содержащая все параметры для установления соединения с БД.

Пример для PostgreSQL:
```
```
postgresql://user:password@host:port/database?param1=value1&param2=value2
```

**Компоненты**:
```
• Protocol: postgresql://, mysql://, redis://
• User: username для аутентификации
• Password: пароль (лучше передавать через env var, не в строке!)
• Host: адрес сервера (можно несколько через запятую для failover)
• Port: порт (5432 для PG, 3306 для MySQL, 6379 для Redis)
• Database: имя базы данных
• Параметры:
  - sslmode=require/verify-full (шифрование)
  - connect_timeout=5 (таймаут подключения)
  - target_session_attrs=read-write (подключаться только к master)
  - application_name=myapp (для мониторинга)
```

**Пример с несколькими хостами (failover)**:
```
postgresql://user:pass@host1:5432,host2:5432/myapp?target_session_attrs=read-write
```

**DevOps-контекст**: 
- Храните connection strings в secrets manager, не в коде
- Используйте переменные окружения: `DATABASE_URL=${DB_URL}`
- Настройте retry logic в приложении на случай временных сбоев подключения

### 11. ⭐ Как работает репликация в PostgreSQL? В чём разница между синхронной и асинхронной?

**Ответ**:
```
Streaming replication в PostgreSQL:
• Master записывает изменения в WAL (Write-Ahead Log)
• WAL sender process отправляет новые записи подключённым репликам
• Реплика (standby) применяет WAL к локальным данным
• При hot_standby=on можно выполнять read-only запросы на реплике

Асинхронная репликация (дефолт):
```
```
Master: записал → отправил реплике → ответил клиенту (не ждёт подтверждения)
✅ Низкая задержка записи
❌ Возможна потеря данных при падении master (данные не успели уйти на реплику)
```

```
Синхронная репликация:
```
```
Master: записал → отправил → ЖДЁТ подтверждения от реплики → ответил клиенту
✅ Нулевая потеря данных (RPO=0)
❌ Задержка записи растёт с сетевой задержкой и количеством синхронных реплик
❌ Если реплика недоступна — запись блокируется (или переключается на асинхронную)
```

**Настройка**:
```conf
# postgresql.conf на master
synchronous_commit = on
synchronous_standby_names = 'standby1'
```

**DevOps-контекст**: 
- Для финансовых данных — синхронная репликация
- Для масштабирования чтения — асинхронные реплики
- Мониторьте `pg_stat_replication` и replication lag
- Для автоматического failover используйте Patroni

### 12. Что такое WAL и зачем он нужен?

**Ответ**:
```
WAL (Write-Ahead Logging) — журнал предварительной записи.

Как работает:
1. Любое изменение данных сначала записывается в WAL
2. Только потом применяется к самим данным
3. После COMMIT запись в WAL считается гарантированной (fsync)

Зачем нужен:
✅ Долговечность: после краша БД воспроизводит WAL и восстанавливает данные
✅ Репликация: slave читает WAL от master и применяет изменения
✅ Point-in-time recovery: можно восстановить БД на любой момент, применяя WAL до нужной точки

Структура:
• Файлы в pg_wal/ (ранее pg_xlog)
• Размер сегмента: 16 МБ по умолчанию
• Каждый рекорд имеет уникальный LSN (Log Sequence Number)
```

**DevOps-контекст**: 
- Следите за местом в pg_wal/ — переполнение остановит БД
- Настройте archive_command для PITR
- WAL — основа репликации, не отключайте и не удаляйте вручную

### 13. Как диагностировать медленный запрос? Какие инструменты использовать?

**Ответ**:
```
Шаги диагностики:
1. Найти медленный запрос:
   - Включить slow query log: log_min_duration_statement = 1000 (мс)
   - Использовать pg_stat_statements для агрегированной статистики

2. Проанализировать план выполнения:
```
```sql
EXPLAIN ANALYZE SELECT ...;  -- покажет план + фактическое время
```

3. Интерпретировать вывод:
   - Seq Scan на большой таблице → нужен индекс
   - Nested Loop с большим числом итераций → рассмотреть Hash Join
   - Sort с disk → увеличить work_mem
   - Высокое время на "Buffers: read" → нехватка кэша

4. Проверить статистику:
```
```sql
ANALYZE table_name;  -- обновить статистику для оптимизатора
```

5. Проверить блокировки:
```
```sql
SELECT * FROM pg_locks WHERE NOT granted;
```

**Инструменты**:
• `EXPLAIN (ANALYZE, BUFFERS)` — детальный план

### 14. Что такое MVCC и как это влияет на производительность?

**Ответ**:
```
MVCC (Multi-Version Concurrency Control) — механизм управления параллельным доступом.

Как работает:
• При UPDATE/DELETE строка не удаляется физически, а помечается как "устаревшая"
• Создаётся новая версия строки с новым xmin/xmax (ID транзакций)
• Читающие транзакции видят "снимок" данных на момент своего начала
• Пишущие не блокируют читающих и наоборот

Преимущества:
✅ Высокая конкурентность: читатели и писатели не блокируют друг друга
✅ Консистентные снимки для длительных запросов (отчёты, экспорт)

Проблемы:
❌ "Раздувание" таблиц: старые версии занимают место
❌ Требуется VACUUM для очистки мёртвых строк
❌ Долгие транзакции блокируют очистку → рост bloat

Мониторинг:
```
```sql
-- Мёртвые строки
SELECT relname, n_dead_tup, n_live_tup 
FROM pg_stat_user_tables 
ORDER BY n_dead_tup DESC;

-- Долгие транзакции, блокирующие vacuum
SELECT pid, now() - xact_start AS duration, query 
FROM pg_stat_activity 
WHERE state != 'idle' AND xact_start < now() - interval '1 hour';
```

**DevOps-контекст**: 
- Не отключайте autovacuum!
- Настройте параметры autovacuum под нагрузку
- Мониторьте dead tuples и bloat — признаки проблем с vacuum

### 15. В чём разница между Redis Sentinel и Redis Cluster?

**Ответ**:
```
Redis Sentinel (High Availability):
• Набор процессов, мониторит master и реплики
• При падении master: голосование → выбор новой реплики как master → переназначение клиентов
• Все данные на одном мастер-узле (реплики — только для чтения)
• Проще в настройке и управлении

Когда использовать:
✅ Нужен HA для одного набора данных
✅ Нагрузка вписывается в память одного узла
✅ Не нужно горизонтальное масштабирование

Redis Cluster (Sharding + HA):
• Данные автоматически распределены по 16384 хэш-слотам
• Каждый узел отвечает за подмножество слотов
• Клиент вычисляет хэш ключа → определяет слот → подключается к нужному узлу
• Каждый шард — master + реплики (как в Sentinel)

Когда использовать:
✅ Данные не помещаются в память одного узла
✅ Нужно масштабировать записи и чтения горизонтально
✅ Готовы к сложностям: cross-slot операции ограничены

Ограничения Cluster:
❌ Команды с несколькими ключами работают только если ключи в одном слоте
❌ Транзакции и Lua-скрипты — только в рамках одного слота
❌ Сложнее бэкапы и отладка

**DevOps-контекст**: 
- Начинайте с Sentinel, переходите на Cluster только при реальной необходимости
- Мониторьте память, лаг репликации, состояние кластера
- Для бэкапов Cluster используйте `redis-cli --cluster backup` или снимки на уровне каждого узла

### 16. Как работает репликация в MongoDB? Что такое oplog?

**Ответ**:
```
Replica set в MongoDB:
• Один primary: принимает все записи
• Несколько secondaries: реплицируют данные, могут обслуживать чтение
• Опционально: arbiter (только голосует), hidden, delayed

Oplog (operations log):
• Специальная capped-коллекция local.oplog.rs
• Хранит все операции записи в порядке применения
• Ограниченный размер (по умолчанию 5% диска)
• Вторичные узлы "догоняют" primary, читая и применяя oplog

Процесс:
1. Клиент пишет на primary → операция применяется → записывается в oplog
2. Secondaries периодически опрашивают primary на наличие новых записей
3. При нахождении: скачивают и асинхронно применяют локально
4. Каждый узел отслеживает прогресс через оптимистичный "timestamp"

Election (выборы нового primary):
• Если primary падает, оставшиеся узлы инициируют голосование
• Кворум: большинство узлов должно проголосовать (избегает split-brain)
• Новый primary начинает принимать записи

**DevOps-контекст**: 
- Размер oplog критичен: если secondary отстанет больше, чем вмещает oplog — потребуется полная ресинхронизация
- Мониторьте через `rs.printReplicationInfo()` и `rs.printSecondaryReplicationInfo()`
- Для кросс-регион репликации учитывайте задержку: настраивайте write concern и read preference

### 17. Что такое point-in-time recovery и как его настроить?

**Ответ**:
```
PITR (Point-in-Time Recovery) — восстановление БД на любой момент в прошлом, а не только на момент последнего бэкапа.

Как работает в PostgreSQL:
1. Base backup: физическая копия данных (pg_basebackup)
2. WAL archiving: настройка archive_command для копирования заполненных WAL-сегментов в безопасное хранилище
3. Восстановление:
   - Разворачиваем base backup
   - Создаём recovery.signal с параметрами:
     • restore_command: как доставать WAL из архива
     • recovery_target_time: до какого момента применять WAL
   - Запускаем БД: она применяет WAL последовательно до указанного момента

Пример настройки архивации:
```
```conf
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'test ! -f /backup/wal/%f && cp %p /backup/wal/%f'
archive_timeout = 300  # форсировать переключение WAL каждые 5 минут
```

**DevOps-контекст**: 
- PITR — must-have для production, где потеря данных недопустима
- Храните архив WAL отдельно от сервера БД
- Автоматизируйте восстановление: скрипты + документация + регулярные учения
- Мониторьте успешность архивации: если WAL не архивируется — PITR не сработает

### 18. Как настроить connection pooling и зачем он нужен?

**Ответ**:
```
Проблема без pooling:
• Установка соединения с БД — дорогая операция (TCP handshake, аутентификация)
• При высокой нагрузке: тысячи соединений → исчерпание лимитов БД, высокая задержка

Как работает pool:
• При старте создаётся пул из N готовых соединений
• При запросе: берётся свободное соединение из пула (не создаётся новое)
• После выполнения: соединение возвращается в пул (не закрывается)

Параметры пула:
```
```python
# SQLAlchemy пример
engine = create_engine(
    "postgresql://user:pass@localhost/db",
    pool_size=10,           # базовое количество соединений
    max_overflow=20,        # сколько можно временно "перелить" сверх pool_size
    pool_timeout=30,        # сколько ждать свободное соединение
    pool_recycle=1800,      # пересоздавать соединение через 30 мин
    pool_pre_ping=True      # проверять соединение перед использованием
)
```

Внешний pooling (PgBouncer):
• Режим session: одно соединение БД на одно клиентское соединение
• Режим transaction: одно соединение БД на много клиентских транзакций (экономнее)

**DevOps-контекст**: 
- Без pooling приложение не масштабируется
- PgBouncer в режиме transaction pooling экономит ресурсы БД
- Настройте `max_connections` в БД с учётом пулов всех приложений
- Мониторьте метрики пула: wait time, overflow, idle connections

### 19. Что такое шардирование и когда его применять?

**Ответ**:
```
Шардирование — горизонтальное разделение одной логической таблицы на физические части (шарды), распределённые по разным узлам.

Стратегии:
1. Range-based: по диапазону ключа (user_id 1-1M → shard1)
   ✅ Простота, диапазонные запросы в рамках одного шарда
   ❌ Неравномерное распределение, сложно добавлять шарды

2. Hash-based: shard = hash(key) % num_shards
   ✅ Равномерное распределение
   ❌ Диапазонные запросы требуют обращения ко всем шардам

3. Directory-based: отдельный сервис хранит маппинг ключ → шард
   ✅ Гибкость
   ❌ Дополнительная сложность, единая точка отказа

Когда применять:
✅ Один узел не вмещает данные (память, диск)
✅ Нагрузка на запись/чтение превышает возможности одного узла
✅ Гео-распределение: данные ближе к пользователям

Проблемы:
❌ JOIN между шардами: либо денормализация, либо координация на уровне приложения
❌ Транзакции между шардами: сложно, медленно
❌ Глобальные уникальные ограничения: требуют координации
❌ Ребалансировка: при добавлении шарда нужно переместить часть данных

**DevOps-контекст**: 
- Шардирование — последняя мера оптимизации, сначала попробуйте вертикальное масштабирование, кэширование
- Выбор distribution key критичен: по нему чаще всего ищут и джойнят
- Мониторьте распределение данных: перекос ведёт к перегрузке отдельных шардов

### 20. Как работают миграции схемы БД? Какие инструменты знаете?

**Ответ**:
```
Миграции — версионируемые скрипты для изменения схемы БД.

Типы:
1. Forward-only: только применение вперёд (up), откат невозможен
2. Reversible: каждая миграция имеет up и down скрипты
3. Transactional: миграция выполняется внутри транзакции (при ошибке — полный откат)

Процесс:
1. Приложение проверяет таблицу миграций (schema_migrations)
2. Находит неприменённые миграции, сортирует по версии
3. Применяет up-скрипты по очереди, записывает версию в таблицу

Инструменты:
• Flyway: простые SQL-скрипты, версия в имени файла (V1__create_users.sql)
• Liquibase: XML/YAML/JSON-декларации, поддержка rollback, сложная логика
• Alembic (Python): код на SQLAlchemy, генерация миграций из моделей
• Django migrations: встроенные, привязаны к ORM

Best practices:
✅ Никогда не меняйте применённые миграции — только добавляйте новые
✅ Тестируйте миграции на staging с продакшен-подобными данными
✅ Разделяйте миграции схемы и миграции данных
✅ Для больших таблиц: используйте online DDL

**DevOps-контекст**: 
- Миграции — часть CI/CD: запускайте при деплое, с проверкой перед применением
- Для zero-downtime деплоя: используйте backward-compatible изменения
- Мониторьте длительность миграций: долгие блокировки могут вызвать downtime

### 21. ⭐ Объясните CAP-теорему. Как она применяется на практике?

**Ответ**:
```
CAP-теорема (Brewer, 2000): в распределённой системе при сетевом разделении (partition) невозможно одновременно гарантировать все три свойства:

C — Consistency (консистентность):
• Все узлы видят одни и те же данные в один момент времени

A — Availability (доступность):
• Каждый запрос получает ответ (успех или ошибка), без гарантии актуальности

P — Partition tolerance (устойчивость к разделению):
• Система работает при потере связи между узлами

При сетевом разделении приходится выбирать:

→ CP-системы (жертвуем доступностью ради консистентности):
• PostgreSQL с синхронной репликацией
• MongoDB с write concern majority
• Отказывают в записи, если нет кворума
• Use case: финансовые данные, где потеря недопустима

→ AP-системы (жертвуем консистентностью ради доступности):
• Cassandra, DynamoDB с eventual consistency
• Принимают записи на всех узлах, разрешают конфликты потом
• Use case: соцсети, каталоги, где допустима временная несогласованность

Типы консистентности:
• Strong: операции видны всем мгновенно в глобальном порядке
• Eventual: при отсутствии новых записей все реплики со временем станут одинаковыми
• Causal: сохраняется причинно-следственная связь

**DevOps-контекст**: 
- Выбор между CP/AP зависит от бизнес-требований
- Eventual consistency требует изменений в приложении: idempotent операции, версионирование
- Мониторьте replication lag и консистентность
- Документируйте гарантии вашей БД: что приложение может ожидать при сбоях

### 24. В чём разница между eventual consistency и strong consistency?

**Ответ**:
```
Strong consistency (линеаризуемость):
• После успешной записи все последующие чтения (с любого узла) вернут новое значение
• Операции видны всем узлам мгновенно в глобальном порядке
• Реализация: блокировки, кворумы, синхронная репликация
• Цена: высокая задержка записи, снижение доступности при сетевых проблемах

Eventual consistency:
• При отсутствии новых записей все реплики со временем станут одинаковыми
• В момент времени разные узлы могут возвращать разные значения
• Реализация: асинхронная репликация, gossip-протоколы
• Цена: приложение должно обрабатывать временную несогласованность

Пример:
```
```
Strong:
1. Клиент пишет "X=1" на узел A
2. Запись подтверждена → все последующие чтения (с A, B, C) вернут X=1

Eventual:
1. Клиент пишет "X=1" на узел A
2. Запись подтверждена на A
3. Чтение с B может вернуть старое значение, пока репликация не дойдёт
4. Через некоторое время все узлы покажут X=1
```

Когда что использовать:
```
→ Strong: финансовые транзакции, инвентарь, блокировки
→ Eventual: лайки, просмотры, кэш, аналитика, соц-граф
```

**DevOps-контекст**: 
- Выбор модели консистентности влияет на архитектуру приложения
- Eventual consistency требует idempotent операций и версионирования
- Мониторьте replication lag как метрику "степени eventual-ности"
- Документируйте гарантии для разработчиков

### 25. Как обеспечить безопасность БД: аутентификация, авторизация, шифрование?

**Ответ**:
```
Аутентификация (кто вы):
• Password: scram-sha-256 (PG), caching_sha2_password (MySQL)
• SSL-сертификаты: клиент аутентифицируется по сертификату
• IAM-аутентификация (cloud): временные токены вместо паролей
• LDAP/AD: централизованное управление пользователями

Авторизация (что можете делать):
• Роли и привилегии: GRANT/REVOKE на уровне БД, схемы, таблицы, колонки
• Принцип минимальных привилегий: приложение получает только необходимые права
• Row-level security (PG): фильтрация строк на уровне БД (multi-tenant)
• Views с ограничениями: виртуальные таблицы с предопределёнными фильтрами

Шифрование:
1. In-transit:
   • SSL/TLS между приложением и БД
   • Настройка: sslmode=require/verify-full, проверка сертификатов
   • Обязательно в production

2. At-rest:
   • TDE (Transparent Data Encryption): шифрование файлов БД
   • Filesystem-level: LUKS (Linux), BitLocker (Windows)
   • Cloud: шифрование EBS/RDS через KMS
   • Ключи шифрования должны храниться отдельно от данных (KMS, Vault)

Аудит:
• Логирование подключений, запросов, изменений прав
• ПГ: log_connections, log_statement, pg_audit расширение
• Алерты на подозрительную активность

**DevOps-контекст**: 
- Шифрование in-transit — обязательно, настройте и протестируйте
- At-rest шифрование требуется регуляторами (GDPR, PCI-DSS)
- Интегрируйте управление секретами: HashiCorp Vault, AWS Secrets Manager
- Регулярно ротируйте пароли и ключи

### 26. Что такое RPO и RTO? Как они влияют на выбор стратегии бэкапа?

**Ответ**:
```
RPO (Recovery Point Objective):
• Максимальная допустимая потеря данных (время между последним бэкапом и сбоем)
• Пример: RPO=15 мин → бэкапы/репликация должны быть не реже чем каждые 15 минут

RTO (Recovery Time Objective):
• Максимальное допустимое время простоя
• Пример: RTO=1 час → процедуры восстановления должны уложиться в 60 минут

Стратегии в зависимости от RPO/RTO:
```
```
| RPO/RTO      | Стратегия              | Пример реализации                    |
|--------------|------------------------|--------------------------------------|
| Часы/дни     | Backup & Restore       | pg_dump раз в сутки в S3             |
| Минуты/часы  | Pilot Light            | Реплика в standby в другом регионе   |
| Секунды/минуты| Warm Standby          | Полноценная уменьшенная копия        |
| ~0/секунды   | Active-Active          | Multi-region с синхронной репликацией|
```

Влияние на выбор:
```
→ Низкий RPO:
   - Частые бэкапы + WAL archiving
   - Синхронная репликация
   - Более дорогое хранилище

→ Низкий RTO:
   - Автоматический failover (Patroni)
   - Предварительно развёрнутая инфраструктура
   - Скрипты восстановления как код
```

**DevOps-контекст**: 
- Определите RPO/RTO с бизнесом: от этого зависит архитектура и бюджет
- Тестируйте восстановление регулярно — без проверки метрики бессмысленны
- Автоматизируйте: чем меньше ручных шагов, тем меньше человеческих ошибок при инциденте

### 27. Как работает DynamoDB? Когда выбирать его вместо PostgreSQL?

**Ответ**:
```
DynamoDB — полностью управляемая NoSQL БД от AWS.

Архитектура:
• Serverless: нет управления серверами, автомасштабирование
• Распределённая: данные автоматически шардируются по партициям
• Репликация: синхронная внутри региона, опционально глобальные таблицы

Модель данных:
• Таблицы → элементы (items) → атрибуты (ключ-значение)
• Ключи: partition key (обязательно) + sort key (опционально)
• Запросы: Query (по partition key) или Scan (дорогой полный проход)

Capacity modes:
1. Provisioned: вы задаёте WCU/RCU, предсказуемая цена
2. On-demand: автомасштабирование, плата за использование

Когда выбирать DynamoDB вместо PostgreSQL:
```
```
✅ DynamoDB:
   - Непредсказуемая или пиковая нагрузка
   - Простая модель данных (ключ-значение, документы)
   - Не нужны сложные запросы, JOIN, транзакции между таблицами
   - Serverless-архитектура, интеграция с Lambda

✅ PostgreSQL:
   - Сложные запросы, аналитика, отчёты
   - Строгая схема, внешние ключи, транзакции
   - Нужны расширения (PostGIS, pgvector, Citus)
   - On-premises или multi-cloud deployment
```

**DevOps-контекст**: 
- Проектируйте схему под запросы: выбор partition key критичен для равномерного распределения
- Мониторьте ConsumedCapacity и ThrottlingEvents — признаки неправильного подбора capacity
- Для аналитики: экспортируйте данные в S3 + Athena, не делайте сложные запросы в DynamoDB напрямую

### 28. Как мониторить БД в production? Какие метрики критичны?

**Ответ**:
```
Ключевые метрики:

Производительность:
• Query latency: p50, p95, p99 время выполнения
• Throughput: QPS, TPS
• Cache hit ratio: цель >99% для PG
• Replication lag: отставание реплик в секундах/байтах

Ресурсы:
• CPU, memory, disk I/O, network I/O
• Connections: active, idle, waiting, max
• Disk usage: данные, WAL, логи, бэкапы

Здоровье:
• Uptime, restarts, crash count
• Locks: количество и длительность блокировок
• Long-running queries: запросы > порога
• Autovacuum stats: задержки, эффективность

Сбор метрик:
• Встроенные: pg_stat_*, information_schema
• Экспортёры: postgres_exporter, mysqld_exporter (для Prometheus)
• APM: Datadog, New Relic (агенты собирают и анализируют)

Алертинг:
• Пороговые: "replication lag > 60s", "disk usage > 80%"
• Аномалии: "резкий рост ошибок подключения"
• Эскалация: warning → critical → страница онколлу

Пример Prometheus alert:
```
```yaml
- alert: PostgreSQLReplicationLag
  expr: pg_replication_lag_seconds > 60
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Replication lag на {{ $labels.instance }}"
```

**DevOps-контекст**: 
- Начинайте с базовых метрик: connections, disk, replication lag
- Настройте алерты на действия: не просто "проблема", а "что делать"
- Храните историю метрик: для постмортема и планирования масштабирования
- Интегрируйте с incident management: PagerDuty, Opsgenie, Slack

### 29. Что делать, если реплика отстала от мастера на несколько часов?

**Ответ**:
```
Диагностика:
1. Проверить причину:
```
```sql
-- PostgreSQL
SELECT client_addr, state, 
       pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes,
       write_lag, flush_lag, replay_lag
FROM pg_stat_replication;
```

2. Возможные причины:
   - Сетевая задержка или разрыв
   - Медленные запросы на реплике (блокируют применение WAL)
   - Нехватка ресурсов на реплике (CPU, disk I/O)
   - Удаление WAL на master до того, как реплика их получила (без replication slot)

Решение:
```
✅ Если реплика ещё синхронизирована (есть нужные WAL):
   - Устранить причину (сеть, нагрузка, ресурсы)
   - Дождаться догонки, мониторить прогресс
   - Временно отключить read-трафик с реплики, если мешает

❌ Если WAL удалены на master (реплика не может догнать):
   - Остановить реплику
   - Сделать новый base backup с master
   - Развернуть на реплике, настроить заново
   - Запустить репликацию с нуля

Профилактика:
✅ Использовать replication slots (предотвращают удаление нужных WAL)
✅ Мониторить lag и настраивать алерты на порог
✅ Выделять реплике достаточно ресурсов (не меньше, чем master)
✅ Избегать долгих запросов на реплике (они блокируют применение WAL)
```

**DevOps-контекст**: 
- Настройте алерты на replication lag до того, как проблема станет критичной
- Документируйте процедуру восстановления — при инциденте нет времени искать информацию
- Тестируйте сценарий "потеря реплики" в staging

### 30. ⭐ Как спроектировать систему, которая выдержит потерю целого региона?

**Ответ**:
```
Архитектура disaster recovery:

1. Multi-region deployment:
   • Активная БД в регионе A, standby в регионе B (или active-active)
   • Репликация: синхронная внутри региона, асинхронная между регионами
   • Глобальный load balancer с health checks и failover

2. Данные:
   • Бэкапы в третьем регионе/провайдере (правило 3-2-1)
   • WAL/binlog архивация в объектное хранилище с versioning
   • Шифрование бэкапов и ключи в отдельном KMS

3. Инфраструктура как код:
   • Terraform/CloudFormation для воспроизводимого развёртывания
   • Конфигурация БД в репозитории, не в ручных настройках

4. Автоматизация восстановления:
   • Runbooks как код: скрипты promote standby, переключения трафика
   • Проверки перед failover: консистентность данных, лаг репликации
   - Тестирование: регулярные учения (chaos engineering)

5. Мониторинг и алертинг:
   • Межрегиональная репликация: lag, error rate
   • Health checks на уровне приложения, не только БД
   • Алерты на деградацию, а не только на полный отказ

Чеклист:
```
```
[ ] Репликация между регионами настроена и мониторится
[ ] Бэкапы хранятся в другом регионе/провайдере
[ ] Инфраструктура описана как код и может быть развёрнута с нуля
[ ] Скрипты восстановления протестированы и задокументированы
[ ] Проведены учения по восстановлению (не реже раза в год)
[ ] Команда знает процедуру и роли при инциденте
```

**DevOps-контекст**: 
- Потеря региона — редкое, но критичное событие: готовьтесь заранее
- Балансируйте стоимость и надёжность: не всем системам нужен active-active
- Автоматизация снижает человеческие ошибки при стрессе инцидента
- Документация и тренировки важнее "идеальной" архитектуры
