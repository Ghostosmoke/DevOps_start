### 1. В чём разница между реляционными и нереляционными базами данных?
**Суть**: Реляционные (SQL) БД используют табличную структуру со строгой схемой и связями; нереляционные (NoSQL) — гибкие модели данных для специфичных сценариев.

**Подробно**:
```
Реляционные БД (SQL):
• Данные в таблицах со строками и колонками
• Строгая схема: типы данных, ограничения, внешние ключи
• Язык запросов: SQL (стандартизированный)
• Гарантии ACID: атомарность, консистентность, изоляция, долговечность
• Примеры: PostgreSQL, MySQL, Oracle, SQL Server

Нереляционные БД (NoSQL):
• Документные: данные в JSON-подобных документах (MongoDB, CouchDB)
• Key-Value: простые пары ключ-значение (Redis, DynamoDB)
• Column-family: данные по колонкам, а не строкам (Cassandra, HBase)
• Graph: узлы и связи для графовых данных (Neo4j, Amazon Neptune)
• Гибкая схема, горизонтальное масштабирование, гарантии BASE
```

| Критерий | SQL | NoSQL |
|----------|-----|-------|
| Схема | Строгая, фиксированная | Гибкая, динамическая |
| Масштабирование | Вертикальное (чаще) | Горизонтальное (легче) |
| Транзакции | Полные ACID | Ограниченные, eventual consistency |
| Запросы | Мощный SQL | Специфичные API/языки |
| Связи | JOIN между таблицами | Денормализация, вложенные документы |

**DevOps-контекст**: 
- PostgreSQL/MySQL — для финансовых данных, заказов, пользователей (где важна консистентность)
- Redis — для кэша, сессий, очередей (высокая скорость, временные данные)
- MongoDB — для логов, аналитики, контента с гибкой структурой
- DynamoDB — для serverless-архитектур, автоматического масштабирования

**Связанные инструменты**: `psql`, `mysql`, `mongosh`, `redis-cli`, `aws dynamodb`.

### 2. Как устроена архитектура PostgreSQL: процессы, память, WAL?
**Суть**: PostgreSQL — процесс-ориентированная СУБД с раздельной памятью, журналированием изменений (WAL) и многоверсионностью (MVCC).

**Подробно**:
```
Процессы PostgreSQL:
• Postmaster — главный процесс, управляет запуском/остановкой
• Backend processes — один на каждое клиентское соединение
• Background workers:
  - autovacuum launcher/worker — очистка старых версий строк
  - wal writer — запись WAL в диск
  - checkpointer — периодическая запись "чекпоинтов"
  - stats collector — сбор статистики

Память (shared_buffers + work_mem):
• shared_buffers — общий кэш данных (обычно 25% RAM)
• work_mem — память на операцию сортировки/хэша (на операцию!)
• maintenance_work_mem — для VACUUM, CREATE INDEX
• wal_buffers — буфер для записи журнала

WAL (Write-Ahead Logging):
• Все изменения сначала пишутся в журнал, потом в данные
• Гарантирует долговечность: после COMMIT данные не потеряются
• Позволяет репликацию: slave читает WAL и применяет изменения
• Файлы в pg_wal/, размер задаётся wal_segment_size

MVCC (Multi-Version Concurrency Control):
• Читающие не блокируют пишущих и наоборот
• Каждая транзакция видит "снимок" данных на момент начала
• Старые версии строк удаляются через VACUUM
```

**DevOps-контекст**: 
- Настройка `shared_buffers` и `work_mem` критична для производительности
- Следите за ростом `pg_wal/` — переполнение остановит БД
- Autovacuum должен работать, иначе таблица "раздуется" от старых версий
- WAL — основа репликации и PITR, не отключайте

**Связанные инструменты**: `pg_stat_activity`, `pg_stat_database`, `pg_wal`, `vacuumlo`.

### 3. Как работает MySQL: storage engines, InnoDB vs MyISAM?
**Суть**: MySQL поддерживает плагины storage engines — разные механизмы хранения с разными компромиссами.

**Подробно**:
```
Storage engines в MySQL:
• InnoDB (default с MySQL 5.5):
  - Поддержка транзакций (ACID)
  - Row-level locking (параллельные записи)
  - Foreign keys, crash recovery
  - Clustered index (данные хранятся в первичном ключе)
  - Использует buffer pool для кэширования

• MyISAM (устаревший):
  - Нет транзакций, нет foreign keys
  - Table-level locking (блокирует всю таблицу при записи)
  - Быстрее для read-heavy workloads без записей
  - Поддерживает FULLTEXT индекс (раньше)

• Memory:
  - Данные только в памяти, теряются при перезагрузке
  - Очень быстро, для временных таблиц

• Archive:
  - Оптимизирован для INSERT-only, сжатие
  - Для логирования, архивов

Архитектура MySQL:
• Server layer: парсинг, оптимизация, кэш запросов
• Storage engine layer: хранение, индексы, транзакции
• Buffer pool: кэш данных и индексов InnoDB
• Redo log: журнал изменений для crash recovery (аналог WAL)
• Undo log: для отката транзакций и MVCC
```

**DevOps-контекст**: 
- Всегда используйте InnoDB для продакшена
- `innodb_buffer_pool_size` — самый важный параметр (70-80% RAM)
- MyISAM может повредиться при краше — избегайте
- MySQL репликация основана на бинарных логах (binlog), не на InnoDB redo log

**Связанные инструменты**: `SHOW ENGINES`, `information_schema`, `mysqltuner`, `pt-query-digest`.

### 4. Что такое ACID и почему это важно для надёжности?
**Суть**: ACID — набор свойств транзакций, гарантирующих надёжность операций с данными даже при сбоях.

**Подробно**:
```
ACID расшифровывается:

A — Atomicity (Атомарность):
• Транзакция выполняется целиком или не выполняется вообще
• Пример: перевод денег — списание и зачисление должны произойти оба или ни одного
• Реализация: через undo log (откат при ошибке)

C — Consistency (Консистентность):
• Транзакция переводит БД из одного валидного состояния в другое
• Все ограничения (constraints, FK, unique) соблюдаются
• Это ответственность приложения + БД

I — Isolation (Изоляция):
• Параллельные транзакции не влияют друг на друга
• Уровни изоляции: Read Uncommitted → Read Committed → Repeatable Read → Serializable
• Реализация: через блокировки или MVCC

D — Durability (Долговечность):
• После COMMIT изменения не теряются даже при сбое питания
• Реализация: WAL/redo log + fsync на диск

Пример нарушения:
```
```
Без атомарности: списали деньги, но не зачислили → потеря средств
Без консистентности: отрицательный баланс при ограничении >= 0
Без изоляции: "грязное чтение" — увидели незакоммиченные данные
Без долговечности: после перезагрузки пропали последние заказы
```

**DevOps-контекст**: 
- Для финансовых операций, заказов, инвентаря — только ACID-совместимые БД
- Некоторые NoSQL жертвуют изоляцией/консистентностью ради скорости (BASE)
- Настройка `synchronous_commit=on` в PostgreSQL гарантирует durability
- Тестируйте recovery после краша — не все конфигурации одинаково надёжны

### 5. Как работают индексы: B-tree, hash, composite?
**Суть**: Индексы — структуры данных для ускорения поиска, ценой замедления записи и потребления памяти/диска.

**Подробно**:
```
B-tree (балансированное дерево) — дефолтный индекс:
• Поддерживает: =, <, >, <=, >=, BETWEEN, LIKE 'prefix%'
• Данные отсортированы, поиск за O(log n)
• Эффективен для диапазонных запросов и сортировки
• Подходит для большинства сценариев

Hash индекс:
• Поддерживает только точное равенство (=)
• Поиск за O(1), но нет сортировки и диапазонов
• В PostgreSQL: только для memory-таблиц или через расширение
• В MySQL: внутри InnoDB используется для adaptive hash index

Composite (составной) индекс:
• Индекс по нескольким колонкам: (last_name, first_name)
• Работает по правилу левой границы: можно искать по (last_name) или (last_name, first_name), но не только по first_name
• Порядок колонок важен: сначала колонки с =, потом с диапазонами

Другие типы:
• GIN/GiST (PostgreSQL): для полнотекстового поиска, JSON, массивов
• BRIN: для больших отсортированных таблиц (временные ряды)
• Partial index: индекс только для подмножества строк (WHERE status='active')
• Covering index: включает все нужные колонки → Index Only Scan

Пример:
```
```sql
-- Таблица
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(100),
    status VARCHAR(20),
    created_at TIMESTAMP
);

-- Индексы
CREATE INDEX idx_users_email ON users(email);                    -- B-tree
CREATE INDEX idx_users_status_created ON users(status, created_at); -- Composite
CREATE INDEX idx_users_active ON users(created_at) WHERE status='active'; -- Partial

-- Запросы, которые используют индексы:
SELECT * FROM users WHERE email = 'a@b.com';                    -- idx_users_email
SELECT * FROM users WHERE status = 'active' ORDER BY created_at; -- idx_users_status_created
SELECT * FROM users WHERE status = 'active' AND created_at > '2025-01-01'; -- тот же индекс
```

**DevOps-контекст**: 
- Индексы ускоряют SELECT, но замедляют INSERT/UPDATE/DELETE
- Не создавайте индексы "на всякий случай" — мониторьте использование через pg_stat_user_indexes
- Composite индексы эффективнее нескольких отдельных
- Частичные индексы экономят место и ускоряют запросы к "горячим" данным

### 6. Что такое транзакции и уровни изоляции?
**Суть**: Транзакция — последовательность операций, выполняемых как единое целое; уровни изоляции определяют, насколько транзакции "видят" изменения друг друга.

**Подробно**:
```
Уровни изоляции (по возрастанию строгости):

1. Read Uncommitted:
   • Можно читать незакоммиченные данные других транзакций
   • Проблема: "dirty read" — прочитали данные, которые потом откатили
   • Практически не используется

2. Read Committed (default в PostgreSQL, Oracle):
   • Видны только закоммиченные данные
   • Проблема: "non-repeatable read" — в одной транзакции два SELECT могут вернуть разные результаты, если между ними другая транзакция закоммитила изменения
   • Подходит для большинства приложений

3. Repeatable Read (default в MySQL/InnoDB):
   • В рамках транзакции повторные SELECT возвращают те же данные
   • Проблема: "phantom read" — новые строки, добавленные другой транзакцией, могут появиться при повторном запросе с условием
   • В PostgreSQL реализуется через MVCC-снапшот

4. Serializable:
   • Полная изоляция: транзакции выполняются как последовательно
   • Проблема: снижение параллелизма, возможны ошибки сериализации (нужен retry)
   • Для критичных финансовых операций

Аномалии параллелизма:
• Dirty read: чтение незакоммиченных данных
• Non-repeatable read: изменение прочитанных данных другой транзакцией
• Phantom read: появление новых строк, удовлетворяющих условию
• Write skew: две транзакции читают одни данные, пишут разные, нарушая бизнес-правило

Пример:
```
```sql
-- Read Committed (PostgreSQL default)
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- 100
-- Другая сессия: UPDATE accounts SET balance = 200 WHERE id = 1; COMMIT;
SELECT balance FROM accounts WHERE id = 1;  -- 200 (видим изменение!)
COMMIT;

-- Repeatable Read
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;  -- 100
-- Другая сессия: UPDATE ... COMMIT;
SELECT balance FROM accounts WHERE id = 1;  -- всё ещё 100 (снапшот!)
COMMIT;
```

**DevOps-контекст**: 
- Read Committed — безопасный дефолт для большинства случаев
- Repeatable Read — если нужна консистентность в рамках транзакции (отчёты, экспорт)
- Serializable — только когда другие уровни недостаточны (и готовьте retry-логику)
- Долгие транзакции блокируют ресурсы — минимизируйте их длительность

### 7. Как работает Redis: in-memory, data structures, persistence?
**Суть**: Redis — in-memory key-value хранилище с богатыми структурами данных и опциональной персистентностью.

**Подробно**:
```
Архитектура Redis:
• Single-threaded event loop: один поток обрабатывает все команды (нет блокировок)
• In-memory: все данные в оперативной памяти → очень быстро (микросекунды)
• Data structures:
  - Strings: простые значения, счётчики, битовые операции
  - Lists: двусвязные списки, очереди (LPUSH/RPOP)
  - Sets: уникальные неупорядоченные значения, операции множеств
  - Sorted Sets: элементы с оценкой, упорядочены по score
  - Hashes: поля-значения (как объекты)
  - Streams: лог событий, потребительские группы (как Kafka lite)
  - Bitmaps, HyperLogLog, Geospatial — специализированные структуры

Persistence (сохранение на диск):
• RDB (snapshot):
  - Периодическое создание снимка данных (fork + copy-on-write)
  - Файл .rdb, компактный, для бэкапов
  - Потеря данных с момента последнего снапшота при краше

• AOF (Append-Only File):
  - Лог всех записывающих команд в формате Redis-протокола
  - Можно настроить fsync: always (медленно, безопасно), everysec (баланс), no (быстро, рискованно)
  - При старте: воспроизведение лога для восстановления
  - Файл больше, но меньше потеря данных

• Комбинированный режим: RDB + AOF (AOF приоритетнее при восстановлении)

Replication:
• Master-slave: асинхронная репликация всех команд
• Slave может принимать read-запросы (масштабирование чтения)
• При разрыве: частичная или полная ресинхронизация

**DevOps-контекст**: 
- Redis не заменяет основную БД — это кэш/очередь/временное хранилище
- Настройте maxmemory + eviction policy (allkeys-lru, volatile-ttl)
- AOF с fsync=everysec — хороший баланс для продакшена
- Используйте Redis Sentinel или Cluster для HA
- Мониторьте память: утечки или рост данных могут "уронить" продакшен

**Связанные инструменты**: `redis-cli`, `INFO memory`, `SLOWLOG`, `redis-benchmark`.

### 8. MongoDB: документная модель, BSON, схема?
**Суть**: MongoDB — документоориентированная БД, хранящая данные в BSON (бинарный JSON) с гибкой схемой.

**Подробно**:
```
Документная модель:
• Данные хранятся в документах (BSON) внутри коллекций
• Документ ≈ объект в коде: вложенные структуры, массивы
• Гибкая схема: разные документы в коллекции могут иметь разные поля
• Но можно задать валидацию схемы через JSON Schema

BSON (Binary JSON):
• Расширение JSON с дополнительными типами: Date, ObjectId, Binary, Decimal128
• Позволяет эффективное парсинг и навигацию по полям
• Поддерживает вложенные документы и массивы

Индексы:
• B-tree по любому полю, включая вложенные (field.subfield)
• Compound, multikey (по массивам), text, geospatial, hashed (для шардирования)
• Partial, sparse, TTL (автоудаление по времени)

Запросы:
• Query language на основе JSON: {age: {$gt: 18}, status: "active"}
• Aggregation pipeline: $match → $group → $sort → $project
• Поддерживает JOIN-подобные операции через $lookup

Пример документа:
```
```json
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "username": "alice",
  "profile": {
    "email": "alice@example.com",

      "notifications": true
    }

  "tags": ["python", "devops"],
  "created_at": ISODate("2024-05-08T10:00:00Z")
}
```

**DevOps-контекст**: 
- Гибкая схема удобна для прототипирования, но в продакшене добавляйте валидацию
- Индексы критичны: без них запросы делают collection scan (O(n))
- Вложенные документы удобны, но не переусердствуйте — сложно индексировать и обновлять
- MongoDB по умолчанию не гарантирует сильную консистентность в распределённых конфигурациях

### 9. Что такое connection pooling и зачем он нужен?
**Суть**: Connection pool — кэш готовых соединений с БД, позволяющий переиспользовать их вместо создания новых на каждый запрос.

**Подробно**:
```
Проблема без pooling:
• Установка соединения с БД — дорогая операция (TCP handshake, аутентификация, выделение ресурсов)
• При высокой нагрузке: тысячи соединений → исчерпание лимитов БД, высокая задержка
• Каждое соединение потребляет память на сервере БД

Как работает pool:
• При старте приложения создаётся пул из N "готовых" соединений
• При запросе: берётся свободное соединение из пула (не создаётся новое)
• После выполнения: соединение возвращается в пул (не закрывается)
• Пул управляет: мин/макс размером, таймаутами, health checks

Параметры пула:
• pool_size: базовое количество соединений
• max_overflow: сколько можно временно "перелить" сверх pool_size
• pool_timeout: сколько ждать свободное соединение при нехватке
• pool_recycle: через сколько времени пересоздавать соединение (предотвращает stale)

Реализации:
• В приложении: SQLAlchemy, psycopg2.pool, HikariCP (Java)
• Внешний прокси: PgBouncer (PostgreSQL), ProxySQL (MySQL)

Пример (SQLAlchemy):
```
```python
engine = create_engine(
    "postgresql://user:pass@localhost/db",
    pool_size=10,
    max_overflow=20,
    pool_timeout=30,
    pool_recycle=1800,
    pool_pre_ping=True  # проверять соединение перед использованием
)
```

**DevOps-контекст**: 
- Без pooling приложение не масштабируется: каждое новое соединение — нагрузка на БД
- PgBouncer в режиме transaction pooling экономит ресурсы (одно соединение БД на много клиентских транзакций)
- Настройте `max_connections` в БД с учётом пулов всех приложений
- Мониторьте метрики пула: wait time, overflow, idle connections

**Связанные инструменты**: PgBouncer, ProxySQL, SQLAlchemy, psycopg2.pool.

### 10. Как работают резервные копии: logical vs physical backup?
**Суть**: Логический бэкап — экспорт данных в читаемом формате; физический — копирование файлов БД на уровне блоков.

**Подробно**:
```
Logical backup (pg_dump, mysqldump):
• Экспорт схемы и/или данных в SQL-скрипт или текст
• Форматы: plain SQL, custom (-Fc), directory, tar
• Преимущества:
  - Читаем человеком, можно редактировать
  - Переносимость: между версиями, ОС, архитектурами
  - Выборочное восстановление: таблицы, схемы
• Недостатки:
  - Медленнее на больших объёмах
  - Занимает больше места (текст)
  - Восстановление дольше (нужно выполнять SQL)

Physical backup (pg_basebackup, файловая копия):
• Копирование файлов данных БД (data directory, WAL)
• Преимущества:
  - Быстрее создание и восстановление (просто копирование)
  - Точная копия, включая индексы, статистику
  - Основа для streaming replication и PITR
• Недостатки:
  - Зависит от версии/ОС/архитектуры
  - Восстанавливает всю БД/кластер, нельзя выборочно
  - Требует остановки или специальной синхронизации

Snapshot-based (cloud):
• Снимки диска на уровне облачного провайдера (EBS, GCP Persistent Disk)
• Мгновенное создание, консистентность зависит от БД
• Часто комбинируют с logical backup для гибкости

Best practices:
• 3-2-1 правило: 3 копии, 2 разных носителя, 1 вне площадки
• Регулярно тестируйте восстановление (бэкап без проверки — не бэкап)
• Шифруйте бэкапы при хранении вне доверенной среды
• Логируйте и мониторьте успешность бэкапов
```

**DevOps-контекст**: 
- Logical backup — для миграций, аудита, частичного восстановления
- Physical backup — для DR, быстрого восстановления всего кластера
- В cloud: используйте managed snapshots + logical backup в S3
- Автоматизируйте и добавляйте алерты при неудаче бэкапа

**Связанные инструменты**: `pg_dump`, `pg_basebackup`, `mysqldump`, `xtrabackup`, `aws backup`.

### 11. ⭐ Как работает репликация: синхронная, асинхронная, полусинхронная?
**Суть**: Репликация — копирование данных с первичного узла на реплики для отказоустойчивости, масштабирования чтения и гео-распределения.

**Подробно**:
```
Модели репликации:

1. Асинхронная (default в PostgreSQL, MySQL):
   • Master записывает данные → отправляет реплике → подтверждает клиенту ДО получения ответа от реплики
   • Преимущества: низкая задержка записи, реплика может отставать
   • Недостатки: возможна потеря данных при падении master (данные не успели уйти на реплику)
   • Use case: масштабирование чтения, когда допустима небольшая потеря (RPO > 0)

2. Синхронная:
   • Master ждёт подтверждения от реплики(ок) перед ответом клиенту
   • Преимущества: нулевая потеря данных (RPO=0), реплики всегда актуальны
   • Недостатки: задержка записи растёт с количеством синхронных реплик и сетевой задержкой
   • Если реплика недоступна — запись блокируется (или переключается на асинхронную)
   • Use case: финансовые данные, где потеря недопустима

3. Полусинхронная (MySQL):
   • Гибрид: master ждёт подтверждения хотя бы от одной реплики
   • Баланс между надёжностью и производительностью
   • Если все реплики отвалились — переключается на асинхронную (настраивается)

Потенциальные проблемы:
• Replication lag: реплика отстаёт от master (нагрузка, сеть, медленные запросы)
• Конфликты записи: если писать на реплики (multi-master) — нужны разрешения конфликтов
• Нагрузка на master: каждая реплика читает WAL/binlog — сеть и диски master нагружаются

Мониторинг:
```
```sql
-- PostgreSQL
SELECT client_addr, state, sync_state, 
       pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes
FROM pg_stat_replication;

-- MySQL
SHOW SLAVE STATUS\G
-- Смотреть: Seconds_Behind_Master, Slave_IO_Running, Slave_SQL_Running
```

**DevOps-контекст**: 
- Для HA: минимум 1 синхронная реплика + мониторинг лага
- Для read scaling: асинхронные реплики + балансировщик запросов
- Настройте алерты на replication lag > порога
- Тестируйте failover: автоматический или ручной, с проверкой консистентности

**Связанные инструменты**: Patroni, pg_auto_failover, MHA (MySQL), Orchestrator.

### 12. Streaming replication в PostgreSQL: как устроен WAL?
**Суть**: Streaming replication — механизм репликации PostgreSQL, при котором slave непрерывно получает и применяет WAL-записи от master.

**Подробно**:
```
WAL (Write-Ahead Log) — основа репликации:
• Все изменения сначала пишутся в WAL, потом в данные
• WAL разбит на сегменты (по 16 МБ по умолчанию) в pg_wal/
• Каждый рекорд в WAL имеет уникальный номер — LSN (Log Sequence Number)

Процесс streaming replication:
1. Master:
   - WAL writer периодически сбрасывает буферы WAL на диск
   - WAL sender process (специальный бэкенд) отправляет новые WAL-записи подключенным репликам

2. Slave (standby):
   - WAL receiver подключается к master и запрашивает WAL с определённого LSN
   - Полученные записи пишутся в локальный pg_wal/
   - Startup process применяет WAL к данным (redo)
   - Если hot_standby=on: можно выполнять read-only запросы во время применения

3. Синхронизация:
   - Реплика отправляет подтверждения (flush_lsn, apply_lsn)
   - Master видит прогресс в pg_stat_replication
   - При разрыве: реплика запрашивает с последнего подтверждённого LSN

Replication slots:
• Механизм, предотвращающий удаление WAL, нужных репликам
• Без слота: если реплика отстала, а master удалил старые WAL — реплика не сможет догнать (требуется новый base backup)
• Слот "привязывает" WAL к реплике: master не удалит, пока слот не продвинется
• Опасность: если реплика упала надолго, WAL могут заполнить диск master → мониторинг!

Настройка синхронной репликации:
```
```conf
# postgresql.conf на master
synchronous_commit = on
synchronous_standby_names = 'standby1'  # или 'FIRST 2 (standby1,standby2)'

# postgresql.conf на standby
primary_conninfo = 'host=master port=5432 user=replicator password=...'
primary_slot_name = 'my_slot'
hot_standby = on
```

**DevOps-контекст**: 
- Streaming replication — основа HA и read scaling в PostgreSQL
- Всегда используйте replication slots в production
- Мониторьте `pg_stat_replication` и размер pg_wal/
- Для автоматического failover нужен внешний инструмент (Patroni), т.к. PostgreSQL не переключает master сам

**Связанные инструменты**: `pg_basebackup`, `pg_receivewal`, `pg_stat_replication`, Patroni.

### 13. Как работает репликация в MongoDB: oplog, replica set?
**Суть**: Репликация в MongoDB реализована через replica set — группу узлов с автоматическим выбором primary и синхронизацией через oplog.

**Подробно**:
```
Replica set архитектура:
• Один primary: принимает все записи
• Несколько secondaries: реплицируют данные, могут обслуживать чтение
• Опционально: arbiter (только голосует, не хранит данные), hidden, delayed

Oplog (operations log):
• Специальная capped-коллекция local.oplog.rs
• Хранит все операции записи (insert, update, delete) в порядке применения
• Ограниченный размер (по умолчанию 5% диска, настраивается)
• Вторичные узлы "догоняют" primary, читая и применяя oplog

Процесс репликации:
1. Клиент пишет на primary → операция применяется → записывается в oplog
2. Вторичные узлы периодически опрашивают primary на наличие новых записей в oplog
3. При нахождении: скачивают и асинхронно применяют локально
4. Каждый узел отслеживает свой прогресс через оптимистичный "timestamp"

Выбор primary (election):
• Если primary становится недоступен, оставшиеся узлы инициируют выборы
• Голосование: каждый узел имеет приоритет, голосует за наиболее подходящего кандидата
• Кворум: большинство узлов должно проголосовать (избегает split-brain)
• Новый primary начинает принимать записи, остальные синхронизируются с ним

Read preferences:
• primary: чтение только с primary (консистентность)
• secondary: чтение с любой вторичной (масштабирование, возможное отставание)
• nearest: чтение с ближайшего узла по задержке
• primaryPreferred / secondaryPreferred: гибкие стратегии

**DevOps-контекст**: 
- Размер oplog критичен: если вторичный узел отстанет больше, чем вмещает oplog — потребуется полная ресинхронизация
- Мониторьте `rs.printReplicationInfo()` и `rs.printSecondaryReplicationInfo()`
- Для кросс-регион репликации учитывайте задержку: настраивайте write concern и read preference
- Автоматический failover работает "из коробки", но тестируйте в staging

**Связанные инструменты**: `mongosh`, `rs.status()`, `db.getReplicationInfo()`, Ops Manager.

### 14. Redis Sentinel и Cluster: в чём разница и когда что использовать?
**Суть**: Sentinel обеспечивает HA для master-slave Redis; Cluster — горизонтальное масштабирование с шардированием.

**Подробно**:
```
Redis Sentinel (High Availability):
• Набор процессов, мониторит мастер и реплики
• Обнаружение сбоев: если мастер не отвечает — голосование за новую реплику как мастер
• Автоматический failover: переназначение клиентов на новый мастер
• Конфигурация: клиенты подключаются к Sentinel, который возвращает актуальный адрес мастера
• Не решает проблему масштабирования: все записи идут на один мастер

Когда использовать:
• Нужен HA для одного набора данных
• Нагрузка вписывается в один узел (по памяти и CPU)
• Проще в настройке и управлении

Redis Cluster (Sharding + HA):
• Данные автоматически шардируются по 16384 хэш-слотам
• Каждый узел отвечает за подмножество слотов
• Клиент вычисляет хэш ключа → определяет слот → подключается к нужному узлу
• Каждый шард — это master + реплики (как в Sentinel)
• Поддерживает перераспределение слотов при добавлении/удалении узлов

Когда использовать:
• Данные не помещаются в память одного узла
• Нужно масштабировать записи и чтения горизонтально
• Готовы к сложностям: cross-slot операции ограничены, сложнее отладка

Ограничения Cluster:
• Команды, затрагивающие несколько ключей, работают только если ключи в одном слоте (или использовать {tag})
• Транзакции и Lua-скрипты — только в рамках одного слота
• Сложнее бэкапы: нужно координировать снимки всех узлов

Пример ключа с tag (для группировки в один слот):
```
```
user:{123}:profile  → хэшируется только "123"
user:{123}:orders   → тот же слот, можно делать мульти-ключ операции
```

**DevOps-контекст**: 
- Sentinel — проще, начните с него, если не упираетесь в лимиты одного узла
- Cluster — для больших данных, но требует изменения кода (учёт слотов)
- Мониторьте память, лаг репликации, состояние кластера через `CLUSTER INFO`
- Бэкапы: для Cluster используйте `redis-cli --cluster backup` или снимки на уровне каждого узла

**Связанные инструменты**: `redis-sentinel`, `redis-cli --cluster`, `CLUSTER NODES`, `SENTINEL masters`.

### 15. Как работает query planner и оптимизатор запросов?
**Суть**: Оптимизатор выбирает наиболее эффективный план выполнения запроса на основе статистики, индексов и стоимости операций.

**Подробно**:
```
Этапы обработки запроса:
1. Parsing: проверка синтаксиса, построение AST
2. Rewriting: применение правил, представлений (views), функций
3. Planning (оптимизация):
   • Генерация возможных планов: какие индексы использовать, порядок JOIN, метод сканирования
   • Оценка стоимости каждого плана (cost-based optimization)
   • Выбор плана с минимальной оценённой стоимостью
4. Execution: выполнение выбранного плана

Типы сканирования:
• Seq Scan: полный проход по таблице (O(n)), если нет подходящего индекса или таблица маленькая
• Index Scan: поиск по индексу + доступ к строке в таблице (2 перехода)
• Index Only Scan: все нужные данные есть в индексе (не нужно обращаться к таблице) — самый быстрый
• Bitmap Scan: построение битовой карты строк по индексу, потом доступ к таблице (эффективно для нескольких условий)

Методы JOIN:
• Nested Loop: для каждой строки левой таблицы ищем в правой (хорошо для маленьких таблиц или с индексом)
• Hash Join: строим хэш-таблицу из одной, потом проходим по другой (эффективно для больших таблиц без индексов)
• Merge Join: обе таблицы сортируются по ключу, потом сливаются (эффективно, если данные уже отсортированы)

Статистика:
• Оптимизатор полагается на статистику: количество строк, распределение значений, корреляции
• В PostgreSQL: таблица pg_statistic, обновляется через ANALYZE
• Устаревшая статистика → неправильный план → медленные запросы

Пример анализа:
```
```sql
EXPLAIN ANALYZE
SELECT u.username, COUNT(o.id)
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.created_at > '2025-01-01'
GROUP BY u.username;

-- Вывод покажет:
-- • Тип сканирования для users/orders
-- • Выбранные индексы
-- • Метод JOIN и его стоимость
-- • Фактическое время выполнения
```

**DevOps-контекст**: 
- Регулярно выполняйте ANALYZE (autovacuum делает это автоматически в PG)
- Используйте EXPLAIN ANALYZE для отладки медленных запросов
- Индексы не всегда помогают: оптимизатор может выбрать Seq Scan, если таблица маленькая или условие не селективное
- Сложные запросы можно "подсказать" через hints (в MySQL) или настройки (в PG: enable_seqscan=off для тестов)

**Связанные инструменты**: `EXPLAIN`, `EXPLAIN ANALYZE`, `pg_stat_statements`, `slow_query_log`.

### 16. Что такое VACUUM и autovacuum в PostgreSQL?
**Суть**: VACUUM — процесс очистки старых версий строк (MVCC) и обновления статистики; autovacuum — автоматический запуск VACUUM.

**Подробно**:
```
Проблема, которую решает VACUUM:
• Благодаря MVCC, UPDATE/DELETE не удаляют строки физически, а помечают как "устаревшие"
• Старые версии занимают место и замедляют сканирование
• Со временем таблица "раздувается" (bloat)

Что делает VACUUM:
• Помечает пространство от удалённых/устаревших строк как доступное для перезаписи
• Обновляет статистику для оптимизатора (если VACUUM ANALYZE)
• Не возвращает место ОС (для этого нужен VACUUM FULL, который блокирует таблицу)

Autovacuum:
• Фоновый процесс, автоматически запускающий VACUUM и ANALYZE
• Параметры:
  - autovacuum_vacuum_threshold: мин. количество изменений перед запуском
  - autovacuum_vacuum_scale_factor: % от размера таблицы, добавляемый к threshold
  - autovacuum_naptime: пауза между проверками таблиц
  - autovacuum_max_workers: сколько воркеров могут работать параллельно

Когда autovacuum не справляется:
• Большие таблицы с частыми обновлениями: настройте более агрессивные параметры на уровне таблицы
• Пиковые нагрузки: autovacuum может создавать нагрузку, настраивайте cost_delay/cost_limit
• Долгие транзакции: блокируют очистку старых версий → мониторинг long-running transactions

Мониторинг:
```
```sql
-- Таблицы, требующие vacuum
SELECT 
    schemaname,
    relname,
    n_dead_tup,
    n_live_tup,
    last_vacuum,
    last_autovacuum,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC;

-- Долгие транзакции, блокирующие vacuum
SELECT 
    pid,
    now() - xact_start AS duration,
    query
FROM pg_stat_activity
WHERE state != 'idle'
  AND xact_start < now() - interval '1 hour'
ORDER BY xact_start;
```

**DevOps-контекст**: 
- Не отключайте autovacuum! Ручной VACUUM — только для экстренных случаев
- Настройте параметры под нагрузку: большие таблицы → меньший scale_factor
- Мониторьте dead tuples и bloat: рост = проблема с vacuum
- VACUUM FULL возвращает место ОС, но требует эксклюзивной блокировки — используйте только в maintenance-окно

**Связанные инструменты**: `pg_stat_user_tables`, `pgstattuple`, `pg_repack` (альтернатива VACUUM FULL без блокировок).

### 17. Как работает шардирование: горизонтальное разделение данных?
**Суть**: Шардирование — разделение одной логической таблицы на физические части (шарды), распределённые по разным узлам.

**Подробно**:
```
Зачем шардировать:
• Один узел не вмещает данные (память, диск)
• Нагрузка на запись/чтение превышает возможности одного узла
• Гео-распределение: данные ближе к пользователям

Стратегии шардирования:
1. Range-based:
   • По диапазону ключа: user_id 1-1000000 → shard1, 1000001-2000000 → shard2
   • Плюсы: простота, диапазонные запросы в рамках одного шарда
   • Минусы: неравномерное распределение (hot spots), сложно добавлять шарды

2. Hash-based:
   • Хэш от ключа определяет шард: shard = hash(key) % num_shards
   • Плюсы: равномерное распределение
   • Минусы: диапазонные запросы требуют обращения ко всем шардам

3. Directory-based:
   • Отдельный сервис (lookup service) хранит маппинг ключ → шард
   • Плюсы: гибкость, можно менять стратегию
   • Минусы: дополнительная сложность, единая точка отказа

Проблемы шардирования:
• JOIN между шардами: либо денормализация, либо координация на уровне приложения
• Транзакции между шардами: двухфазный коммит (сложно, медленно) или отказ от распределённых транзакций
• Глобальные уникальные ограничения: требуют координации
• Ребалансировка: при добавлении шарда нужно переместить часть данных (downtime или онлайн-миграция)

Реализации:
• Citus (PostgreSQL): прозрачное шардирование, распределённые запросы
• MongoDB: встроенное шардирование с config servers и mongos router
• Приложение-уровень: ручное определение шарда по ключу

**DevOps-контекст**: 
- Шардирование — последняя мера оптимизации, сначала попробуйте вертикальное масштабирование, кэширование, оптимизацию запросов
- Выбор distribution key критичен: по нему чаще всего ищут и джойнят
- Мониторьте распределение данных: перекос (data skew) ведёт к перегрузке отдельных шардов
- Планируйте стратегию ребалансировки заранее

**Связанные инструменты**: Citus, MongoDB sharding, Vitess (MySQL), приложение-логика.

### 18. Point-in-time recovery (PITR): как работает восстановление на момент времени?
**Суть**: PITR — возможность восстановить БД не только на момент последнего бэкапа, а на любой момент в прошлом, используя журнал изменений.

**Подробно**:
```
Как работает в PostgreSQL:
1. Base backup: физическая копия данных (pg_basebackup)
2. WAL archiving: настройка archive_command для копирования заполненных WAL-сегментов в безопасное хранилище
3. Восстановление:
   - Разворачиваем base backup
   - Создаём recovery.signal и postgresql.auto.conf с параметрами:
     • restore_command: как доставать WAL из архива
     • recovery_target_time: до какого момента применять WAL
   - Запускаем БД: она применяет WAL последовательно до указанного момента

Параметры восстановления:
• recovery_target_time: '2026-05-08 14:30:00' — до конкретного времени
• recovery_target_xid: до конкретного transaction ID
• recovery_target_name: до named restore point (через pg_create_restore_point)
• recovery_target_action: promote (стать master) или pause (остановиться для проверки)

Преимущества:
• Защита от человеческих ошибок: "удалил не ту таблицу" → откат на минуту назад
• Точное восстановление после инцидента: до момента перед сбоем
• Гибкость: можно восстановить в отдельный инстанс для анализа

Ограничения:
• Требует места для архива WAL (может быть много)
• Восстановление занимает время (применение всех WAL)
• Нужно тестировать процедуру заранее

Пример архивации:
```
```conf
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'test ! -f /backup/wal/%f && cp %p /backup/wal/%f'
archive_timeout = 300  # форсировать переключение WAL каждые 5 минут
```

**DevOps-контекст**: 
- PITR — must-have для production, где потеря данных недопустима
- Храните архив WAL отдельно от сервера БД (иначе при падении диска потеряете и данные, и архив)
- Автоматизируйте восстановление: скрипты + документация + регулярные учения
- Мониторьте успешность архивации: если WAL не архивируется — PITR не сработает

**Связанные инструменты**: `pg_basebackup`, `pg_waldump`, `pg_create_restore_point`, WAL-G, pgBackRest.

### 19. Что такое connection string, failover, retry logic?
**Суть**: Механизмы устойчивого подключения приложения к БД при сбоях, переключениях и временных ошибках.

**Подробно**:
```
Connection string:
• Строка подключения, содержащая параметры для установления соединения
• Пример (PostgreSQL):
  postgresql://user:pass@host1:5432,host2:5432/db?target_session_attrs=read-write&connect_timeout=5

• Параметры:
  - host: один или несколько хостов (для failover)
  - target_session_attrs: read-write (только master), read-any (любой узел)
  - connect_timeout: сколько ждать установления соединения
  - options: дополнительные параметры (sslmode, application_name)

Failover в строке подключения:
• Указание нескольких хостов: приложение пробует подключиться по очереди
• Поддержка зависит от драйвера: libpq (PostgreSQL), mysql-connector (MySQL)
• Не заменяет полноценный HA-инструмент (Patroni, ProxySQL), но даёт базовую отказоустойчивость

Retry logic:
• Автоматическое повторение операции при временных ошибках
• Типы ошибок для retry:
  - Connection refused / timeout (сеть, перезагрузка БД)
  - Serialization failure / deadlock (конфликт транзакций)
  - Too many connections (временная нехватка ресурсов)
• Стратегии:
  - Fixed delay: ждать фиксированное время между попытками
  - Exponential backoff: задержка растёт (2^attempt), предотвращает лавину
  - Jitter: случайная добавка к задержке, предотвращает синхронизацию клиентов

Реализация (Python, tenacity):
```
```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, max=10))
def query_db(sql):
    with engine.connect() as conn:
        return conn.execute(text(sql))
```

**DevOps-контекст**: 
- Всегда настраивайте connect_timeout и retry в production-приложениях
- Различайте временные и постоянные ошибки: не ретрайте валидационные ошибки
- Логгируйте ретраи: частые ретраи — сигнал о проблемах в БД или сети
- Используйте circuit breaker: после нескольких неудач — быстро возвращать ошибку, не нагружая БД

**Связанные инструменты**: libpq, tenacity, resilience4j, Istio retry policies.

### 20. Как работают миграции схемы БД: forward-only, reversible?
**Суть**: Миграции — версионируемые скрипты для изменения схемы БД, позволяющие синхронизировать структуру между окружениями.

**Подробно**:
```
Типы миграций:
1. Forward-only (необратимые):
   • Только применение вперёд (up), откат (down) не предусмотрен или невозможен
   • Пример: удаление колонки, потеря данных
   • Проще в реализации, но рискованнее

2. Reversible (обратимые):
   • Каждая миграция имеет up и down скрипты
   • Позволяет откатиться при проблемах
   • Требует аккуратности: down должен корректно отменять up

3. Transactional:
   • Миграция выполняется внутри транзакции
   • При ошибке — полный откат (атомарность)
   • Поддерживается не всеми БД/операциями (в MySQL DDL не всегда транзакционно)

Процесс применения:
```
```
1. Приложение/инструмент проверяет таблицу миграций (например, schema_migrations)
2. Находит неприменённые миграции, сортирует по версии/времени
3. Применяет up-скрипты по очереди, записывает версию в таблицу
4. При ошибке: останавливается, оставляет БД в консистентном состоянии (если транзакционно)

Инструменты:
• Flyway: простые SQL-скрипты, версия в имени файла (V1__create_users.sql)
• Liquibase: XML/YAML/JSON-декларации, поддержка rollback, сложная логика
• Alembic (Python): код на SQLAlchemy, генерация миграций из моделей
• Django migrations: встроенные, привязаны к ORM

Best practices:
• Никогда не меняйте применённые миграции — только добавляйте новые
• Тестируйте миграции на staging с продакшен-подобными данными
• Разделяйте миграции схемы и миграции данных (data migrations)
• Для больших таблиц: используйте online DDL (PostgreSQL: ADD COLUMN с DEFAULT, MySQL: pt-online-schema-change)

Пример (Flyway):
```
```
db/migration/
├── V1__create_users_table.sql
├── V2__add_email_index.sql
├── V3__add_status_column.sql

-- V1__create_users_table.sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    created_at TIMESTAMP DEFAULT NOW()
);
```

**DevOps-контекст**: 
- Миграции — часть CI/CD: запускайте при деплое, с проверкой перед применением
- Для zero-downtime деплоя: используйте backward-compatible изменения (добавляйте колонки с DEFAULT, не удаляйте сразу)
- Мониторьте длительность миграций: долгие блокировки могут вызвать downtime
- Храните историю миграций в репозитории — это документация эволюции схемы

**Связанные инструменты**: Flyway, Liquibase, Alembic, pt-online-schema-change, pgonline.

### 21. ⭐ Архитектура распределённых БД: CAP-теорема, консистентность
**Суть**: В распределённых системах невозможно одновременно гарантировать все три свойства: Consistency, Availability, Partition tolerance.

**Подробно**:
```
CAP-теорема (Brewer, 2000):
• Consistency (C): все узлы видят одни и те же данные в один момент времени
• Availability (A): каждый запрос получает ответ (успех или ошибка), без гарантии актуальности
• Partition tolerance (P): система работает при потере связи между узлами

В распределённой системе при сетевом разделении (partition) приходится выбирать:
• CP: жертвуем доступностью ради консистентности (отказываем в записи, если нет кворума)
  - Примеры: PostgreSQL с синхронной репликацией, MongoDB с write concern majority
• AP: жертвуем консистентностью ради доступности (принимаем записи на всех узлах, разрешаем конфликты потом)
  - Примеры: Cassandra, DynamoDB с eventual consistency

Типы консистентности:
1. Strong (линеаризуемость):
   • Операции видны всем узлам мгновенно в глобальном порядке
   • Самый строгий, но самый медленный

2. Sequential:
   • Операции от одного клиента видны в порядке отправки
   • Разные клиенты могут видеть разный порядок

3. Causal:
   • Сохраняется причинно-следственная связь: если операция B зависит от A, все увидят A→B

4. Eventual:
   • При отсутствии новых записей все реплики со временем станут одинаковыми
   • Быстро, масштабируемо, но приложение должно обрабатывать временную несогласованность

Конфликты при eventual consistency:
• Last-write-wins: побеждает запись с более поздним таймстампом (потеря данных!)
• Vector clocks: отслеживание причинности, приложение разрешает конфликты
• CRDT (Conflict-free Replicated Data Types): математические структуры, гарантирующие сходимость

**DevOps-контекст**: 
- Выбор между CP/AP зависит от бизнес-требований: финансы → CP, соцсеть → AP
- Eventual consistency требует изменений в приложении: idempotent операции, версионирование, разрешение конфликтов
- Мониторьте replication lag и консистентность: метрики "stale reads", конфликтующие записи
- Документируйте гарантии вашей БД: что приложение может ожидать при сбоях

**Связанные инструменты**: Consul (service discovery), Kafka (event sourcing), CRDT-библиотеки.

### 22. ⭐ Как работает Citus/TimescaleDB: шардирование + репликация?
**Суть**: Расширения PostgreSQL для горизонтального масштабирования: Citus — для шардирования, TimescaleDB — для временных рядов.

**Подробно**:
```
Citus (дистрибутивный PostgreSQL):
Архитектура:
• Coordinator node: принимает запросы, планирует выполнение, собирает результаты
• Worker nodes: хранят шарды данных, выполняют локальные запросы
• Прозрачность: приложение подключается к coordinator как к обычной БД

Шардирование:
• Таблицы делятся на шарды по distribution column (обычно tenant_id, user_id)
• Каждый шард — обычная таблица на worker'е
• Распределённые запросы: coordinator разбивает запрос, отправляет на нужные шарды, объединяет результаты

Оптимизации:
• Router queries: запросы с фильтром по distribution column идут напрямую на один шард (быстро)
• Distributed joins: если таблицы распределены по одному ключу — join выполняется локально на шардах
• Reference tables: маленькие справочники копируются на все шарды для эффективных join

TimescaleDB (временные ряды на PostgreSQL):
Концепция hypertable:
• Логическая таблица, автоматически разбитая на chunks по времени (и опционально по пространству)
• Каждый chunk — обычная таблица, можно перемещать между табличными пространствами

Оптимизации для временных рядов:
• Автоматическое создание индексов по времени
• Compression: старые данные сжимаются колонками (до 90% экономии)
• Continuous aggregates: материализованные представления, обновляемые инкрементально
• Retention policies: автоудаление старых данных

Общее:
• Оба расширения используют стандартный PostgreSQL-протокол — совместимы с существующими драйверами
• Репликация: каждый узел (coordinator/worker) может быть частью streaming replication
• Бэкапы: координатор + все воркеры, или использовать pgBackRest с поддержкой распределённых БД

**DevOps-контекст**: 
- Citus — когда нужно масштабировать многопользовательское приложение (SaaS, multi-tenant)
- TimescaleDB — для метрик, мониторинга, IoT, где данные привязаны ко времени
- Планируйте distribution key заранее: смена требует полной перестройки
- Мониторьте распределение данных: перекос ведёт к перегрузке отдельных воркеров
- Бэкапы сложнее: нужно координировать снимки всех узлов или использовать специализированные инструменты

**Связанные инструменты**: Citus, TimescaleDB, pgBackRest, patroni для координации.

### 23. ⭐ Профилирование и тюнинг: shared_buffers, work_mem, checkpoint
**Суть**: Настройка параметров PostgreSQL для баланса между производительностью, стабильностью и потреблением ресурсов.

**Подробно**:
```
Ключевые параметры памяти:

shared_buffers:
• Общий кэш данных и индексов
• Рекомендация: 25-40% от RAM (не больше, т.к. ОС тоже кэширует)
• Слишком мало → частые чтения с диска
• Слишком много → вытеснение кэша ОС, долгий старт

work_mem:
• Память на операцию сортировки или хэш-таблицы (на операцию, не на сессию!)
• Один сложный запрос с несколькими сортировками может использовать N * work_mem
• Рекомендация: 4-16MB для общего случая, увеличивать для аналитических запросов
• Слишком много → OOM при параллельных сложных запросах

maintenance_work_mem:
• Для операций обслуживания: VACUUM, CREATE INDEX, ALTER TABLE
• Можно ставить больше: 256MB-2GB, т.к. операции редкие и последовательные
• Ускоряет создание индексов на больших таблицах

effective_cache_size:
• Подсказка оптимизатору, сколько памяти доступно для кэширования (БД + ОС)
• Рекомендация: 50-75% от RAM
• Влияет на выбор плана: при большом значении оптимизатор чаще выбирает Index Scan вместо Seq Scan

Checkpoint tuning:

checkpoint_timeout:
• Максимальное время между автоматическими чекпоинтами
• По умолчанию 5 мин, можно увеличить до 15-30 мин для снижения I/O
• Но: больше данных в WAL → дольше восстановление после краша

max_wal_size:
• Максимальный размер WAL перед форсированным чекпоинтом
• Увеличение снижает частоту чекпоинтов, но требует больше места в pg_wal

checkpoint_completion_target:
• Доля времени чекпоинта, в течение которой происходит запись (0.9 = 90% времени)
• Значение ближе к 1.0 "размазывает" запись, снижая пики I/O

Мониторинг и настройка:
```
```sql

FROM pg_settings

- В cloud: учитывайте, что инстансы имеют ограниченную сеть и диски — настройте под них

### 24. DynamoDB: архитектура, partition keys, capacity modes
**Суть**: DynamoDB — полностью управляемая NoSQL БД от AWS с автоматическим масштабированием и предсказуемой производительностью.

**Подробно**:
```
Архитектура:
• Serverless: нет управления серверами, автомасштабирование
• Распределённая: данные автоматически шардируются по партициям
• Репликация: синхронная внутри региона, опционально глобальные таблицы (multi-region)

Модель данных:
• Таблицы: коллекция элементов (items)
• Элемент: набор атрибутов (ключ-значение), как документ
• Ключи:
  - Partition key (обязательно): определяет, на каком шарде хранится элемент
  - Sort key (опционально): сортировка элементов с одинаковым partition key
  - Composite key: partition + sort → диапазонные запросы по sort key

Запросы:
• Query: по partition key (обязательно) + опционально фильтры по sort key
• Scan: полный проход по таблице (дорого, избегайте)
• Глобальные и локальные вторичные индексы (GSI/LSI): альтернативные ключи для запросов

Capacity modes:
1. Provisioned:
   • Вы задаёте WCU/RCU (write/read capacity units)
   • Предсказуемая цена, можно настроить auto-scaling
   • Подходит для стабильной нагрузки

2. On-demand:
   • Автоматическое масштабирование под нагрузку
   • Плата за фактическое использование
   • Подходит для непредсказуемой или пиковой нагрузки

Оптимизации:
• Выбор partition key: равномерное распределение запросов (избегайте "hot partitions")
• Использование sort key для группировки и диапазонных запросов
• Projection в индексах: копировать только нужные атрибуты (экономия места и денег)
• TTL: автоудаление устаревших данных (для сессий, кэша)

Ограничения:
• Размер элемента: до 400 KB
• Нет JOIN, нет транзакций между таблицами (только внутри одной, до 25 элементов)
• Сложные запросы требуют денормализации или выноса логики в приложение

**DevOps-контекст**: 
- Проектируйте схему под запросы, а не под "нормализованную" модель
- Мониторьте ConsumedCapacity и ThrottlingEvents — признаки неправильного подбора capacity
- Используйте DAX (кэш) для read-heavy workloads с низкой задержкой
- Для аналитики: экспортируйте данные в S3 + Athena/Redshift, не делайте сложные запросы в DynamoDB напрямую

**Связанные инструменты**: AWS CLI, boto3, DynamoDB Streams, DAX, AWS Backup.

### 25. Disaster recovery: RPO, RTO, стратегии восстановления
**Суть**: План восстановления после катастрофы определяет, как быстро и с какой потерей данных система вернётся в работу.

**Подробно**:
```
Ключевые метрики:
• RPO (Recovery Point Objective): максимальная допустимая потеря данных (время между последним бэкапом и сбоем)
  - Пример: RPO=15 мин → бэкапы/репликация должны быть не реже чем каждые 15 минут

• RTO (Recovery Time Objective): максимальное допустимое время простоя
  - Пример: RTO=1 час → процедуры восстановления должны уложиться в 60 минут

Стратегии DR (по возрастанию сложности и стоимости):

1. Backup & Restore:
   • Регулярные бэкапы в удалённое хранилище
   • Восстановление: развёртывание новой БД из бэкапа
   - RPO: часы, RTO: часы-дни
   - Дёшево, подходит для некритичных систем

2. Pilot Light:
   • Минимальная копия инфраструктуры в другом регионе (БД в standby, без трафика)
   • При сбое: масштабирование и переключение трафика
   - RPO: минуты (репликация), RTO: десятки минут
   - Баланс стоимости и скорости

3. Warm Standby:
   • Полноценная, но уменьшенная копия системы в другом регионе
   • Принимает read-трафик или готов к быстрому переключению
   - RPO: секунды-минуты, RTO: минуты
   - Дороже, для критичных систем

4. Multi-site Active-Active:
   • Система работает в нескольких регионах одновременно
   • Трафик распределяется, данные реплицируются
   - RPO: ~0, RTO: секунды
   - Максимальная доступность, максимальная стоимость и сложность

Тестирование DR:
• Регулярные учения: восстановление из бэкапа, failover реплики
• Документация: пошаговые runbooks, контакты, эскалация
• Автоматизация: скрипты восстановления, инфраструктура как код

**DevOps-контекст**: 
- Определите RPO/RTO с бизнесом: от этого зависит архитектура и бюджет
- Multi-region — не панацея: сложность консистентности, стоимость трафика, тестирование
- Храните бэкапы и инфраструктуру в другом облаке/провайдере (избегайте единой точки отказа)
- Автоматизируйте восстановление: чем меньше ручных шагов, тем меньше человеческих ошибок при инциденте

**Связанные инструменты**: AWS Backup, pgBackRest, Terraform, Chaos Engineering инструменты.

### 26. Security model БД: аутентификация, авторизация, шифрование
**Суть**: Многоуровневая защита данных: кто может подключиться, что может делать, и как защищены данные в покое и при передаче.

**Подробно**:
```
Аутентификация (кто вы):
• Password-based: scram-sha-256 (PostgreSQL), caching_sha2_password (MySQL)
• SSL-сертификаты: клиент аутентифицируется по сертификату
• IAM-аутентификация (cloud): временные токены вместо паролей
• LDAP/Active Directory: централизованное управление пользователями

Авторизация (что можете делать):
• Роли и привилегии: GRANT/REVOKE прав на уровне БД, схемы, таблицы, колонки
• Принцип минимальных привилегий: приложение получает только необходимые права
• Row-level security (PostgreSQL): фильтрация строк на уровне БД (multi-tenant)
• Views с ограничениями: виртуальные таблицы с предопределёнными фильтрами

Шифрование:
1. In-transit (при передаче):
   • SSL/TLS между приложением и БД
   • Настройка: sslmode=require/verify-full, проверка сертификатов
   • Обязательно в production, особенно в публичных сетях

2. At-rest (на диске):
   • TDE (Transparent Data Encryption): шифрование файлов БД (Enterprise-версии)
   • Filesystem-level: LUKS (Linux), BitLocker (Windows)
   • Cloud: шифрование EBS/RDS через KMS
   • Ключи шифрования должны храниться отдельно от данных (KMS, Vault)

Аудит и мониторинг:
• Логирование подключений, запросов, изменений прав
• ПГ: log_connections, log_statement, pg_audit расширение
• MySQL: audit log plugin, Enterprise Audit
• Алерты на подозрительную активность: множественные неудачные логины, доступ из новых IP

Best practices:
• Никогда не используйте дефолтные пароли и пользователей
• Регулярно ротируйте пароли и ключи
• Разделяйте доступ: dev, staging, production — разные учётки и сети
• Сканируйте зависимости приложения на уязвимости (включая драйверы БД)

**DevOps-контекст**: 
- Шифрование in-transit — обязательно, настройте и протестируйте
- At-rest шифрование требуется регуляторами (GDPR, PCI-DSS) — планируйте заранее
- Аудит-логи занимают место и влияют на производительность — настраивайте выборочно
- Интегрируйте управление секретами: HashiCorp Vault, AWS Secrets Manager, Azure Key Vault

**Связанные инструменты**: pg_audit, HashiCorp Vault, AWS KMS, OpenSSL, fail2ban.

### 27. Как работает мониторинг БД: метрики, алерты, slow queries
**Суть**: Наблюдаемость за БД позволяет обнаруживать проблемы до того, как они повлияют на пользователей.

**Подробно**:
```
Ключевые метрики:

Производительность:
• Query latency: p50, p95, p99 время выполнения запросов
• Throughput: QPS (queries per second), TPS (transactions per second)
• Cache hit ratio: % запросов, обслуженных из кэша (цель >99% для PG)
• Replication lag: отставание реплик в секундах/байтах

Ресурсы:
• CPU, memory, disk I/O, network I/O
• Connections: active, idle, waiting, max
• Disk usage: данные, WAL, логи, бэкапы

Здоровье:
• Uptime, restarts, crash count
• Locks: количество и длительность блокировок
• Long-running queries: запросы, выполняющиеся дольше порога
• Autovacuum stats: задержки, эффективность

Сбор метрик:
• Встроенные: pg_stat_*, information_schema, SHOW STATUS
• Экспортёры: postgres_exporter, mysqld_exporter, redis_exporter (для Prometheus)
• APM: Datadog, New Relic, Dynatrace (агенты собирают и анализируют)

Slow query log:
• Логирование запросов, выполняющихся дольше порога
• Настройка:
  - PostgreSQL: log_min_duration_statement = 1000 (мс)
  - MySQL: slow_query_log = ON, long_query_time = 1
• Анализ: pt-query-digest, pgBadger, персистентное хранение для трендов

Алертинг:
• Пороговые: "replication lag > 60s", "disk usage > 80%"
• Аномалии: "резкий рост ошибок подключения", "падение throughput"
• Эскалация: warning → critical → страница онколлу
• Подавление: не спамить при известных инцидентах или плановых работах

**DevOps-контекст**: 
- Начинайте с базовых метрик: connections, disk, replication lag
- Настройте алерты на действия: не просто "проблема", а "что делать"
- Храните историю метрик: для постмортема и планирования масштабирования
- Интегрируйте с incident management: PagerDuty, Opsgenie, Slack

**Связанные инструменты**: Prometheus + Grafana, pg_stat_statements, pgBadger, pt-query-digest, Percona Monitoring and Management.

### 28. Multi-region deployment: active-active, active-passive
**Суть**: Размещение БД в нескольких регионах для снижения задержки, повышения доступности и соответствия регуляторным требованиям.

**Подробно**:
```
Архитектурные паттерны:

1. Active-Passive (один регион пишет, другие — read-only или standby):
   • Проще в реализации и консистентности
   • Переключение при сбое региона: promote standby, перенаправить трафик
   • RPO зависит от репликации (асинхронная → возможна потеря)
   • Пример: PostgreSQL streaming replication между регионами

2. Active-Active (запись в нескольких регионах):
   • Максимальная доступность и низкая задержка для локальных пользователей
   • Сложность: разрешение конфликтов при одновременной записи в разных регионах
   • Подходы:
     - Last-write-wins (просто, но потеря данных)
     - CRDT (математически согласованные структуры)
     - Приложение-уровень: версионирование, ручное слияние
   • Примеры: DynamoDB Global Tables, Cassandra multi-DC, YugabyteDB

3. Geo-partitioning (данные привязаны к региону пользователя):
   • Данные европейского пользователя хранятся в ЕС (соответствие GDPR)
   • Запросы маршрутизируются по локации пользователя
   • Репликация для аналитики или аварийного восстановления

Проблемы multi-region:
• Задержка сети: репликация между континентами — сотни миллисекунд
• Консистентность: чем строже, тем выше задержка записи
• Стоимость: трафик между регионами, дублирование инфраструктуры
• Сложность операций: деплой, миграции, бэкапы нужно координировать

Best practices:
• Определите требования: нужна ли запись во всех регионах или достаточно чтения?
• Тестируйте failover: автоматический и ручной, с проверкой данных
• Мониторьте межрегиональную задержку и консистентность
• Документируйте процедуру переключения: кто, когда, как принимает решение

**DevOps-контекст**: 
- Multi-region — не "включить и забыть", а постоянная операционная нагрузка
- Используйте managed-сервисы (RDS Multi-AZ, Aurora Global) для снижения операционных затрат
- Планируйте бюджет: межрегиональный трафик может быть дорогим
- Автоматизируйте деплой: инфраструктура как код (Terraform) для воспроизводимости

**Связанные инструменты**: AWS Global Accelerator, Terraform, Consul, Istio multi-cluster.

### 29. Альтернативные подходы: NewSQL, serverless DB, vector DB
**Суть**: Новые классы баз данных для специфичных сценариев: горизонтальное масштабирование с SQL, автоматическое управление, поиск по векторам.

**Подробно**:
```
NewSQL (SQL + горизонтальное масштабирование):
• Цель: сохранить реляционную модель и ACID, но масштабироваться как NoSQL
• Примеры: CockroachDB, YugabyteDB, TiDB, Google Spanner
• Архитектура:
  - Распределённое хранение с репликацией
  - Распределённые транзакции (через 2PC или более продвинутые протоколы)
  - SQL-интерфейс, совместимость с существующими приложениями
• Use case: когда нужен масштаб NoSQL, но нельзя отказаться от реляционной модели

Serverless databases:
• Цель: убрать управление инфраструктурой, платить только за использование
• Примеры: Aurora Serverless, DynamoDB on-demand, PlanetScale, Neon
• Особенности:
  - Автомасштабирование под нагрузку (вплоть до нуля)
  - Пауза при неактивности (экономия)
  - Ограничения: холодный старт, лимиты на длительность запросов
• Use case: непредсказуемая нагрузка, dev/staging, стартапы

Vector databases (для AI/ML):
• Цель: эффективный поиск по векторным эмбеддингам (похожесть, а не точное совпадение)
• Примеры: Pinecone, Weaviate, Milvus, pgvector (расширение PostgreSQL)
• Индексы: HNSW, IVF, PQ — для быстрого приближённого поиска ближайших соседей (ANN)
• Use case: семантический поиск, рекомендации, RAG-приложения с LLM

Выбор подхода:
```
```
| Требование | Рекомендация |
|------------|-------------|
| Горизонтальное масштабирование + строгая консистентность | NewSQL (CockroachDB) |
| Непредсказуемая нагрузка, минимальные операционные затраты | Serverless (Aurora Serverless) |
| Поиск по смыслу, AI-приложения | Vector DB (pgvector, Pinecone) |
| Простота, экосистема, предсказуемость | Классический PostgreSQL/MySQL |
```

**DevOps-контекст**: 
- Не гонитесь за "модными" технологиями: оцените зрелость, экосистему, навыки команды
- Serverless экономит деньги при переменной нагрузке, но может быть дороже при стабильной высокой
- Vector search можно добавить в существующую БД через расширение (pgvector) — не обязательно заводить новый сервис
- Тестируйте в staging: новые технологии могут иметь неожиданные ограничения

**Связанные инструменты**: pgvector, CockroachDB, PlanetScale, Pinecone, FAISS.

### 30. ⭐ Принципы проектирования отказоустойчивой архитектуры БД
**Суть**: Набор архитектурных принципов для создания систем, устойчивых к сбоям, нагрузке и человеческим ошибкам.

**Подробно**:
```
Принцип 1: Изоляция сбоев (Bulkheads)
• Разделяйте БД по доменам/сервисам: сбой в одной не должен валить другие
• Используйте отдельные пулы соединений, лимиты ресурсов
• Пример: users-db, orders-db, analytics-db — разные инстансы

Принцип 2: Грейсфул деградация
• При проблемах с БД: возвращать кэшированные данные, упрощённые ответы, а не 500
• Circuit breaker: после нескольких ошибок — быстро возвращать ошибку, не нагружая БД
• Timeout на всех уровнях: приложение → драйвер → сеть → БД

Принцип 3: Идемпотентность и компенсирующие транзакции
• Операции с побочными эффектами должны быть безопасны при повторе
• Если нельзя сделать атомарно — проектируйте компенсирующие действия (откат)
• Пример: "списать деньги" → если "зачислить" упало, запустить компенсацию "вернуть деньги"

Принцип 4: Наблюдаемость и само-диагностика
• Метрики, логи, трейсы — не для отладки постфактум, а для предотвращения
• Алерты на ранние признаки: рост лага, падение cache hit, увеличение lock wait
• Health checks: не только "БД отвечает", но и "может выполнять типичные операции"

Принцип 5: Автоматизация восстановления
• Runbooks как код: скрипты восстановления в репозитории
• Автоматический failover с проверками: не переключать, если реплика отстаёт
• Тестирование восстановления в CI/CD: "chaos engineering" для БД

Принцип 6: Эволюционируемость схемы
• Изменения схемы должны быть backward-compatible
• Добавляйте колонки с DEFAULT, не удаляйте сразу — сначала депопулируйте
• Используйте feature flags для постепенного внедрения изменений

Архитектурный чеклист:
```
```
[ ] Есть репликация с мониторингом лага
[ ] Бэкапы тестируются регулярно (не реже раза в квартал)
[ ] Приложение использует retry + circuit breaker при работе с БД
[ ] Критичные операции идемпотентны или имеют компенсацию
[ ] Метрики БД интегрированы в общий мониторинг и алертинг
[ ] Схема БД версионируется и мигрируется через CI/CD
[ ] Есть документированный план DR с назначенными ролями
```

**DevOps-контекст**: 
- Отказоустойчивость — не фича, а результат системного подхода на всех уровнях
- Инвестиции в архитектуру окупаются при росте: меньше инцидентов, быстрее восстановление
- Начинайте с малого: один принцип за итерацию, но двигайтесь последовательно
- Документируйте решения: почему выбрали такую репликацию, какие компромиссы приняли
