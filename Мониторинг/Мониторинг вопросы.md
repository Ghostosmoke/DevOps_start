## 🟢 Junior (базовое понимание)

### 1. Что такое система контроля версий и чем распределённая (Git) отличается от централизованной (SVN)?
**Кратко**: СКВ отслеживает изменения в коде; в распределённой каждый имеет полную копию истории.

**Подробно**: Централизованная (SVN) хранит историю только на сервере — для коммита или просмотра истории нужен доступ к сети. В Git каждый `clone` — это полный бэкап: можно коммитить, ветвиться, смотреть `blame` офлайн. Синхронизация с удалённым репо — явная (`fetch`/`push`).

**DevOps-контекст**: Распределённость критична для отказоустойчивости: если сервер упал, любой разработчик или раннер может временно стать источником истины. Также ускоряет CI: раннеры работают с локальной историей без сетевых задержек.

**Команды**: `git clone`, `git commit`, `git push` vs `svn checkout`, `svn commit`.

### 2. Какие три основных состояния файлов в Git ты знаешь?
**Кратко**: `untracked` (не отслеживается), `modified` (изменён), `staged` (добавлен в индекс).

**Подробно**: 
- `untracked`: новый файл, Git его не видит в истории → `git add` переводит в `staged`
- `modified`: файл из репо изменён в рабочей папке → `git diff` покажет изменения
- `staged`: изменения подготовлены к коммиту → `git diff --staged` покажет, что войдёт в коммит

**DevOps-контекст**: Понимание состояний помогает избежать ошибок в CI: например, не закоммитить `.env` из-за забытого `.gitignore`, или не пропустить критичный файл из-за слепого `git add .`.

**Команды**: `git status`, `git add`, `git restore --staged`.

### 3. В чём разница между `git add` и `git commit`?
**Кратко**: `add` помещает изменения в индекс (подготовка), `commit` фиксирует индекс в истории.

**Подробно**: `git add file` копирует текущее содержимое файла в объект `blob` и обновляет `index`. `git commit` создаёт объект `tree` из индекса, затем объект `commit`, связывает с родителем и обновляет ссылку ветки.

**DevOps-контекст**: Разделение позволяет атомарно группировать изменения: например, закоммитить только фикс бага, оставив в рабочей папке экспериментальный код для следующего коммита. В пайплайнах это позволяет точечно тестировать изменения.

**Команды**: `git add -p` для интерактивного выбора, `git commit -m "msg"`.

### 4. Что делает команда `git status` и зачем она нужна перед коммитом?
**Кратко**: Показывает состояние файлов: какие изменены, добавлены, удалены, что в индексе.

**Подробно**: Сравнивает `HEAD` (последний коммит), `index` (подготовленное) и рабочую папку. Выводит цветом и кратко (`git status -s`): `??` новый, `M` изменён, `D` удалён, `A` добавлен.

**DevOps-контекст**: Предотвращает коммит секретов, логов, временных файлов. В CI можно использовать `git status --porcelain` для скриптовой проверки "чистоты" рабочей папки перед сборкой.

**Команды**: `git status --short`, `git status --porcelain=v1`.

### 5. Как создать новую ветку и переключиться на неё одной командой?
**Кратко**: `git checkout -b <name>` или современная `git switch -c <name>`.

**Подробно**: Команда создаёт файл `.git/refs/heads/<name>` с хешом текущего коммита, затем обновляет `HEAD` на `ref: refs/heads/<name>` и делает checkout соответствующего `tree` в рабочую папку.

**DevOps-контекст**: В командах с частыми ветками (`feature/*`, `hotfix/*`) это базовая операция. В пайплайнах ветки часто создаются автоматически по именам коммитов или задач в трекере.

**Команды**: `git switch -c feature/JIRA-123`, `git branch -m old new` для переименования.

### 6. Что такое `HEAD` и что значит "detached HEAD"?
**Кратко**: `HEAD` — указатель на текущую ветку или коммит; "detached" — когда он указывает напрямую на коммит, а не на ветку.

**Подробно**: Обычно `HEAD` содержит `ref: refs/heads/main`. При `git checkout abc123` он становится `abc123`. Новые коммиты в этом состоянии не привязаны к ветке и могут быть потеряны при переключении.

**DevOps-контекст**: В CI часто проверяют конкретный коммит или тег — это detached HEAD. Важно не пытаться пушить из такого состояния без создания ветки.

**Команды**: `git checkout -b temp` для сохранения коммитов из detached, `git symbolic-ref HEAD` чтобы проверить, на что указывает HEAD.

### 7. Как отменить изменения в файле, который ещё не добавлен в индекс?
**Кратко**: `git restore <file>` или `git checkout -- <file>`.

**Подробно**: Команда копирует содержимое файла из `HEAD` (последнего коммита) в рабочую папку, перезаписывая локальные изменения. Если файл новый (`untracked`) — сначала нужно добавить, потом отменять.

**DevOps-контекст**: В скриптах можно использовать для "сброса" тестовых конфигов перед запуском. В CI — для очистки артефактов между этапами сборки.

**Команды**: `git restore .` для всех файлов, `git restore --source=main file.txt` для взятия из другой ветки.

### 8. Зачем нужен файл `.gitignore` и какие файлы туда обычно добавляют в DevOps-проектах?
**Кратко**: Исключает файлы из отслеживания; в DevOps — секреты, артефакты, зависимости, логи.

**Подробно**: `.gitignore` применяется до `git add`: даже `git add .` проигнорирует указанные паттерны. Поддерживает глобальный (`~/.gitignore`) и локальный (`.git/info/exclude`) игнор.

**DevOps-контекст**: Предотвращает утечку секретов в репо. В CI можно проверять, что `.gitignore` покрывает все чувствительные файлы, с помощью `gitleaks` или `trufflehog`.

**Команды**: `git check-ignore -v <file>` для отладки, почему файл игнорируется.

### 9. Что делает `git clone --depth=1` и зачем это используют в CI?
**Кратко**: Скачивает только последний коммит (shallow clone), экономя время и место.

**Подробно**: Не загружает полную историю, только текущий `tree` и `commit`. Ограничивает операции: нельзя делать `rebase` на старые коммиты, `git log` обрезан.

**DevOps-контекст**: Критично для ephemeral CI-раннеров: ускоряет клонирование с минут до секунд, экономит дисковое пространство. Для тегированных релизов можно использовать `--depth=1 --branch=v1.2.3`.

**Команды**: `git fetch --unshallow` чтобы превратить shallow в полный клон при необходимости.

[[#🗂️ Оглавление для навигации|↑ К оглавлению]]

### 10. Как посмотреть историю коммитов в компактном формате?
**Кратко**: `git log --oneline --graph --all`.

**Подробно**: `--oneline` — хеш + сообщение в одну строку, `--graph` — ASCII-ветвление, `--all` — все ветки. Можно добавить `--decorate` для отображения тегов и имён веток.

**DevOps-контекст**: Быстрый аудит изменений перед деплоем. В скриптах: `git log --format=%H%n%s -1` для получения хеша и сообщения последнего коммита.

**Команды**: `git log -n 10 --stat` для просмотра изменённых файлов, `git log --since="1 week"` для фильтрации по времени.

## 🟡 Middle (применение, нюансы)

### 11. ⭐ В чём разница между `git merge` и `git rebase`? Когда что использовать?
**Кратко**: `merge` создаёт коммит слияния, сохраняя историю ветвления; `rebase` переписывает историю, делая её линейной.

**Подробно**: 
- `merge`: безопасен для публичных веток, сохраняет контекст "что и когда ветвилось", но может создавать шум.
- `rebase`: чистая история, упрощает `bisect`/`blame`, но меняет хеши коммитов — нельзя использовать на ветках, которые уже используют другие.

**DevOps-контекст**: В CI/CD часто используют `rebase` для feature-веток перед мержом в `main`, чтобы избежать лишних коммитов слияния. В релизных ветках (`release/*`) — только `merge` для аудита.

**Команды**: `git merge --no-ff` для сохранения истории, `git rebase -i` для интерактивной чистки, `git pull --rebase` для настройки по умолчанию.

[[#🗂️ Оглавление для навигации|↑ К оглавлению]]

### 12. Что такое fast-forward merge и как его отключить? Зачем это может понадобиться?
**Кратко**: Когда ветка не уходила вперёд, Git просто передвигает указатель; отключается `--no-ff`.

**Подробно**: В режиме FF не создаётся новый коммит — история выглядит линейно. `--no-ff` принудительно создаёт merge-commit, сохраняя информацию о том, что это была отдельная ветка.

**DevOps-контекст**: В проектах с семантическим версионированием и генерацией changelog по коммитам важно сохранять структуру ветвления — `--no-ff` позволяет автоматически группировать изменения по фичам.

**Команды**: `git merge --no-ff feature`, `git config merge.ff false` для глобального отключения.

### 13. Как безопасно отменить последний коммит, если он ещё не отправлен на сервер?
**Кратко**: `git reset --soft HEAD~1` (сохранить изменения в индексе) или `--mixed` (по умолчанию, оставить в рабочей папке).

**Подробно**: 
- `--soft`: отменяет коммит, но оставляет файлы в индексе → можно добавить забытые файлы и перекоммитить.
- `--mixed`: отменяет коммит и индекс → файлы останутся изменёнными в рабочей папке.
- `--hard`: ⚠️ полностью удаляет изменения — использовать только если уверен.

**DevOps-контекст**: В пайплайнах можно использовать `reset --soft` для "переупаковки" коммитов перед пушем, если автоматический коммит оказался неполным.

**Команды**: `git commit --amend` для простого исправления сообщения или добавления забытого файла.

### 14. ⭐ Что делает `git fetch` и чем он отличается от `git pull`? Почему в CI часто используют `fetch`?
**Кратко**: `fetch` скачивает метаданные, не меняя рабочую копию; `pull` = `fetch` + `merge`/`rebase`.

**Подробно**: `fetch` обновляет только `refs/remotes/origin/*` — можно безопасно анализировать, что изменилось на сервере. `pull` сразу пытается влить изменения, что может вызвать конфликты в headless-средах.

**DevOps-контекст**: В CI сначала делают `fetch`, проверяют `CI_COMMIT_SHA`, теги, запускают пре-чеки, и только потом — `merge` или `rebase` по стратегии. Это даёт контроль и предсказуемость.

**Команды**: `git fetch --tags --prune`, `git merge origin/main` или `git rebase origin/main` вручную после анализа.

### 15. Как разрешить конфликт слияния и что делать, если запутался в процессе?
**Кратко**: Отредактировать файлы с маркерами `<<<<<<<`, `=======`, `>>>>>>>`, затем `git add` и `git commit`.

**Подробно**: При конфликте Git помечает проблемные места. Можно использовать `git mergetool` (настроенный diff-тул), или вручную выбрать нужные части. Если запутался — `git merge --abort` отменит слияние и вернёт к состоянию до `merge`.

**DevOps-контекст**: В командах настраивают `rerere.enabled=true`, чтобы Git запоминал, как ты разрешал похожие конфликты, и предлагал автоматическое решение в будущем. В CI можно автоматически отклонять PR с конфликтами.

**Команды**: `git status` для списка конфликтующих файлов, `git checkout --ours/--theirs file` для выбора одной из версий.

### 16. Что такое `git reflog` и в каких ситуациях он может спасти?
**Кратко**: Журнал всех перемещений HEAD и веток; позволяет восстановить "потерянные" коммиты.

**Подробно**: В отличие от `git log`, `reflog` показывает даже недостижимые из веток коммиты. Хранится локально 90 дней по умолчанию. Пример: после `git reset --hard HEAD~1` можно вернуть коммит через `git reset --hard HEAD@{1}`.

**DevOps-контекст**: При отладке пайплайнов можно использовать `reflog` в раннере, чтобы понять, какой коммит реально проверялся. Также спасает при случайном `reset` в production-ветке (если ещё не пушил).

**Команды**: `git reflog show main`, `git diff HEAD@{0} HEAD@{2}`, `git branch recovery HEAD@{3}`.

### 17. Как добавить существующий проект в репозиторий, не теряя историю, если нужно объединить с другим репо?
**Кратко**: Использовать `git subtree` или `git submodule`, либо `git filter-repo` для сложного мержа историй.

**Подробно**: 
- `submodule`: ссылка на внешний репо в определённом коммите — подходит для независимых библиотек.
- `subtree`: встраивает историю другого репо в подпапку — проще для монорепозиториев.
- `filter-repo`: переписывает пути и историю перед мержом — для сложных миграций.

**DevOps-контекст**: При переходе от нескольких репо к монорепозиторию важно сохранить историю для `blame` и аудита. `subtree` проще в поддержке в пайплайнах, чем `submodule`.

**Команды**: `git subtree add --prefix=libs/auth https://repo auth-main --squash`, `git submodule add https://repo libs/auth`.

### 18. Зачем подписывать коммиты (GPG/SSH) и как это настроить?
**Кратко**: Подпись подтверждает авторство и целостность; защищает от подделки коммитов.

**Подробно**: При `commit.gpgsign=true` Git создаёт криптографическую подпись с использованием приватного ключа. Сервер (GitHub/GitLab) может требовать подписанные коммиты для мержа в защищённые ветки.

**DevOps-контекст**: Критично для соблюдения стандартов безопасности (SOC2, ISO27001) и предотвращения атак типа "подмена кода в форке". В пайплайнах можно проверять подпись перед деплоем.

**Команды**: `git config --global gpg.format ssh`, `git config --global user.signingkey ~/.ssh/id_ed25519.pub`, `git config --global commit.gpgsign true`.

### 19. Как работает `git stash` и чем отличается от временного коммита?
**Кратко**: `stash` временно сохраняет изменения вне истории; не создаёт коммит в ветке.

**Подробно**: `git stash push` создаёт специальный коммит в `refs/stash`, содержащий индекс и рабочую папку. `git stash pop` применяет и удаляет из стека. В отличие от временного коммита, stash не виден в `git log` и не пушится.

**DevOps-контекст**: Удобно для переключения контекста в раннере: например, сохранить WIP-изменения, проверить хотфикс, затем вернуться. Не подходит для долгосрочного хранения — stash локален.

**Команды**: `git stash -u` (сохранить untracked файлы), `git stash list`, `git stash apply stash@{2}`.

### 20. Что такое `git bisect` и как с его помощью найти баг-коммит?
**Кратко**: Бинарный поиск по истории для определения коммита, в котором появился баг.

**Подробно**: Указываешь "хороший" (например, `v1.0`) и "плохой" (`HEAD`) коммиты. Git автоматически переключает коммиты посередине, ты тестируешь и помечаешь `git bisect good/bad`. За ~log₂(n) шагов находишь виновника.

**DevOps-контекст**: В автоматизированных пайплайнах можно запускать `git bisect run ./test.sh` — скрипт возвращает 0 (хорошо) / 1 (плохо), и bisect работает полностью автоматически.

**Команды**: `git bisect start`, `git bisect good v1.0`, `git bisect bad HEAD`, `git bisect run npm test`, `git bisect reset` для выхода.

### 21. ⭐ Опиши, как устроена внутренняя база данных Git: какие типы объектов существуют и как они связаны?
**Кратко**: 4 типа объектов: `blob` (файл), `tree` (директория), `commit` (снимок), `tag` (метка); все хранятся в `.git/objects` по хешу.

**Подробно**: 
- `blob`: сырые данные файла + заголовок `blob <size>\0`; хеш = SHA-1(заголовок + данные)
- `tree`: список записей `<режим> <имя> <хеш>`; хеш = от сериализованного списка
- `commit`: заголовок (`tree`, `parent`, `author`, `committer`, `message`) + хеш корневого tree
- `tag`: ссылается на объект + метаданные + подпись (для аннотированных)
- Связи: `commit → tree → (blob | tree)*`; цепочка коммитов через `parent`

**DevOps-контекст**: Понимание объектов позволяет оптимизировать хранилище: например, `git repack` объединяет объекты в пакеты с дельта-компрессией. Также помогает при восстановлении повреждённых репо: `git fsck` проверяет целостность хешей.

**Команды**: `git cat-file -t <hash>` (тип объекта), `git cat-file -p <hash>` (содержимое), `git fsck --full`.

### 22. Почему объекты Git неизменяемы и как это влияет на дедупликацию и производительность?
**Кратко**: Неизменяемость гарантирует целостность через хеши; позволяет дедуплицировать идентичные файлы/деревья.

**Подробно**: Если два коммита содержат одинаковый файл — он хранится как один `blob` с одним хешом. При изменении одной строки создаётся новый `blob`, но все остальные файлы ссылаются на старые объекты. Это экономит место и ускоряет операции сравнения.

**DevOps-контекст**: В больших репо с частыми изменениями конфигов дедупликация критична: без неё история раздувалась бы экспоненциально. Также неизменяемость позволяет безопасно кэшировать объекты в распределённых системах (например, в Git-зеркалах).

**Команды**: `git count-objects -vH` для статистики хранилища, `git verify-pack -v .git/objects/pack/*.idx` для анализа паков.

### 23. Как работает дельта-компрессия и пакинг объектов? Когда и зачем запускать `git gc`?
**Кратко**: Дельта-компрессия хранит похожие объекты как разницы; пакинг объединяет объекты в один файл для ускорения доступа.

**Подробно**: При `git repack` Git анализирует объекты, находит похожие `blob`/`tree` и хранит базовый объект + дельты. Паки (`*.pack`) индексируются (`*.idx`) для быстрого поиска по хешу. `git gc` автоматически запускает пакинг при достижении порогов (число объектов, размер).

**DevOps-контекст**: В монорепозиториях с тысячами разработчиков регулярный `gc` критичен для производительности `clone` и `fetch`. В CI можно запускать `git gc --auto` после сборки, чтобы не раздувать кэш.

**Команды**: `git gc --aggressive --prune=now` для глубокой оптимизации, `git config gc.auto 6700` для настройки триггеров.

### 24. Что происходит "под капотом" при `git push`: как Git определяет, какие объекты отправлять?
**Кратко**: Сравнивает локальные и удалённые хеши веток, вычисляет недостающие объекты через граф достижимости, отправляет дельта-сжатые пакеты.

**Подробно**: 
1. Клиент отправляет `want <sha>` для целевого коммита и `have <sha>` для известных объектов
2. Сервер строит граф, находит разницу (objects reachable from `want`, но не от `have`)
3. Объекты упаковываются с дельта-компрессией, передаются по протоколу (smart HTTP / SSH)
4. Сервер проверяет хеши, обновляет ссылку, запускает хуки

**DevOps-контекст**: Понимание этого помогает оптимизировать пуши: например, делать `fetch` перед `push`, чтобы уменьшить разницу. Также объясняет, почему пуш больших бинарников медленный — они не дедуплицируются так же эффективно, как код.

**Команды**: `GIT_TRACE_PACKET=1 git push` для отладки протокола, `git push --dry-run` для проверки без отправки.

### 25. ⭐ Как безопасно переписывать историю в feature-ветке и почему `--force-with-lease` безопаснее `--force`?
**Кратко**: Переписывать можно только локальные/неопубликованные ветки; `--force-with-lease` проверяет, что удалённая ветка не изменилась с момента последнего `fetch`.

**Подробно**: 
- `--force` слепо перезаписывает remote, даже если туда запушили новые коммиты → потеря чужой работы
- `--force-with-lease` сравнивает ожидаемый хеш (из `refs/remotes/origin/branch`) с реальным на сервере; если не совпадает — отклоняет пуш
- Перед переписыванием: убедиться, что ветка не используется другими, сделать бэкап (`git branch backup`)

**DevOps-контекст**: В командах с активным параллельным развитием `--force-with-lease` — обязательный стандарт. В защищённых ветках сервер вообще блокирует force-push через Branch Protection Rules.

**Команды**: `git push --force-with-lease origin feature`, `git config push.default simple` для предотвращения случайных пушей не в ту ветку.

### 26. Как удалить чувствительные данные (пароль, ключ) из всей истории репозитория? Какие есть инструменты и подводные камни?
**Кратко**: Использовать `git filter-repo` (или BFG) для переписывания истории; обязательно ротировать секрет и проверить аудит.

**Подробно**: 
- `git filter-repo --invert-paths --path secret.env --force` удалит файл из всех коммитов
- Старые объекты останутся в `.git/objects` до `git gc --prune=now`
- После переписывания: `git push --force-with-lease`, уведомить команду, проверить, что никто не клонировал после утечки
- ⚠️ `git rm` или `commit --amend` не удаляют данные из истории — они остаются доступны по хешу

**DevOps-контекст**: В инцидент-менеджменте утечка секрета — критичный инцидент: сначала ротируй ключ в источнике (Vault, AWS), потом чисти историю. Автоматизируй сканирование в пре-коммит хуках и в CI (`gitleaks`, `trufflehog`).

**Команды**: `git filter-repo --path-glob '*.env' --invert-paths`, `BFG --delete-files secret.key`, `git reflog expire --expire=now --all && git gc --prune=now`.

### 27. В чём преимущества и риски использования монорепозитория с точки зрения работы Git?
**Кратко**: Плюсы: атомарные изменения, единое версионирование; минусы: размер репо, скорость операций, сложность прав доступа.

**Подробно**: 
- ✅ Атомарные коммиты: изменение в сервисе + инфраструктуре в одном коммите
- ✅ Единый `git blame` и история для кросс-сервисных изменений
- ❌ `clone` и `status` замедляются с ростом числа файлов
- ❌ Сложнее настраивать fine-grained доступ (все видят всё)

**DevOps-контекст**: Для монорепозиториев критичны: `--filter=blob:none`, `sparse-checkout`, кэширование зависимостей, `path-based triggers` в CI (запускать пайплайн только если изменился `apps/service-a/**`). Инструменты: `bazel`, `nx`, `turborepo` для инкрементальной сборки.

**Команды**: `git sparse-checkout set apps/service-a infra/terraform`, `git clone --filter=blob:none --sparse <url>`.

### 28. Как оптимизировать клонирование и работу с огромным репозиторием (>10 Гб, 100k+ файлов)?
**Кратко**: Использовать shallow/partial clone, sparse checkout, пакинг, кэширование.

**Подробно**: 
- `--depth=1` для получения только актуального кода (если не нужна история)
- `--filter=blob:none` + `--sparse` для загрузки метаданных и только нужных подпапок
- На сервере: регулярный `git repack -Ad --depth=250 --window=250` для оптимального пакинга
- На клиенте: `core.preloadindex=true`, `fetch.parallel=10` для ускорения операций

**DevOps-контекст**: В крупных организациях используют зеркала репо с оптимизированной структурой, CDN для раздачи `.pack`-файлов, прокси для Git-трафика. В CI — кэширование `.git` между запусками через `cache` или `artifact`.

**Команды**: `git config --global fetch.parallel 10`, `git clone --filter=blob:none --sparse --depth=1 <url>`, `git sparse-checkout init --cone`.

### 29. Как устроены хуки в Git и как их использовать для enforcement политик в команде?
**Кратко**: Хуки — исполняемые скрипты в `.git/hooks/`, срабатывают при событиях; для командного enforcement используют распространяемые фреймворки.

**Подробно**: 
- Клиентские хуки: `pre-commit`, `commit-msg`, `pre-push` — локальные, не коммитятся
- Серверные: `pre-receive`, `update`, `post-receive` — настраиваются на Git-сервере
- Для распространения: `husky` (JS), `pre-commit framework` (Python, мультиязычный), или копирование через `init.templateDir`

**DevOps-контекст**: В пайплайнах хуки дублируются: `pre-commit` → линт в CI, `commit-msg` → валидация формата сообщения в `merge_request`. Серверные хуки заменяются на Webhooks + CI-статусы для гибкости.

**Команды**: `pre-commit install`, `git config init.templateDir ~/.git-templates`, пример `.pre-commit-config.yaml` с `gitleaks`, `black`, `terraform validate`.

### 30. ⭐ Как спроектировать стратегию ветвления и релизов для проекта с ежедневными деплоями и 50+ разработчиками?
**Кратко**: Trunk-Based Development + feature flags + короткоживущие ветки + автоматизированные чеки + защищённый `main`.

**Подробно**: 
- `main` — всегда деплоябельный, защищён: требуется 2 аппрувера, зелёный CI, подписанные коммиты
- Ветки `feature/*` живут <1-2 дня, мержатся через PR с squash/rebase
- Feature flags для недоделанных функций: код в `main`, но функционал выключен
- Релизы через теги: `git tag -a v1.2.3 -m "release"` → CI собирает артефакты, деплоит
- Откаты: `git revert` (не `reset`), чтобы сохранить аудит

**DevOps-контекст**: Такая стратегия позволяет делать 10-100 деплоев в день без хаоса. Требует: покрытие тестами >80%, автоматический rollback при росте ошибок, мониторинг после деплоя. Инструменты: GitHub Actions/GitLab CI для пайплайнов, ArgoCD для GitOps-синхронизации.

**Команды**: Настройка Branch Protection в веб-интерфейсе, `git tag -s v1.2.3` для подписанного тега, `git revert -m 1 <merge-commit>` для отката мержа.
