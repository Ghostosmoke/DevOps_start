# 🟢 Junior (базовое понимание)

### 1. Что такое ядро Linux и чем оно отличается от пользовательского пространства?

1. **Кратко:** Ядро — привилегированная часть ОС (Ring 0), управляющая железом; User Space — обычные программы (Ring 3), работающие через syscalls.
2. **Подробно:** Ядро работает в защищённом режиме, имеет прямой доступ к памяти, устройствам, планировщику. Пользовательские процессы изолированы: любой доступ к ресурсам идёт через системные вызовы (`open`, `read`, `fork`). Переключение контекста между режимами — дорогая операция.
3. **DevOps-контекст:** Понимание границ объясняет, почему контейнеры делят ядро хоста, почему `strace` работает (перехват syscalls), и почему падение ядра = крах всего хоста. Также важно для настройки `sysctl`, `seccomp`, `cgroups`.
4. **Команды:** `dmesg`, `cat /proc/sys/kernel/*`, `strace -p <PID>`, `man 2 syscalls`.

### 2. Как посмотреть занимаемое место на диске и количество свободных inodes?

1. **Кратко:** `df -h` — место в блоках, `df -i` — inodes.
2. **Подробно:** Файловая система хранит метаданные файлов в inodes. Если inodes закончились (например, много мелких файлов), создать новый файл нельзя, даже если есть свободное место. `df -i` показывает использование inodes в процентах.
3. **DevOps-контекст:** Критично для мониторинга кэшей CI, логов, временных файлов. Частая причина "тихих" падений сборщиков: место есть, а записать нельзя. В Prometheus-экспортерах есть метрики `node_filesystem_files_free`.
4. **Команды:** `df -hi`, `stat -f .`, `find /path -xdev -type f | wc -l` (приблизительная оценка).

### 3. ⭐ В чём разница между `kill`, `kill -15` и `kill -9`?

1. **Кратко:** `kill` по умолчанию шлёт `SIGTERM` (15), `kill -15` — явно `SIGTERM`, `kill -9` — `SIGKILL`.
2. **Подробно:** `SIGTERM` можно перехватить в коде, выполнить cleanup (закрыть соединения, сохранить состояние) и корректно завершиться. `SIGKILL` обрабатывается ядром мгновенно, приложение не может выполнить никаких действий. `kill -0` проверяет существование процесса без отправки сигнала.
3. **DevOps-контекст:** Graceful shutdown в Kubernetes/Docker relies on `SIGTERM` с `terminationGracePeriodSeconds`. `SIGKILL` = потеря соединений, повреждение БД, некорректный статус в service discovery. Всегда давайте приложению время на завершение.
4. **Команды:** `kill -15 <PID>`, `kill -0 <PID>` (проверка), `systemctl stop <service>`, `pkill -TERM -f pattern`.

### 4. Что делают `chmod 755` и `chmod 644`?

1. **Кратко:** `755` = владелец `rwx`, группа/остальные `r-x`; `644` = владелец `rw-`, остальные `r--`.
2. **Подробно:** Цифры в восьмеричной системе: `r=4, w=2, x=1`. Суммируются по категориям: владелец/группа/остальные. `x` для файла = можно выполнить, для директории = можно войти (`cd`).
3. **DevOps-контекст:** Фундамент безопасности. Неправильные права = уязвимости (мировые `w`) или отказ в обслуживании (нет `x` на директории). В контейнерах важно: процесс внутри может не иметь прав на запись в смонтированный volume.
4. **Команды:** `chmod 755 script.sh`, `chmod 644 config.yml`, `find . -type f -exec chmod 644 {} \;`, `ls -l`.

### 5. Как посмотреть логи systemd-сервиса и отфильтровать их по времени?

1. **Кратко:** `journalctl -u <service> --since "2026-04-11 10:00" --until "11:00"`.
2. **Подробно:** `systemd-journald` собирает логи в бинарный формат с метаданными (юнит, приоритет, PID, hostname). Поддерживает фильтрацию по юниту, времени, уровню (`-p`), по полю (`_PID=`, `SYSLOG_IDENTIFIER=`).
3. **DevOps-контекст:** В CI/CD и продакшене `journalctl` заменяет парсинг текстовых `/var/log/*.log`. Позволяет быстро искать инциденты, экспортировать логи в Loki/ELK через `journalctl -o json`.
4. **Команды:** `journalctl -u nginx -f -p err`, `journalctl --disk-usage`, `journalctl --vacuum-time=7d`, `journalctl -b -1` (логи прошлого бута).

### 6. Что такое swap и при каких условиях система начинает его активно использовать?

1. **Кратко:** Swap — резервная область на диске для выгрузки неиспользуемых страниц памяти.
2. **Подробно:** Используется, когда RAM заканчивается или страницы долго не accessed (неактивны). Поведение настраивается через `vm.swappiness` (0–100, по умолч. 60): чем выше, тем агрессивнее ядро свопит. При `swappiness=1` — только при критической нехватке.
3. **DevOps-контекст:** В высоконагруженных системах (БД, кэши) swap часто отключают или ставят `swappiness=1`, так как дисковый I/O убивает latency. Но полное отключение (`swapoff`) рискованно: при утечке памяти OOM Killer сработает жёстче.
4. **Команды:** `free -h`, `cat /proc/sys/vm/swappiness`, `sysctl -w vm.swappiness=10`, `swapon --show`.

# 🟡 Middle (применение, нюансы)

### 7. Что такое zombie-процессы, как они появляются и как их правильно устранять?

1. **Кратко:** Zombie — процесс, завершившийся, но не "прочитанный" родителем (не вызван `waitpid`).
2. **Подробно:** Ядро сохраняет exit status и минимальные метаданные, пока родитель не вызовет `wait()`. Устраняется: 1) исправлением кода родителя (обработка `SIGCHLD`), 2) убийством родителя (zombie переходит к init, который делает `reap`), 3) перезапуском сервиса.
3. **DevOps-контекст:** В CI-раннерах и оркестраторах зомби накапливаются при неправильном завершении дочерних процессов → утечка PID, отказ `fork()`. Мониторинг: метрика `node_procs_zombie` в node_exporter.
4. **Команды:** `ps aux | awk '$8=="Z"'`, `pstree -p`, `kill -9 <PPID>` (если родитель не репит), `top` (строка `zombie`).

### 8. ⭐ Как работает механизм `capabilities` в Linux и чем он безопаснее `setuid`?

1. **Кратко:** Capabilities делят root-права на гранулярные биты (напр. `CAP_NET_BIND_SERVICE`); `setuid` даёт полный root.
2. **Подробно:** Вместо `setuid root` на бинарнике, процессу выдаётся только нужный capability. Принцип наименьших привилегий: даже при компрометации процесса атакующий не получит полный доступ. Capabilities привязываются к файлу (`setcap`) или процессу (`capsh`).
3. **DevOps-контекст:** Критично для контейнеров и сервисов, слушающих порты <1024, меняющих время, монтирующих ФС. В Docker: `--cap-add=NET_BIND_SERVICE` вместо `--privileged`. В Kubernetes: `securityContext.capabilities.add`.
4. **Команды:** `capsh --print`, `setcap cap_net_bind_service=+ep /usr/bin/myapp`, `getcap -r / 2>/dev/null`, `docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE`.

### 9. Почему команда `df` показывает, что место закончилось, а `du` этого не подтверждает?

1. **Кратко:** Файл удалён из каталога, но открыт процессом → inode не освобождён, место занято.
2. **Подробно:** `df` читает суперблок ФС (показывает реальное занятое место на диске). `du` обходит каталоги и суммирует размеры файлов. Если ссылка удалена (`unlink`), но процесс держит дескриптор, `du` файл не видит, но место остаётся занятым.
3. **DevOps-контекст:** Частая ситуация: логи ротейтятся (`logrotate`), но старые дескрипторы остаются открыты → сервер "заполняется" без видимых файлов. Решение: перезапуск сервиса или `> /proc/<PID>/fd/<N>` для очистки.
4. **Команды:** `lsof +L1` (файлы с 0 ссылками), `lsof -p <PID> | grep deleted`, `systemctl restart <service>`.

### 10. Как диагностировать и решить проблему с тысячами соединений в состоянии `CLOSE_WAIT`?

1. **Кратко:** `CLOSE_WAIT` означает: удалённая сторона закрыла соединение, локальное приложение не вызвало `close()`.
2. **Подробно:** Это баг в приложении: после получения `EOF` (конец потока) нужно явно закрыть сокет. Если приложение не закрывает — дескрипторы утекают, пока не исчерпается `ulimit -n`. Решение: фикс кода, настройка таймаутов, рестарт сервиса.
3. **DevOps-контекст:** В прокси/балансировщиках ведёт к утечке FD, отказу новых соединений. Мониторинг: `ss -tnp state close-wait | wc -l` в Prometheus через `node_netstat_Tcp_CurrEstab`. Профилактика: `SO_LINGER`, `tcp_fin_timeout`.
4. **Команды:** `ss -tnp state close-wait`, `lsof -i :PORT`, `strace -p <PID> -e trace=network`, `kill -15 <PID>` для graceful restart.

### 11. ⭐ Как ограничить потребление памяти и CPU для процесса исключительно через `systemd`?

1. **Кратко:** Через директивы `MemoryMax=` и `CPUQuota=` в `[Service]` юните systemd.
2. **Подробно:** Systemd создаёт cgroup для сервиса и применяет лимиты. `CPUQuota=200%` = 2 ядра, `MemoryMax=1G` = жёсткий лимит RSS. При превышении memory — OOM Killer внутри cgroup, CPU — троттлинг. Работает без ручного управления cgroups.
3. **DevOps-контекст:** В bare-metal предотвращает "noisy neighbor" проблему. Аналог `resources.limits` в Kubernetes. Важно: лимиты применяются к процессу и всем его потомкам, что удобно для сервисов с воркерами.
4. **Команды:** `systemctl show myapp -p MemoryMax,CPUQuota`, `systemctl edit myapp` (drop-in), `cat /sys/fs/cgroup/system.slice/myapp.service/memory.max`.

### 12. Что такое `OOM Killer` и как настроить его приоритет (шанс выжить) для конкретного процесса?

1. **Кратко:** OOM Killer — механизм ядра, убивающий процессы при нехватке памяти; приоритет настраивается через `oom_score_adj` (-1000 до 1000).
2. **Подробно:** Ядро считает `oom_score` на основе RSS, времени работы, привилегий, дочерних процессов. `oom_score_adj` сдвигает результат: `-1000` защищает процесс (никогда не будет убит), `+1000` — первый кандидат. Значение наследуется потомками.
3. **DevOps-контекст:** Критично для БД, etcd, kube-apiserver: без настройки OOM может убить важное, а оставить память поедающий баг. В systemd: `OOMScoreAdjust=-900` в юните. Мониторинг: `node_oom_events` в Prometheus.
4. **Команды:** `echo -900 > /proc/<PID>/oom_score_adj`, `cat /proc/<PID>/oom_score`, `systemd: [Service] OOMScoreAdjust=-900`.

# 🔴 Senior (архитектура, trade-offs, troubleshooting)

### 13. ⭐ Как работает Virtual File System (VFS) и какие слои проходит запрос при вызове `read()`?

1. **Кратко:** VFS — абстрактный слой, унифицирующий разные ФС; `read()` → VFS → конкретный FS driver → block layer → disk.
2. **Подробно:** Приложение вызывает syscall → glibc → trap в ядро → VFS ищет inode/dentry в кэше → вызывает `.read` из файловой структуры → page cache → если miss, I/O scheduler → драйвер диска → прерывание → копирование в user buffer.
3. **DevOps-контекст:** Понимание VFS необходимо для оптимизации I/O: выбор ФС (xfs vs ext4), настройка `vm.dirty_ratio`, `direct I/O`, `O_SYNC`. Объясняет, почему `fsync` дорогой, и как работает кэширование.
4. **Команды:** `strace -e trace=open,read,write`, `cat /proc/mounts`, `iostat -x 1`, `bpftrace -e 'tracepoint:syscalls:sys_enter_read { printf("%s\n", str(args->buf)); }'`.

### 14. Как именно `namespaces` и `cgroups` обеспечивают изоляцию контейнеров? Какие namespaces критичны для безопасности?

1. **Кратко:** Namespaces изолируют пространство имён (PID, NET, MNT и т.д.), cgroups ограничивают ресурсы. Вместе = контейнер.
2. **Подробно:** 
    - `PID` — своя таблица процессов, `NET` — свой стек и интерфейсы, `MNT` — точка монтирования, `USER` — маппинг UID/GID, `UTS` — hostname, `IPC` — shared memory.
    - Без `USER`/`PID` контейнер уязвим: процесс может видеть/убивать процессы хоста.
3. **DevOps-контекст:** Фундамент Docker/containerd. Понимание namespaces объясняет, почему `ps` в контейнере видит только свои процессы, и почему `--privileged` опасен (отключает изоляцию). В Kubernetes: `securityContext`, `hostPID`, `hostNetwork`.
4. **Команды:** `unshare --mount --pid --fork /bin/bash`, `ls -l /proc/<PID>/ns/`, `cat /sys/fs/cgroup/.../memory.max`, `docker inspect --format='{{.HostConfig.PidMode}}'`.

### 15. Как отладить зависший процесс, который не реагирует на сигналы, занимает 100% CPU и не пишет в логи?

1. **Кратко:** Сначала `kill -SIGQUIT` (дамп стека), затем `gdb` или `strace`. Если ядро зависло — `sysrq`.
2. **Подробно:** 
    - `strace -p` покажет, на каком syscall висит (напр. `futex` = deadlock).
    - `gdb -p <PID> -ex "thread apply all bt" -ex quit` даст стек вызовов всех потоков.
    - Если CPU 100% — spinlock или busy loop: `perf top -p <PID>` покажет функцию.
    - Если процесс в `D`-state (uninterruptible sleep) — проблема в драйвере/диске: `dmesg`, `iostat`.
3. **DevOps-контекст:** В продакшене deadlocks и busy loops блокируют очереди. Нужен быстрый дамп состояния без убийства процесса. Автоматизация: `systemd` может делать `coredump` при краше, `pstack` в скриптах мониторинга.
4. **Команды:** `kill -3 <PID>` (Java thread dump), `gdb -batch -p <PID> -ex "bt"`, `echo t > /proc/sysrq-trigger` (show tasks), `perf record -p <PID> sleep 10`.

### 16. ⭐ В чём разница между `soft` и `hard` limits в `/etc/security/limits.conf`? Как они соотносятся с `ulimit` и cgroups?

1. **Кратко:** `soft` — предупреждение, процесс может запросить больше до `hard`; `hard` — абсолютный предел, меняется только root. `ulimit` читает/меняет текущие soft limits.
2. **Подробно:** `limits.conf` применяется PAM при логине. `ulimit -S`/`-H` в shell. В systemd лимиты переопределяются через `LimitNOFILE`, `LimitMEMLOCK`. cgroups v2 заменяет многие из них, но `limits.conf` всё ещё работает для user sessions.
3. **DevOps-контекст:** Утечки FD, превышение лимитов стека/памяти падают молча. Важно для Java (`nofile`), БД (открытые файлы), сетевых демонов. В контейнерах: `--ulimit nofile=65536:65536` в Docker, `securityContext` в K8s.
4. **Команды:** `ulimit -a`, `cat /etc/security/limits.conf`, `systemctl show myapp -p LimitNOFILE`, `cat /proc/<PID>/limits`.

### 17. Как работает `eBPF` в современном Linux и какие DevOps-задачи он решает лучше традиционных инструментов?

1. **Кратко:** eBPF — безопасная виртуальная машина в ядре; позволяет запускать пользовательский код в ядре без пересборки модулей.
2. **Подробно:** Традиционные инструменты (`strace`, `perf`) медленные или требуют перезагрузки. eBPF работает через hook'и (tracepoints, kprobes, cgroups), даёт метрики в реальном времени с минимальным оверхедом (<1%). Код верифицируется перед загрузкой — не может крашнуть ядро.
3. **DevOps-контекст:** DevOps использует eBPF для: сетевой видимости (Cilium), профилирования CPU/IO (bcc), безопасности (Falco), tracing без restart. Замена `tcpdump` на `bpftrace`, мониторинг задержек системных вызовов, детектирование аномалий.
4. **Команды:** `bpftool prog list`, `bcc-tools`, `cilium status`, `bpftrace -e 'tracepoint:syscalls:sys_enter_open { printf("%s\n", str(args->filename)); }'`, `kubectl gadget`.

### 18. Как безопасно обновить ядро Linux на продакшен-кластере без простоя сервисов? Какие ограничения у live-patching?

1. **Кратко:** Live Patching (Kpatch/KGraft) подменяет функции ядра на лету; без простоя обновляются только критические баги/уязвимости.
2. **Подробно:** 
    - Ограничения: нельзя менять структуры данных, добавлять модули, менять ABI. Только точечные патчи.
    - Полное обновление ядра требует reboot + `kexec` (загрузка нового ядра из текущего, минуя BIOS POST).
    - Стратегия: rolling update в кластере, дренаж ноды, обновление, проверка, возврат трафика.
3. **DevOps-контекст:** В enterprise (финтех, телеком) reboot запрещён в часы пик. Live patching закрывает CVE до планового окна. `kexec` ускоряет reboot с минут до секунд. Важно: тестировать патчи в staging, иметь rollback-план.
4. **Команды:** `kpatch install`, `kpatch list`, `kexec -l /boot/vmlinuz... --initrd=... --append="..."`, `systemctl kexec`, `uname -r` для проверки версии.
