# 🟢 Junior (базовое понимание)

### 1. Что такое система контроля версий и чем распределённая (Git) отличается от централизованной (SVN)?

1. **Кратко:** СКВ отслеживает изменения в коде; в распределённой каждый имеет полную копию истории.
2. **Подробно:** Централизованная (SVN) хранит историю только на сервере — для коммита, просмотра истории нужен сеть. В Git каждый `clone` — это полный бэкап: можно коммитить, ветвиться, смотреть `blame` офлайн. Синхронизация — явная (`fetch`/`push`).
3. **DevOps-контекст:** Распределённость критична для отказоустойчивости: если сервер упал, любой разработчик может временно стать источником истины. Также ускоряет CI: раннеры могут работать с локальной историей без сетевых задержек.
4. **Команды:** `git clone`, `git commit`, `git push` vs `svn checkout`, `svn commit`.

### 2. Какие три основных состояния файлов в Git ты знаешь?

1. **Кратко:** `untracked` (не отслеживается), `modified` (изменён), `staged` (добавлен в индекс).
2. **Подробно:**
    - `untracked`: новый файл, Git его не видит в истории → `git add` переводит в `staged`
    - `modified`: файл из репо изменён в рабочей папке → `git diff` покажет изменения
    - `staged`: изменения подготовлены к коммиту → `git diff --staged` покажет, что войдёт в коммит
3. **DevOps-контекст:** Понимание состояний помогает избежать ошибок в CI: например, не закоммитить `.env` из-за забытого `.gitignore`, или не пропустить критичный файл из-за `git add .` без проверки.
4. **Команды:** `git status`, `git add`, `git restore --staged`.

### 3. В чём разница между `git add` и `git commit`?

1. **Кратко:** `add` помещает изменения в индекс (подготовка), `commit` фиксирует индекс в истории.
2. **Подробно:** `git add file` копирует текущее содержимое файла в объект `blob` и обновляет `index`. `git commit` создаёт объект `tree` из индекса, затем объект `commit`, связывает с родителем и обновляет ссылку ветки.
3. **DevOps-контекст:** Разделение позволяет атомарно группировать изменения: например, закоммитить только фикс бага, оставив в рабочей папке экспериментальный код для следующего коммита. В пайплайнах это позволяет точечно тестировать изменения.
4. **Команды:** `git add -p` для интерактивного выбора, `git commit -m "msg"`.

### 4. Что делает команда `git status` и зачем она нужна перед коммитом?

1. **Кратко:** Показывает состояние файлов: какие изменены, добавлены, удалены, что в индексе.
2. **Подробно:** Сравнивает `HEAD` (последний коммит), `index` (подготовленное) и рабочую папку. Выводит цветом и кратко (`git status -s`): `??` новый, `M` изменён, `D` удалён, `A` добавлен.
3. **DevOps-контекст:** Предотвращает коммит секретов, логов, временных файлов. В CI можно использовать `git status --porcelain` для скриптовой проверки "чистоты" сборки.
4. **Команды:** `git status --short`, `git status --porcelain=v1`.

### 5. Как создать новую ветку и переключиться на неё одной командой?

1. **Кратко:** `git checkout -b <name>` или современная `git switch -c <name>`.
2. **Подробно:** Команда создаёт файл `.git/refs/heads/<name>` с хешом текущего коммита, затем обновляет `HEAD` на `ref: refs/heads/<name>` и делает checkout соответствующего `tree` в рабочую папку.
3. **DevOps-контекст:** В командах с частыми ветками (`feature/*`, `hotfix/*`) это базовая операция. В пайплайнах ветки часто создаются автоматически по именам коммитов или задач в трекере.
4. **Команды:** `git switch -c feature/JIRA-123`, `git branch -m old new` для переименования.

### 6. Что такое `HEAD` и что значит "detached HEAD"?

1. **Кратко:** `HEAD` — указатель на текущую ветку или коммит; "detached" — когда он указывает напрямую на коммит, а не на ветку.
2. **Подробно:** Обычно `HEAD` содержит `ref: refs/heads/main`. При `git checkout abc123` он становится `abc123`. Новые коммиты в этом состоянии не привязаны к ветке и могут быть потеряны при переключении.
3. **DevOps-контекст:** В CI часто проверяют конкретный коммит или тег — это detached HEAD. Важно не пытаться пушить из такого состояния без создания ветки.
4. **Команды:** `git checkout -b temp` для сохранения коммитов из detached, `git symbolic-ref HEAD` чтобы проверить, на что указывает HEAD.

### 7. Как отменить изменения в файле, который ещё не добавлен в индекс?

1. **Кратко:** `git restore <file>` или `git checkout -- <file>`.
2. **Подробно:** Команда копирует содержимое файла из `HEAD` (последнего коммита) в рабочую папку, перезаписывая локальные изменения. Если файл новый (`untracked`) — сначала нужно добавить, потом отменять.
3. **DevOps-контекст:** В скриптах можно использовать для "сброса" тестовых конфигов перед запуском. В CI — для очистки артефактов между этапами.
4. **Команды:** `git restore .` для всех файлов, `git restore --source=main file.txt` для взятия из другой ветки.

### 8. Зачем нужен файл `.gitignore` и какие файлы туда обычно добавляют в DevOps-проектах?

1. **Кратко:** Исключает файлы из отслеживания; в DevOps — секреты, артефакты, зависимости, логи.
2. **Подробно:** `.gitignore` применяется до `git add`: даже `git add .` проигнорирует указанные паттерны. Поддерживает глобальный (`~/.gitignore`) и локальный (`.git/info/exclude`) игнор.
3. **DevOps-контекст:** Предотвращает утечку секретов в репо. В CI можно проверять, что `.gitignore` покрывает все чувствительные файлы, с помощью `gitleaks` или `trufflehog`.
4. **Команды:** `git check-ignore -v <file>` для отладки, почему файл игнорируется.

### 9. Что делает `git clone --depth=1` и зачем это используют в CI?

1. **Кратко:** Скачивает только последний коммит (shallow clone), экономя время и место.
2. **Подробно:** Не загружает полную историю, только текущий `tree` и `commit`. Ограничивает операции: нельзя делать `rebase` на старые коммиты, `git log` обрезан.
3. **DevOps-контекст:** Критично для ephemeral CI-раннеров: ускоряет клонирование с минут до секунд, экономит дисковое пространство. Для тегированных релизов можно использовать `--depth=1 --branch=v1.2.3`.
4. **Команды:** `git fetch --unshallow` чтобы превратить shallow в полный клон при необходимости.

### 10. Как посмотреть историю коммитов в компактном формате?

1. **Кратко:** `git log --oneline --graph --all`.
2. **Подробно:** `--oneline` — хеш + сообщение в одну строку, `--graph` — ASCII-ветвление, `--all` — все ветки. Можно добавить `--decorate` для отображения тегов и имён веток.
3. **DevOps-контекст:** Быстрый аудит изменений перед деплоем. В скриптах: `git log --format=%H%n%s -1` для получения хеша и сообщения последнего коммита.
4. **Команды:** `git log -n 10 --stat` для просмотра изменённых файлов, `git log --since="1 week"` для фильтрации по времени.

# 🟡 Middle (применение, нюансы)

### 11. ⭐ В чём разница между `git merge` и `git rebase`? Когда что использовать?

1. **Кратко:** `merge` создаёт коммит слияния, сохраняя историю ветвления; `rebase` переписывает историю, делая её линейной.
2. **Подробно:**
    - `merge`: безопасен для публичных веток, сохраняет контекст "что и когда ветвилось", но может создавать шум.
    - `rebase`: чистая история, упрощает `bisect`/`blame`, но меняет хеши коммитов — нельзя использовать на ветках, которые уже используют другие.
3. **DevOps-контекст:** В CI/CD часто используют `rebase` для feature-веток перед мержом в `main`, чтобы избежать лишних коммитов слияния. В релизных ветках (`release/*`) — только `merge` для аудита.
4. **Команды:** `git merge --no-ff` для сохранения истории, `git rebase -i` для интерактивной чистки, `git pull --rebase` для настройки по умолчанию.

### 12. Что такое fast-forward merge и как его отключить? Зачем это может понадобиться?

1. **Кратко:** Когда ветка не уходила вперёд, Git просто передвигает указатель; отключается `--no-ff`.
2. **Подробно:** В режиме FF не создаётся новый коммит — история выглядит линейно. `--no-ff` принудительно создаёт merge-commit, сохраняя информацию о том, что это была отдельная ветка.
3. **DevOps-контекст:** В проектах с семантическим версионированием и генерацией changelog по коммитам важно сохранять структуру ветвления — `--no-ff` позволяет автоматически группировать изменения по фичам.
4. **Команды:** `git merge --no-ff feature`, `git config merge.ff false` для глобального отключения.

### 13. Как безопасно отменить последний коммит, если он ещё не отправлен на сервер?

1. **Кратко:** `git reset --soft HEAD~1` (сохранить изменения в индексе) или `--mixed` (по умолчанию, оставить в рабочей папке).
2. **Подробно:**
    - `--soft`: отменяет коммит, но оставляет файлы в индексе → можно добавить забытые файлы и перекоммитить.
    - `--mixed`: отменяет коммит и индекс → файлы останутся изменёнными в рабочей папке.
    - `--hard`: ⚠️ полностью удаляет изменения — использовать только если уверен.
3. **DevOps-контекст:** В пайплайнах можно использовать `reset --soft` для "переупаковки" коммитов перед пушем, если автоматический коммит оказался неполным.
4. **Команды:** `git commit --amend` для простого исправления сообщения или добавления забытого файла.

### 14. ⭐ Что делает `git fetch` и чем он отличается от `git pull`? Почему в CI часто используют `fetch`?

1. **Кратко:** `fetch` скачивает метаданные, не меняя рабочую копию; `pull` = `fetch` + `merge`/`rebase`.
2. **Подробно:** `fetch` обновляет только `refs/remotes/origin/*` — можно безопасно анализировать, что изменилось на сервере. `pull` сразу пытается влить изменения, что может вызвать конфликты в headless-средах.
3. **DevOps-контекст:** В CI сначала делают `fetch`, проверяют `CI_COMMIT_SHA`, теги, запускают пре-чеки, и только потом — `merge` или `rebase` по стратегии. Это даёт контроль и предсказуемость.
4. **Команды:** `git fetch --tags --prune`, `git merge origin/main` или `git rebase origin/main` вручную после анализа.

### 15. Как разрешить конфликт слияния и что делать, если запутался в процессе?

1. **Кратко:** Отредактировать файлы с маркерами `<<<<<<<`, `=======`, `>>>>>>>`, затем `git add` и `git commit`.
2. **Подробно:** При конфликте Git помечает проблемные места. Можно использовать `git mergetool` (настроенный diff-тул), или вручную выбрать нужные части. Если запутался — `git merge --abort` отменит слияние и вернёт к состоянию до `merge`.
3. **DevOps-контекст:** В командах настраивают `rerere.enabled=true`, чтобы Git запоминал, как ты разрешал похожие конфликты, и предлагал автоматическое решение в будущем. В CI можно автоматически отклонять PR с конфликтами.
4. **Команды:** `git status` для списка конфликтующих файлов, `git checkout --ours/--theirs file` для выбора одной из версий.

### 16. Что такое `git reflog` и в каких ситуациях он может спасти?

1. **Кратко:** Журнал всех перемещений HEAD и веток; позволяет восстановить "потерянные" коммиты.
2. **Подробно:** В отличие от `git log`, `reflog` показывает даже недостижимые из веток коммиты. Хранится локально 90 дней по умолчанию. Пример: после `git reset --hard HEAD~1` можно вернуть коммит через `git reset --hard HEAD@{1}`.
3. **DevOps-контекст:** При отладке пайплайнов можно использовать `reflog` в раннере, чтобы понять, какой коммит реально проверялся. Также спасает при случайном `reset` в production-ветке (если ещё не пушил).
4. **Команды:** `git reflog show main`, `git diff HEAD@{0} HEAD@{2}`, `git branch recovery HEAD@{3}`.

### 17. Как добавить существующий проект в репозиторий, не теряя историю, если нужно объединить с другим репо?

1. **Кратко:** Использовать `git subtree` или `git submodule`, либо `git filter-repo` для сложного мержа историй.
2. **Подробно:**
    - `submodule`: ссылка на внешний репо в определённом коммите — подходит для независимых библиотек.
    - `subtree`: встраивает историю другого репо в подпапку — проще для монорепозиториев.
    - `filter-repo`: переписывает пути и историю перед мержом — для сложных миграций.
3. **DevOps-контекст:** При переходе от нескольких репо к монорепозиторию важно сохранить историю для `blame` и аудита. `subtree` проще в поддержке в пайплайнах, чем `submodule`.
4. **Команды:** `git subtree add --prefix=libs/auth https://repo auth-main --squash`, `git submodule add https://repo libs/auth`.

### 18. Зачем подписывать коммиты (GPG/SSH) и как это настроить?

1. **Кратко:** Подпись подтверждает авторство и целостность; защищает от подделки коммитов.
2. **Подробно:** При `commit.gpgsign=true` Git создаёт криптографическую подпись с использованием приватного ключа. Сервер (GitHub/GitLab) может требовать подписанные коммиты для мержа в защищённые ветки.
3. **DevOps-контекст:** Критично для соблюдения стандартов безопасности (SOC2, ISO27001) и предотвращения атак типа "подмена кода в форке". В пайплайнах можно проверять подпись перед деплоем.
4. **Команды:** `git config --global gpg.format ssh`, `git config --global user.signingkey ~/.ssh/id_ed25519.pub`, `git config --global commit.gpgsign true`.

### 19. Как работает `git stash` и чем отличается от временного коммита?

1. **Кратко:** `stash` временно сохраняет изменения вне истории; не создаёт коммит в ветке.
2. **Подробно:** `git stash push` создаёт специальный коммит в `refs/stash`, содержащий индекс и рабочую папку. `git stash pop` применяет и удаляет из стека. В отличие от временного коммита, stash не виден в `git log` и не пушится.
3. **DevOps-контекст:** Удобно для переключения контекста в раннере: например, сохранить WIP-изменения, проверить хотфикс, затем вернуться. Не подходит для долгосрочного хранения — stash локален.
4. **Команды:** `git stash -u` (сохранить untracked файлы), `git stash list`, `git stash apply stash@{2}`.

### 20. Что такое `git bisect` и как с его помощью найти баг-коммит?

1. **Кратко:** Бинарный поиск по истории для определения коммита, в котором появился баг.
2. **Подробно:** Указываешь "хороший" (например, `v1.0`) и "плохой" (`HEAD`) коммиты. Git автоматически переключает коммиты посередине, ты тестируешь и помечаешь `git bisect good/bad`. За ~log₂(n) шагов находишь виновника.
3. **DevOps-контекст:** В автоматизированных пайплайнах можно запускать `git bisect run ./test.sh` — скрипт возвращает 0 (хорошо) / 1 (плохо), и bisect работает полностью автоматически.
4. **Команды:** `git bisect start`, `git bisect good v1.0`, `git bisect bad HEAD`, `git bisect run npm test`, `git bisect reset` для выхода.

# 🔴 Senior (архитектура, trade-offs, troubleshooting)

### 21. ⭐ Опиши, как устроена внутренняя база данных Git: какие типы объектов существуют и как они связаны?

1. **Кратко:** 4 типа объектов: `blob` (файл), `tree` (директория), `commit` (снимок), `tag` (метка); все хранятся в `.git/objects` по хешу.
2. **Подробно:**
    - `blob`: сырые данные файла + заголовок `blob <size>\0`; хеш = SHA-1(заголовок + данные)
    - `tree`: список записей `<режим> <имя> <хеш>`; хеш = от сериализованного списка
    - `commit`: заголовок (`tree`, `parent`, `author`, `committer`, `message`) + хеш корневого tree
    - `tag`: ссылается на объект + метаданные + подпись (для аннотированных)
    - Связи: `commit → tree → (blob | tree)*`; цепочка коммитов через `parent`
3. **DevOps-контекст:** Понимание объектов позволяет оптимизировать хранилище: например, `git repack` объединяет объекты в пакеты с дельта-компрессией. Также помогает при восстановлении повреждённых репо: `git fsck` проверяет целостность хешей.
4. **Команды:** `git cat-file -t <hash>` (тип объекта), `git cat-file -p <hash>` (содержимое), `git fsck --full`.

### 22. Почему объекты Git неизменяемы и как это влияет на дедупликацию и производительность?

1. **Кратко:** Неизменяемость гарантирует целостность через хеши; позволяет дедуплицировать идентичные файлы/деревья.
2. **Подробно:** Если два коммита содержат одинаковый файл — он хранится как один `blob` с одним хешом. При изменении одной строки создаётся новый `blob`, но все остальные файлы ссылаются на старые объекты. Это экономит место и ускоряет операции сравнения.
3. **DevOps-контекст:** В больших репо с частыми изменениями конфигов дедупликация критична: без неё история раздувалась бы экспоненциально. Также неизменяемость позволяет безопасно кэшировать объекты в распределённых системах (например, в Git-зеркалах).
4. **Команды:** `git count-objects -vH` для статистики хранилища, `git verify-pack -v .git/objects/pack/*.idx` для анализа паков.

### 23. Как работает дельта-компрессия и пакинг объектов? Когда и зачем запускать `git gc`?

1. **Кратко:** Дельта-компрессия хранит похожие объекты как разницы; пакинг объединяет объекты в один файл для ускорения доступа.
2. **Подробно:** При `git repack` Git анализирует объекты, находит похожие `blob`/`tree` и хранит базовый объект + дельты. Паки (`*.pack`) индексируются (`*.idx`) для быстрого поиска по хешу. `git gc` автоматически запускает пакинг при достижении порогов (число объектов, размер).
3. **DevOps-контекст:** В монорепозиториях с тысячами разработчиков регулярный `gc` критичен для производительности `clone` и `fetch`. В CI можно запускать `git gc --auto` после сборки, чтобы не раздувать кэш.
4. **Команды:** `git gc --aggressive --prune=now` для глубокой оптимизации, `git config gc.auto 6700` для настройки триггеров.

### 24. Что происходит "под капотом" при `git push`: как Git определяет, какие объекты отправлять?

1. **Кратко:** Сравнивает локальные и удалённые хеши веток, вычисляет недостающие объекты через граф достижимости, отправляет дельта-сжатые пакеты.
2. **Подробно:**
    - Клиент отправляет `want <sha>` для целевого коммита и `have <sha>` для известных объектов
    - Сервер строит граф, находит разницу (objects reachable from `want`, но не от `have`)
    - Объекты упаковываются с дельта-компрессией, передаются по протоколу (smart HTTP / SSH)
    - Сервер проверяет хеши, обновляет ссылку, запускает хуки
3. **DevOps-контекст:** Понимание этого помогает оптимизировать пуши: например, делать `fetch` перед `push`, чтобы уменьшить разницу. Также объясняет, почему пуш больших бинарников медленный — они не дедуплицируются так же эффективно, как код.
4. **Команды:** `GIT_TRACE_PACKET=1 git push` для отладки протокола, `git push --dry-run` для проверки без отправки.

### 25. ⭐ Как безопасно переписывать историю в feature-ветке и почему `--force-with-lease` безопаснее `--force`?

1. **Кратко:** Переписывать можно только локальные/неопубликованные ветки; `--force-with-lease` проверяет, что удалённая ветка не изменилась с момента последнего `fetch`.
2. **Подробно:**
    - `--force` слепо перезаписывает remote, даже если туда запушили новые коммиты → потеря чужой работы
    - `--force-with-lease` сравнивает ожидаемый хеш (из `refs/remotes/origin/branch`) с реальным на сервере; если не совпадает — отклоняет пуш
    - Перед переписыванием: убедиться, что ветка не используется другими, сделать бэкап (`git branch backup`)
3. **DevOps-контекст:** В командах с активным параллельным развитием `--force-with-lease` — обязательный стандарт. В защищённых ветках сервер вообще блокирует force-push через Branch Protection Rules.
4. **Команды:** `git push --force-with-lease origin feature`, `git config push.default simple` для предотвращения случайных пушей не в ту ветку.

### 26. Как удалить чувствительные данные (пароль, ключ) из всей истории репозитория? Какие есть инструменты и подводные камни?

1. **Кратко:** Использовать `git filter-repo` (или BFG) для переписывания истории; обязательно ротировать секрет и проверить аудит.
2. **Подробно:**
    - `git filter-repo --invert-paths --path secret.env --force` удалит файл из всех коммитов
    - Старые объекты останутся в `.git/objects` до `git gc --prune=now`
    - После переписывания: `git push --force-with-lease`, уведомить команду, проверить, что никто не клонировал после утечки
    - ⚠️ `git rm` или `commit --amend` не удаляют данные из истории — они остаются доступны по хешу
3. **DevOps-контекст:** В инцидент-менеджменте утечка секрета — критичный инцидент: сначала ротируй ключ в источнике (Vault, AWS), потом чисти историю. Автоматизируй сканирование в пре-коммит хуках и в CI (`gitleaks`, `trufflehog`).
4. **Команды:** `git filter-repo --path-glob '*.env' --invert-paths`, `BFG --delete-files secret.key`, `git reflog expire --expire=now --all && git gc --prune=now`.

### 27. В чём преимущества и риски использования монорепозитория с точки зрения работы Git?

1. **Кратко:** Плюсы: атомарные изменения, единое версионирование; минусы: размер репо, скорость операций, сложность прав доступа.
2. **Подробно:**
    - ✅ Атомарные коммиты: изменение в сервисе + инфраструктуре в одном коммите
    - ✅ Единый `git blame` и история для кросс-сервисных изменений
    - ❌ `clone` и `status` замедляются с ростом числа файлов
    - ❌ Сложнее настраивать fine-grained доступ (все видят всё)
3. **DevOps-контекст:** Для монорепозиториев критичны: `--filter=blob:none`, `sparse-checkout`, кэширование зависимостей, `path-based triggers` в CI (запускать пайплайн только если изменился `apps/service-a/**`). Инструменты: `bazel`, `nx`, `turborepo` для инкрементальной сборки.
4. **Команды:** `git sparse-checkout set apps/service-a infra/terraform`, `git clone --filter=blob:none --sparse <url>`.

### 28. Как оптимизировать клонирование и работу с огромным репозиторием (>10 Гб, 100k+ файлов)?

1. **Кратко:** Использовать shallow/partial clone, sparse checkout, пакинг, кэширование.
2. **Подробно:**
    - `--depth=1` для получения только актуального кода (если не нужна история)
    - `--filter=blob:none` + `--sparse` для загрузки метаданных и только нужных подпапок
    - На сервере: регулярный `git repack -Ad --depth=250 --window=250` для оптимального пакинга
    - На клиенте: `core.preloadindex=true`, `fetch.parallel=10` для ускорения операций
3. **DevOps-контекст:** В крупных организациях используют зеркала репо с оптимизированной структурой, CDN для раздачи `.pack`-файлов, прокси для Git-трафика. В CI — кэширование `.git` между запусками через `cache` или `artifact`.
4. **Команды:** `git config --global fetch.parallel 10`, `git clone --filter=blob:none --sparse --depth=1 <url>`, `git sparse-checkout init --cone`.

### 29. Как устроены хуки в Git и как их использовать для enforcement политик в команде?

1. **Кратко:** Хуки — исполняемые скрипты в `.git/hooks/`, срабатывают при событиях; для командного enforcement используют распространяемые фреймворки.
2. **Подробно:**
    - Клиентские хуки: `pre-commit`, `commit-msg`, `pre-push` — локальные, не коммитятся
    - Серверные: `pre-receive`, `update`, `post-receive` — настраиваются на Git-сервере
    - Для распространения: `husky` (JS), `pre-commit framework` (Python, мультиязычный), или копирование через `init.templateDir`
3. **DevOps-контекст:** В пайплайнах хуки дублируются: `pre-commit` → линт в CI, `commit-msg` → валидация формата сообщения в `merge_request`. Серверные хуки заменяются на Webhooks + CI-статусы для гибкости.
4. **Команды:** `pre-commit install`, `git config init.templateDir ~/.git-templates`, пример `.pre-commit-config.yaml` с `gitleaks`, `black`, `terraform validate`.

### 30. ⭐ Как спроектировать стратегию ветвления и релизов для проекта с ежедневными деплоями и 50+ разработчиками?

1. **Кратко:** Trunk-Based Development + feature flags + короткоживущие ветки + автоматизированные чеки + защищённый `main`.
2. **Подробно:**
    - `main` — всегда деплоябельный, защищён: требуется 2 аппрувера, зелёный CI, подписанные коммиты
    - Ветки `feature/*` живут <1-2 дня, мержатся через PR с squash/rebase
    - Feature flags для недоделанных функций: код в `main`, но функционал выключен
    - Релизы через теги: `git tag -a v1.2.3 -m "release"` → CI собирает артефакты, деплоит
    - Откаты: `git revert` (не `reset`), чтобы сохранить аудит
3. **DevOps-контекст:** Такая стратегия позволяет делать 10-100 деплоев в день без хаоса. Требует: покрытие тестами >80%, автоматический rollback при росте ошибок, мониторинг после деплоя. Инструменты: GitHub Actions/GitLab CI для пайплайнов, ArgoCD для GitOps-синхронизации.
4. **Команды:** Настройка Branch Protection в веб-интерфейсе, `git tag -s v1.2.3` для подписанного тега, `git revert -m 1 <merge-commit>` для отката мержа.
