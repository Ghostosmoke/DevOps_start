### 1. Что такое DevOps и в чём его главная цель?

1. **Кратко:** Культура и практики, объединяющие разработку и эксплуатацию для ускорения доставки ценности при сохранении надёжности.
2. **Подробно:** DevOps — не просто инструменты, а цикл: Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → Feedback. Ключевые принципы: автоматизация, измерение, совместная ответственность, непрерывное улучшение.
3. **DevOps-контекст:** На практике это означает: инфраструктура как код, самообслуживание для разработчиков, мониторинг как часть требования, постмортемы без поиска виноватых.
4. **Команды/Инструменты:** `git`, `docker`, `terraform`, `kubectl`, `prometheus`, `github actions`.

### 2. Чем виртуализация отличается от контейнеризации?

1. **Кратко:** ВМ эмулирует железо + гостевую ОС, контейнеры делят ядро хоста, изолируя процессы.
2. **Подробно:**
    - **VM**: Hypervisor → Guest OS → App. Полная изоляция, но накладные расходы на ОС (1-2 ГБ на инстанс).
    - **Container**: Container Runtime → изолированные процессы через namespaces/cgroups. Запуск за секунды, минимальный оверхед.
3. **DevOps-контекст:** Контейнеры идеальны для микросервисов и CI/CD: быстрый старт, переносимость, эффективное использование ресурсов. ВМ — для легаси, требований к ядру, жёсткой изоляции.
4. **Команды/Инструменты:** `docker run`, `podman`, `qemu`, `vboxmanage`, `systemd-nspawn`.

### 3. Что такое Infrastructure as Code (IaC) и зачем он нужен?

1. **Кратко:** Описание инфраструктуры в декларативных/императивных файлах, которые можно версионировать, тестировать и применять автоматически.
2. **Подробно:** IaC устраняет "снежинки" (уникальные серверы), обеспечивает идемпотентность (повторное применение даёт тот же результат), позволяет код-ревью инфраструктуры и откат изменений.
3. **DevOps-контекст:** Без IaC невозможно масштабирование, аудит и быстрое восстановление. В связке с GitOps — инфраструктура синхронизируется с репозиторием автоматически.
4. **Команды/Инструменты:** `terraform apply`, `ansible-playbook`, `pulumi up`, `aws cdk deploy`.

### 4. Какие основные компоненты Linux-системы ты знаешь?

1. **Кратко:** Ядро (kernel), shell, пользовательское пространство, система инициализации, драйверы.
2. **Подробно:**
    - **Kernel**: управляет ресурсами (CPU, memory, I/O), системные вызовы
    - **Shell**: интерфейс для команд (bash, zsh)
    - **Systemd/init**: запуск сервисов, зависимости, логирование
    - **Package manager**: установка ПО (apt, yum, apk)
3. **DevOps-контекст:** Понимание слоёв помогает отлаживать контейнеры (какое ядро используется?), настраивать автозапуск сервисов, оптимизировать образы (multi-stage builds).
4. **Команды/Инструменты:** `uname -r`, `ps aux`, `systemctl status`, `journalctl`, `strace`.

### 5. Что такое процесс, поток и как они связаны с ресурсами?

1. **Кратко:** Процесс — изолированный экземпляр программы с памятью; поток — лёгкая единица выполнения внутри процесса, разделяющая память.
2. **Подробно:**
    - **Process**: свой PID, адресное пространство, файловые дескрипторы
    - **Thread**: общий код/данные процесса, свой стек и регистры
    - **CPU scheduling**: ядро переключает контекст между процессами/потоками
3. **DevOps-контекст:** В контейнерах важно ограничивать CPU/память через cgroups, чтобы один "шумный" процесс не влиял на соседей. Мониторинг: `top`, `htop`, `pidstat`.
4. **Команды/Инструменты:** `ps -eo pid,ppid,cmd,%mem,%cpu`, `top -H`, `cgroups v2: /sys/fs/cgroup/`.

### 6. Как работает сеть: модель OSI и TCP/IP на пальцах?

1. **Кратко:** OSI — 7 уровней абстракции; TCP/IP — 4 уровня, на которых работает интернет.
2. **Подробно:**
    ```
    OSI:       TCP/IP:      Пример:
    7. Application  ─┐
    8. Presentation  ├─ Application  → HTTP, DNS, TLS
    9. Session      ─┘
    10. Transport   ─┬─ Transport    → TCP (надёжный), UDP (быстрый)
    11. Network     ─┼─ Internet     → IP, routing, ICMP
    12. Data Link   ─┴─ Link         → Ethernet, MAC, ARP
    13. Physical
    ```
3. **DevOps-контекст:** Понимание уровней помогает отлаживать: "не пингуется" (L3), "порт закрыт" (L4), "SSL handshake failed" (L5-7). В K8s: CNI-плагины работают на L2/L3, service mesh — на L7.
4. **Команды/Инструменты:** `ping` (ICMP), `curl -v` (HTTP), `tcpdump -i eth0 port 443`, `ss -tlnp`.

### 7. Что такое порт, сокет и как приложение слушает соединения?

1. **Кратко:** Порт — логическая точка доставки в ОС; сокет — конечная точка связи (IP:port); приложение создаёт socket, bind, listen, accept.
2. **Подробно:**
    - **Port 0-1023**: privileged (требуют root), 1024-65535: user
    - **Socket**: `AF_INET` (IPv4) + `SOCK_STREAM` (TCP) или `SOCK_DGRAM` (UDP)
    - **Listen queue**: backlog — сколько соединений ждать до accept()
3. **DevOps-контекст:** В контейнерах важно пробрасывать порты (`-p 8080:80`), в K8s — настраивать Services и Ingress. Мониторинг: `netstat`, `ss` показывают, кто что слушает.
4. **Команды/Инструменты:** `ss -tlnp`, `lsof -i :8080`, `nc -zv host port`, `iptables -L`.

### 8. Что такое переменные окружения и зачем они в контейнерах?

1. **Кратко:** Ключ-значение, доступные процессу; в контейнерах — основной способ конфигурации без изменения образа.
2. **Подробно:** Переменные наследуются от родительского процесса, могут переопределяться. В Docker: `ENV` в Dockerfile (билд-тайм) и `-e` при запуске (ран-тайм). В K8s: `env`, `envFrom`, ConfigMap/Secret.
3. **DevOps-контекст:** Позволяют делать один образ для всех сред (dev/stage/prod), меняя только конфиг. Безопасность: секреты через Secret, а не plain env.
4. **Команды/Инструменты:** `docker run -e VAR=value`, `kubectl exec -it pod -- env`, `printenv`, `envsubst`.

### 9. Чем отличается stateful от stateless приложения?

1. **Кратко:** Stateless не хранит состояние между запросами (каждый запрос независим); stateful хранит сессию/данные локально.
2. **Подробно:**
    - **Stateless**: легко масштабировать горизонтально, любой инстанс обработает запрос
    - **Stateful**: требует sticky sessions или внешнего хранилища состояния (БД, Redis)
3. **DevOps-контекст:** В K8s stateless — Deployment + Service, stateful — StatefulSet + PersistentVolume. Для stateless проще rolling update и откат.
4. **Команды/Инструменты:** `kubectl rollout status`, `redis-cli`, `pg_isready`, `sticky-session` в Ingress.

### 10. Что такое health check и зачем он нужен в оркестрации?

1. **Кратко:** Механизм проверки, что приложение готово принимать трафик (liveness) и работает корректно (readiness).
2. **Подробно:**
    - **Liveness**: "жив ли процесс?" — если нет, перезапустить
    - **Readiness**: "готов ли обрабатывать запросы?" — если нет, исключить из балансировки
    - **Startup**: для медленного старта, чтобы не убивать приложение до инициализации
3. **DevOps-контекст:** В K8s без readiness-проба приложение получит трафик до загрузки кэша/БД → ошибки. Liveness помогает самовосстановление без вмешательства.
4. **Команды/Инструменты:** `kubectl describe pod | grep -A5 Probes`, `curl http://localhost:8080/health`, `exec: ["/bin/grpc_health_probe"]`.

### 11. ⭐ Как работает DNS и почему это критично для микросервисов?

1. **Кратко:** DNS преобразует имена в IP; в микросервисах — основа service discovery, балансировки и отказоустойчивости.
2. **Подробно:**
    - **Иерархия**: корень → TLD (.com) → домен → поддомен
    - **Запись A/AAAA**: имя → IPv4/IPv6
    - **CNAME**: алиас → другое имя
    - **SRV**: сервис + порт (используется в service mesh)
    - **TTL**: кэширование записей, баланс между актуальностью и нагрузкой
3. **DevOps-контекст:** В K8s CoreDNS даёт имена вида `service.namespace.svc.cluster.local`. При деплое нового пода DNS обновляется с задержкой (TTL + кэш клиента) — важно для rolling update. Service mesh (Istio) перехватывает DNS для трафика.
4. **Команды/Инструменты:** `dig +short service.namespace.svc.cluster.local`, `nslookup`, `kubectl get endpoints`, `coredns` логирование.

### 12. Что такое load balancer и какие алгоритмы балансировки ты знаешь?

1. **Кратко:** Распределяет входящий трафик между инстансами; алгоритмы: round-robin, least-connections, IP-hash, weighted.
2. **Подробно:**
    - **L4 (TCP/UDP)**: балансирует по портам, не видя контента (быстро, но без интеллекта)
    - **L7 (HTTP)**: видит URL, заголовки, может делать routing, SSL termination, кэширование
    - **Health checks**: исключает нездоровые бэкенды из ротации
3. **DevOps-контекст:** В облаке: AWS ALB/NLB, GCP Load Balancer. В K8s: Service type=LoadBalancer (облачный) или Ingress (L7). Для canary-деплоя — взвешенная маршрутизация.
4. **Команды/Инструменты:** `nginx` upstream, `haproxy.cfg`, `kubectl expose`, `aws elbv2 describe-target-health`.

### 13. Как устроена файловая система Linux: inodes, mount, permissions?

1. **Кратко:** Inode хранит метаданные файла; mount подключает ФС в дерево; permissions (rwx) управляют доступом.
2. **Подробно:**
    - **Inode**: номер, тип, права, владелец, размер, ссылки, блоки данных
    - **Hard link**: ещё одна ссылка на тот же inode; **symlink**: файл с путём к другому файлу
    - **Mount**: точка подключения другой ФС (диск, сеть, tmpfs)
    - **Permissions**: user/group/other + special bits (SUID, SGID, sticky)
3. **DevOps-контекст:** В контейнерах важно понимать, что изменения в слоях — copy-on-write. Persistent Volumes в K8s — это mount внешних хранилищ. Права влияют на безопасность: не запускать контейнеры от root.
4. **Команды/Инструменты:** `ls -li`, `stat file`, `mount | grep /dev`, `chmod 644`, `find / -inum 12345`.

### 14. ⭐ Что такое immutable infrastructure и какие у неё плюсы?

1. **Кратко:** Серверы/контейнеры не изменяются после развёртывания; для изменений создаётся новый образ и заменяется старый.
2. **Подробно:**
    - **Принцип**: конфигурация в образе, данные — во внешних хранилищах
    - **Плюсы**: предсказуемость, лёгкий откат, устранение дрейфа конфигурации
    - **Минусы**: требует автоматизации, больше артефактов, сложнее отладка "на месте"
3. **DevOps-контекст:** В связке с CI/CD: каждый коммит → новый Docker image → деплой. В K8s: Deployment обновляет Pods, а не меняет запущенные. Для БД: миграции применяются до деплоя, данные вынесены.
4. **Команды/Инструменты:** `docker build -t app:$GIT_SHA`, `kubectl set image`, `terraform taint`, `packer build`.

### 15. Как работает Docker: namespaces, cgroups, union filesystem?

1. **Кратко:** Namespaces изолируют ресурсы, cgroups ограничивают потребление, union FS (overlay2) объединяет слои образа.
2. **Подробно:**
    - **Namespaces**: PID, net, mount, UTS, IPC, user — каждый контейнер видит своё пространство
    - **Cgroups v2**: иерархия лимитов на CPU, memory, I/O, PIDs
    - **OverlayFS**: нижний слой (образ) + верхний (контейнер) = единое представление; изменения пишутся в верхний слой
3. **DevOps-контекст:** Понимание помогает оптимизировать образы (минимизировать слои), настраивать лимиты в K8s (`resources.limits`), отлаживать проблемы с правами (user namespace).
4. **Команды/Инструменты:** `docker inspect --format='{{.GraphDriver.Data}}'`, `ls /proc/$$/ns/`, `cat /sys/fs/cgroup/cpu.max`.

### 16. Что такое sidecar-паттерн и где он применяется в K8s?

1. **Кратко:** Вспомогательный контейнер в том же Pod, что и основное приложение; разделяет сеть/хранилище, расширяет функционал.
2. **Подробно:**
    - **Использование**: логирование (fluentd), прокси (Envoy), мониторинг (exporter), синхронизация конфигов
    - **Преимущества**: разделение ответственности, переиспользование, независимое обновление
    - **Ограничения**: общий lifecycle (Pod перезапускается целиком), конкуренция за ресурсы
3. **DevOps-контекст:** В K8s sidecar — основа для service mesh (istio-proxy), сбора метрик, шифрования трафика. Важно настраивать `resources` для каждого контейнера, чтобы sidecar не "задушил" приложение.
4. **Команды/Инструменты:** `kubectl describe pod | grep -A10 Containers`, `istioctl proxy-config`, `multicontainer pod spec`.

### 17. Чем отличается horizontal от vertical scaling? Когда что выбирать?

1. **Кратко:** Horizontal — добавить инстансов; Vertical — увеличить ресурсы одного инстанса.
2. **Подробно:**
    - **Horizontal (+)**: отказоустойчивость, гибкость, подходит для stateless; **(-)**: сложность балансировки, общее состояние
    - **Vertical (+)**: просто, не требует изменений в коде; **(-)**: предел железа, downtime при апгрейде, единая точка отказа
3. **DevOps-контекст:** В облаке horizontal scaling — основа автоскейлинга (K8s HPA, AWS ASG). Vertical — для монолитов, БД (с оговорками), или как временное решение. Часто комбинируют: вертикально — БД, горизонтально — приложение.
4. **Команды/Инструменты:** `kubectl autoscale deployment`, `aws autoscaling put-scaling-policy`, `terraform count/for_each`.

### 18. Что такое eventual consistency и почему это важно в распределённых системах?

1. **Кратко:** Модель согласованности: после обновления данные станут одинаковыми на всех узлах "в конечном счёте", но не мгновенно.
2. **Подробно:**
    - **Strong consistency**: чтение всегда видит последнюю запись (требует блокировок, медленнее)
    - **Eventual**: высокая доступность, низкая задержка, но возможны "устаревшие" чтения
    - **Компромисс**: согласно CAP-теореме, при сетевом разделении выбираем AP или CP
3. **DevOps-контекст:** В распределённых БД (Cassandra, DynamoDB), кэшах (Redis), service mesh — понимание модели помогает проектировать приложения: например, не полагаться на мгновенную видимость данных после записи.
4. **Команды/Инструменты:** `consul kv put/get`, `cassandra consistency level`, `redis INFO replication`.

### 19. Как работают reverse proxy и чем Nginx отличается от API Gateway?

1. **Кратко:** Reverse proxy принимает запросы клиентов и перенаправляет бэкендам; API Gateway — расширенный proxy с аутентификацией, rate limiting, трансформацией.
2. **Подробно:**

    - **API Gateway**: маршрутизация по путям/методам, JWT-валидация, throttling, трансформация запросов/ответов, агрегация сервисов
3. **DevOps-контекст:** В K8s: Ingress Controller (nginx-ingress, traefik) — reverse proxy; Istio Gateway + VirtualService — API Gateway-подобная функциональность. Выбор зависит от сложности требований к безопасности и маршрутизации.
4. **Команды/Инструменты:** `nginx.conf` upstream/location, `kubectl get ingress`, `istioctl analyze`, `kong admin API`.

### 20. Что такое circuit breaker и зачем он нужен в микросервисах?

1. **Кратко:** Паттерн, который "разрывает цепь" при сбоях зависимого сервиса, предотвращая каскадные отказы.
2. **Подробно:**
    - **Состояния**: Closed (норма) → Open (сбой, запросы блокируются) → Half-Open (тестовый запрос) → Closed/Open
    - **Параметры**: failure threshold, timeout, success threshold
    - **Fallback**: возврат кэша, дефолтного значения, заглушки
3. **DevOps-контекст:** Реализуется в коде (Resilience4j, Polly) или в service mesh (Istio: `outlierDetection`). Критичен для отказоустойчивости: лучше быстро вернуть ошибку, чем держать клиента в ожидании и исчерпать ресурсы.
4. **Команды/Инструменты:** `istioctl proxy-config cluster`, `kubectl exec -- curl localhost:15000/stats`, библиотеки: `resilience4j`, `go-hystrix`.

### 21. ⭐ Как устроен Kubernetes control plane и как компоненты общаются между собой?

1. **Кратко:** Control plane управляет кластером: API Server — единая точка входа, etcd — хранилище состояния, Controller Manager и Scheduler — логика, kubelet — агент на нодах.
2. **Подробно:**
    ```
    Компоненты:
    • kube-apiserver: REST API, аутентификация, авторизация, admission-контроль
    • etcd: распределённое KV-хранилище (Raft), хранит всё состояние кластера
    • kube-controller-manager: контроллеры (ReplicaSet, Node, Job) — следят за желаемым состоянием
    • kube-scheduler: назначает поды на ноды по ресурсам, аффинити, тейнтам
    • cloud-controller-manager: интеграция с облаком (балансировщики, диски)

    Коммуникация:
    • Все компоненты общаются через API Server (не напрямую!)
    • Контроллеры используют watch-механизм: подписываются на изменения в etcd через API
    • kubelet получает задания через API, отчитывается о статусе ноды/подов
    ```
3. **DevOps-контекст:** Понимание архитектуры критично для отладки: "почему под не запускается?" → проверить API Server логи, квоты, scheduler events. Для HA: несколько реплик API Server + etcd cluster (3/5/7 нод). Безопасность: RBAC, admission webhooks, network policies.
4. **Команды/Инструменты:** `kubectl get --raw /healthz`, `etcdctl endpoint status`, `kubectl describe node`, `audit logs: /var/log/kubernetes/`.

### 22. Что такое CAP-теорема и как она влияет на выбор БД в облаке?

1. **Кратко:** В распределённой системе нельзя одновременно гарантировать: Consistency (согласованность), Availability (доступность), Partition tolerance (устойчивость к разделению) — максимум два из трёх.
2. **Подробно:**
    - **CP-системы** (Consistency + Partition): при разделении жертвуют доступностью (возвращают ошибку). Примеры: etcd, ZooKeeper, MongoDB (в строгом режиме).
    - **AP-системы** (Availability + Partition): при разделении жертвуют согласованностью (возвращают возможные устаревшие данные). Примеры: Cassandra, DynamoDB, CouchDB.
    - **P обязательна** в реальных сетях: разделение — не исключение, а норма.
3. **DevOps-контекст:** Выбор БД зависит от требований: для конфигураций/координации — CP (etcd), для пользовательских данных с высокой доступностью — AP (DynamoDB). Часто используют полиглот-персистентность: разные БД для разных задач.
4. **Команды/Инструменты:** `etcdctl member list`, `aws dynamodb describe-table`, `mongodb consistency level`, `cassandra consistency QUORUM`.

### 23. Как работает service mesh (Istio/Linkerd) и зачем он нужен?

1. **Кратко:** Выделенный инфраструктурный слой для управления сервис-сервис коммуникацией: трафик, безопасность, наблюдаемость — без изменений в коде приложения.
2. **Подробно:**
    - **Data plane**: sidecar-прокси (Envoy в Istio) перехватывает весь входящий/исходящий трафик пода
    - **Control plane**: управляет конфигурацией прокси (маршруты, политики, сертификаты)
    - **Функции**: mTLS, retry/timeout, circuit breaker, canary-деплой, распределённый трейсинг, метрики
3. **DevOps-контекст:** Service mesh решает проблемы микросервисов: сложность управления трафиком, безопасность "по умолчанию", единый стандарт наблюдаемости. Цена: оверхед на CPU/память (sidecar), сложность отладки, необходимость обучения.
4. **Команды/Инструменты:** `istioctl install`, `kubectl label namespace default istio-injection=enabled`, `istioctl proxy-config listener`, `kiali` для визуализации.

### 24. ⭐ Что такое GitOps и чем он отличается от классического CI/CD?

1. **Кратко:** GitOps — операционная модель, где желаемое состояние инфраструктуры и приложений хранится в Git, а агент в кластере автоматически синхронизирует реальное состояние с ним.
2. **Подробно:**
    ```
    Классический CI/CD:
    [Code] → [CI: build/test] → [CD: push to prod via kubectl/helm]
    → Состояние управляется "извне", сложно отследить дрейф

    GitOps (ArgoCD/Flux):
    [Git Repo: desired state] ←pull→ [Agent in Cluster] → apply → [Actual State]
    → Кластер сам тянет изменения, дрейф обнаруживается и исправляется автоматически
    ```
    - **Принципы**: декларативность, версионирование, автоматическая синхронизация, аудит через Git history
3. **DevOps-контекст:** GitOps даёт: простой откат (git revert), прозрачность (кто что изменил — в коммите), безопасность (нет доступа к kubeconfig снаружи). Ограничения: требует дисциплины (никаких `kubectl edit` в prod), сложнее для динамических ресурсов.
4. **Команды/Инструменты:** `argocd app sync`, `flux reconcile`, `kubectl get applications.argoproj.io`, `git diff` для аудита.

### 25. Как обеспечить zero-downtime deployment на уровне архитектуры?

1. **Кратко:** Комбинация стратегий деплоя, балансировки, миграций и отката, чтобы пользователи не замечали обновления.
2. **Подробно:**
    - **Деплой**: Blue-Green (мгновенный переключатель), Canary (постепенный трафик), Rolling Update (постепенная замена)
    - **Балансировка**: health checks исключают нездоровые инстансы из ротации до/во время деплоя
    - **Миграции БД**: backward-compatible изменения (expand/contract), выполнять до деплоя кода
    - **Откат**: быстрый (переключение трафика) или безопасный (git revert + rebuild)
3. **DevOps-контекст:** В K8s: `strategy.rollingUpdate.maxUnavailable=0` + readiness probes. Для БД: использовать онлайн-миграции (`pt-online-schema-change`, `gh-ost`). Тестирование: chaos engineering, canary-анализ метрик перед полным роллаутом.
4. **Команды/Инструменты:** `kubectl rollout status`, `istio virtualservice weight`, `flyway migrate`, `argocd app set --sync-policy automated`.

### 26. Что такое observability и чем отличается от мониторинга?

1. **Кратко:** Мониторинг отвечает на известные вопросы ("сервер упал?"), observability позволяет исследовать неизвестные проблемы через метрики, логи и трейсы.
2. **Подробно:**
    - **Три столпа**:
        - **Metrics**: агрегированные числовые данные (CPU, RPS, error rate) — для алертинга и трендов
        - **Logs**: структурированные события с контекстом — для отладки конкретных инцидентов
        - **Traces**: распределённые цепочки запросов — для анализа задержек и зависимостей
    - **Принцип**: инструментировать код для экспорта высокоразмерных данных, чтобы можно было задавать новые вопросы постфактум
3. **DevOps-контекст:** В микросервисах без observability невозможно понять, где именно сбой. Стандарт: OpenTelemetry для инструментирования, Prometheus/Grafana для метрик, Loki/ELK для логов, Jaeger/Tempo для трейсов.
4. **Команды/Инструменты:** `otel-collector config`, `promql: rate(http_requests_total[5m])`, `jaeger-ui`, `loki logql`.

### 27. Как работают distributed tracing и зачем нужны span/context?

1. **Кратко:** Distributed tracing отслеживает запрос через множество сервисов; span — единица работы, context — передаётся между сервисами для связывания спанов.
2. **Подробно:**
    - **Trace**: уникальный ID, охватывающий весь пользовательский запрос
    - **Span**: операция внутри сервиса (имя, время, статус, теги, ссылки на родительский спан)
    - **Context propagation**: заголовки (W3C Trace-Context, B3) передаются в HTTP/gRPC, чтобы следующий сервис мог продолжить трассировку
    - **Sampling**: не все запросы трассируются (стоимость), стратегии: always, ratio, adaptive
3. **DevOps-контекст:** Без трейсинга в микросервисах невозможно найти "узкое место" в цепочке из 10+ сервисов. Инструментирование через OpenTelemetry — стандарт, чтобы не зависеть от вендора. Визуализация: Jaeger, Grafana Tempo.
4. **Команды/Инструменты:** `otel-cli span`, `kubectl exec -- curl -H 'traceparent: ...'`, `jaeger query --service=api`, `prometheus: otel_span_duration_seconds`.

### 28. Что такое chaos engineering и как применять его безопасно?

1. **Кратко:** Практика намеренного внесения сбоев в продакшен-подобную среду для проверки отказоустойчивости и улучшения системы.
2. **Подробно:**
    - **Принципы**: начать с гипотезы ("если упадёт БД, система переключится на реплику"), проводить в контролируемых условиях, автоматизировать откат
    - **Эксперименты**: задержка сети, убийство подов, исчерпание памяти, отключение зон доступности
    - **Безопасность**: бласт-радиус (ограничить влияние), мониторинг в реальном времени, "стоп-кран" для немедленного отката
3. **DevOps-контекст:** Chaos engineering — не "ломать прод", а систематически проверять resilience. Инструменты: Chaos Mesh (K8s-native), Litmus, AWS Fault Injection Simulator. Интеграция в CI: запускать эксперименты на staging перед релизом.
4. **Команды/Инструменты:** `chaos-mesh experiment create`, `litmuschaos apply -f pod-delete.yaml`, `aws fis start-experiment`, `prometheus alert: chaos_experiment_active`.

### 29. Как спроектировать multi-region архитектуру с учётом RTO/RPO?

1. **Кратко:** Multi-region — развёртывание в нескольких гео-локациях для отказоустойчивости; RTO (время восстановления) и RPO (потеря данных) определяют стратегию репликации и переключения.
2. **Подробно:**
    - **RTO/RPO матрица**:
        | Цель | Стратегия | Пример |
        |------|-----------|--------|
        | RTO < 5 мин, RPO = 0 | Active-Active, синхронная репликация | Финтех, критичные сервисы |
        | RTO < 15 мин, RPO < 5 мин | Active-Passive, асинхронная репликация | E-commerce, медиа |
        | RTO < 1 час, RPO < 1 час | Backup/Restore, cold standby | Внутренние инструменты |
    - **Компоненты**: глобальный DNS/балансировщик (Route53, Cloudflare), репликация БД (multi-master, read replicas), синхронизация конфигов (GitOps), кросс-регион мониторинг
3. **DevOps-контекст:** В облаке: AWS Multi-AZ + Cross-Region Replication, GCP Multi-Regional. В K8s: Federation v2 (Karmada, Clusternet) или независимые кластера с GitOps-синхронизацией. Тестирование: регулярные DR-учения, chaos-эксперименты на регион.
4. **Команды/Инструменты:** `aws route53 create-health-check`, `kubectl config use-context`, `argocd app sync --project multi-region`, `prometheus federation`.

### 30. ⭐ Какие паттерны отказоустойчивости ты знаешь и как их комбинировать?

1. **Кратко:** Набор паттернов для построения resilient-систем: retry, timeout, circuit breaker, bulkhead, fallback, rate limiting — комбинируются в зависимости от сценария.
2. **Подробно:**
    ```
    Паттерны и когда применять:
    • Retry + Exponential Backoff: временные сбои (сеть, таймаут БД)
    • Timeout: предотвращение зависания при ожидании
    • Circuit Breaker: каскадные сбои, перегрузка зависимостей
    • Bulkhead: изоляция ресурсов (отдельные пулы потоков для разных сервисов)
    • Fallback: возврат кэша/дефолта при полной недоступности
    • Rate Limiting / Throttling: защита от перегрузки, DDoS

    Комбинация (пример для HTTP-клиента):
    1. Установить timeout (5s)
    2. При ошибке — экспоненциальный retry (3 попытки)
    3. Если ошибки продолжаются — открыть circuit breaker на 30s
    4. В открытом состоянии — возвращать fallback (кэш)
    5. Ограничить общий RPS через rate limiter
    ```
3. **DevOps-контекст:** Реализация: в коде (библиотеки), в service mesh (Istio: `retry`, `outlierDetection`), в API Gateway (rate limiting). Мониторинг: метрики по состояниям circuit breaker, retry count, fallback usage — для настройки порогов. Тестирование: chaos-эксперименты + нагрузочное тестирование.
4. **Команды/Инструменты:** `istioctl proxy-config route`, `resilience4j config`, `envoy rate_limit filter`, `prometheus: circuit_breaker_open`.
